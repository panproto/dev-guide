6  panproto-gat – The GAT Engine

panproto-gat is Level 0 of the panproto crate hierarchy: the mathematical foundation on which everything else is built. This crate implements generalized algebraic theories (GATs) as defined by Cartmell (1986), providing the data structures and algorithms for sorts, operations, equations, theory morphisms, and colimits.

This chapter walks through every major type and function in the crate, explaining both the Rust implementation and the mathematical concepts it encodes.

6.1 Sorts and Sort Parameters

A sort is a type in a GAT. Sorts may be simple (zero parameters) or dependent (parameterized by terms of other sorts). The distinction between simple and dependent sorts is what separates GATs from ordinary algebraic theories in the sense of Lawvere (1963): in an ordinary algebraic theory, all sorts are simple.

6.1.1 SortParam

A SortParam names a dependency of a sort on a term of another sort. For example, in the theory of categories, the hom-sort Hom(a: Ob, b: Ob) has two parameters, each of sort Ob.

6.1.2 Sort

A Sort combines a name with zero or more SortParam entries. Simple sorts have an empty parameter list.

use std::sync::Arc;

/// The primitive value kind that a value sort ranges over.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub enum ValueKind {
    /// Boolean values.
    Bool,
    /// Integer values.
    Int,
    /// Floating-point values.
    Float,
    /// String values.
    Str,
    /// Byte-sequence values.
    Bytes,
    /// Opaque token values.
    Token,
    /// Null / unit values.
    Null,
    /// Any value kind (polymorphic).
    Any,
}

/// The kind of a sort, distinguishing structural sorts from value/coercion sorts.
#[derive(Debug, Clone, Default, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub enum SortKind {
    /// Standard structural sort (vertices, edges, constraints).
    #[default]
    Structural,
    /// Value sort: carries data of a specific kind.
    Val(ValueKind),
    /// Coercion sort: a directed morphism between value kinds.
    Coercion {
        /// The source value kind.
        from: ValueKind,
        /// The target value kind.
        to: ValueKind,
    },

The arity() method returns the number of parameters – this is the key invariant that theory morphisms must preserve.

Tip

In panproto’s schema theories, most sorts are simple (Vertex, Edge, Constraint). Dependent sorts appear when a schema theory needs to express constraints that depend on a vertex (e.g., Constraint(v: Vertex)), or when modeling categories where morphisms depend on their source and target objects.

6.1.3 Concrete Examples

Here are some sorts you will encounter in panproto protocol definitions:

  • Vertex – simple sort, arity 0. The vertices of a schema graph.
  • Edge – simple sort, arity 0. Directed edges between vertices.
  • Hom(a: Ob, b: Ob) – dependent sort, arity 2. Morphisms in a category.
  • Constraint(v: Vertex) – dependent sort, arity 1. Constraints attached to a specific vertex.

6.2 Operations

An operation (also called a term constructor) is a function in a GAT. Each operation has a name, typed inputs, and a typed output. Operations with zero inputs are constants (nullary constructors).

use std::sync::Arc;

/// An operation (term constructor) in a GAT.
///
/// Operations are the functions/constructors of a GAT. Each operation
/// has typed inputs and a typed output, where types are sort names.
///
/// # Examples
///
/// - `src: Edge → Vertex` (graph source map)
/// - `add: (a: Int, b: Int) → Int` (integer addition)
/// - `id: (x: Ob) → Hom(x, x)` (identity morphism)
///
/// Based on the formal definition of GAT operations from Cartmell (1986).
#[derive(Debug, Clone, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub struct Operation {
    /// The operation name (e.g., "src", "tgt", "compose").
    pub name: Arc<str>,
    /// Typed inputs as `(param_name, sort_name)` pairs.
    pub inputs: Vec<(Arc<str>, Arc<str>)>,
    /// The output sort name.
    pub output: Arc<str>,
}

The inputs field stores (param_name, sort_name) pairs. This is sufficient for first-order operations. Higher-order operations (operations whose inputs are themselves operations) are not directly supported – panproto does not need them for its protocol modeling use case.

6.2.1 Operation Signatures in Practice

In a graph schema theory, you would see operations like:

  • src: (e: Edge) -> Vertex – the source vertex of an edge.
  • tgt: (e: Edge) -> Vertex – the target vertex of an edge.
  • unit: () -> Carrier – a nullary operation (constant) in a monoid theory.

The type signature of an operation is the pair (input_sorts, output_sort). Theory morphisms must preserve these signatures up to the sort mapping.

6.3 Equations

Equations (axioms) constrain the behavior of operations. An equation consists of a name and two Term trees that must be considered equal. panproto uses a simple term language:

  • Term::Var(name) – a variable reference.
  • Term::App(op_name, args) – application of an operation to arguments.

Equations are not computationally evaluated; they serve as proof obligations that theory morphisms must preserve. When panproto-gat checks a morphism, it verifies that for each equation lhs = rhs in the domain theory, the mapped equation F(lhs) = F(rhs) exists in the codomain theory.

6.4 Sort Kinds

Sorts are classified by their SortKind, which distinguishes structural sorts from value-level sorts:

  • Structural (default): standard schema-level sorts like Vertex, Edge, Constraint.
  • Val(kind): value sorts carrying data of a specific ValueKind (Bool, Int, Float, Str, Bytes, Token, Null, Any).
  • Coercion { from, to }: coercion sorts representing directed morphisms between value kinds.
  • Merger(kind): merger sorts that combine values of a specific kind.

Value sorts bridge the schema level and the instance level. A sort like Val(Str) classifies string values; Coercion { from: Str, to: Int } classifies functions that convert strings to integers (e.g., parse_int).

6.5 Directed Equations

A DirectedEquation extends the standard Equation with a computation direction. Where an equation lhs = rhs is an undirected assertion, a directed equation lhs ⟶ rhs specifies a rewrite rule: when the migration engine encounters a value matching lhs, it rewrites to rhs using the implementation term.

Each directed equation carries:

  • impl_term: a panproto_expr::Expr — the computable implementation of the rewrite.
  • inverse: an optional panproto_expr::Expr for the backward (put) direction.

The expression language is defined in the panproto-expr crate — a pure functional language with lambda calculus, pattern matching, records, lists, and ~50 built-in operations.

6.6 Conflict Policies

A ConflictPolicy specifies how merge conflicts are resolved for values of a given kind:

  • KeepLeft — always prefer the left branch.
  • KeepRight — always prefer the right branch.
  • Fail — report a conflict (default behavior).
  • Custom(expr) — evaluate a panproto_expr::Expr to decide.

6.7 Theory

The Theory struct is the central type of panproto-gat. It groups sorts, operations, equations, directed equations, and conflict policies under a name, with optional inheritance via extends.

/// A conflict resolution strategy for merge operations.
#[derive(Debug, Clone, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub enum ConflictStrategy {
    /// Keep the left (ours) value.
    KeepLeft,
    /// Keep the right (theirs) value.
    KeepRight,
    /// Fail on conflict.
    Fail,
    /// Custom resolution via an expression.
    Custom(panproto_expr::Expr),
}

/// A conflict resolution policy for merge operations.
#[derive(Debug, Clone, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub struct ConflictPolicy {
    /// A human-readable name for this policy.
    pub name: Arc<str>,
    /// The value kind this policy applies to.
    pub value_kind: ValueKind,
    /// The strategy to use when values conflict.
    pub strategy: ConflictStrategy,
}

/// A generalized algebraic theory (GAT).
///
/// Theories are named collections of sorts, operations, and equations,
/// with optional inheritance via `extends`. When a theory extends another,
/// it inherits all of the parent's sorts, operations, and equations.
///
/// # Index cache
///
/// Precomputed `FxHashMap` indices provide O(1) lookup by name for sorts,
/// operations, and equations. These are rebuilt from the vectors at
/// construction and deserialization time.

6.7.1 Theory Inheritance

The extends field enables compositional theory construction. When theory B extends theory A, B inherits all of A’s sorts, operations, and equations. The resolve_theory function computes the transitive closure of the inheritance chain, merging declarations and detecting cycles.

Diamond inheritance is handled correctly: if theories B and C both extend A, and theory D extends both B and C, the shared declarations from A appear exactly once in the resolved theory D.

Note

For the conceptual introduction to GATs, see the tutorial chapter 3.

6.8 Theory Morphisms

A theory morphism is a structure-preserving map from one theory to another. It maps sorts to sorts and operations to operations, subject to three preservation conditions:

  1. Arity preservation: if sort S has arity n, then F(S) must also have arity n.
  2. Signature preservation: if operation op: (a: A, b: B) -> C, then F(op): (a: F(A), b: F(B)) -> F(C).
  3. Equation preservation: if lhs = rhs is an equation in the domain, then F(lhs) = F(rhs) must be an equation in the codomain.

/// A structure-preserving map between two theories.
///
/// Maps sorts to sorts and operations to operations. A valid morphism
/// must preserve sort arities, operation type signatures, and equations.
#[derive(Debug, Clone, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
pub struct TheoryMorphism {
    /// A human-readable name for this morphism.
    pub name: Arc<str>,
    /// The name of the domain theory.
    pub domain: Arc<str>,
    /// The name of the codomain theory.
    pub codomain: Arc<str>,
    /// Mapping from domain sort names to codomain sort names.
    pub sort_map: HashMap<Arc<str>, Arc<str>>,
    /// Mapping from domain operation names to codomain operation names.
    pub op_map: HashMap<Arc<str>, Arc<str>>,

The check_morphism function validates all three conditions. This is the critical correctness check for schema migrations: a migration specification is valid if and only if it induces a valid theory morphism between the source and target schema theories.

6.8.1 Why Morphisms Matter for Migration

When you migrate data from schema v1 to schema v2, the migration is mathematically a theory morphism from v2’s schema theory to v1’s schema theory (note the direction reversal – this is the contravariance of restriction functors). The CompiledMigration in panproto-inst is the executable form of this morphism, and the wtype_restrict pipeline is the functor it induces on instances.

This is the fundamental guarantee that makes panproto migrations principled: because the restrict operation is functorial, composing two migrations and restricting is the same as restricting twice (Spivak 2012). There are no “migration bugs” that arise from order-of-operations issues.

6.8.2 Inducing Schema Renames

Theory morphisms induce schema-level renames via induce_schema_renames(). Sort-map entries where old ≠ new become NameSite::VertexKind renames (since sorts map to vertex kinds at the schema level), and op-map entries where old ≠ new become NameSite::EdgeKind renames. This is the top of the morphism tower described in Section 18.2.

6.9 Identity and Naming

The ident module provides three types that separate identity from display name:

  • ScopeTag: An opaque scope tag distinguishing different naming contexts. Generated via ScopeTag::fresh().
  • Ident: A first-class identifier where equality uses (scope, index) only, not the display name. Follows GATlab (Lynch et al. 2024).
  • Name: An interned string handle (Arc<str>) with a pointer-equality fast path. Drop-in replacement for String in hot-path types.

The NameSite enum enumerates the nine naming sites (edge labels, vertex IDs, vertex kinds, edge kinds, NSIDs, constraint sorts, instance anchors, theory names, sort names). SiteRename is a site-qualified rename operation used in schema morphism provenance and VCS commit metadata.

See Section 18.2 for the full contributor guide.

6.10 Colimits (Merging Theories)

The colimit function computes the pushout of two theories over a shared base theory. This is how panproto composes protocol fragments modularly.

6.10.2 Conflict Detection

The colimit function detects three kinds of conflicts:

  • Sort conflicts: T1 and T2 both declare a sort with the same name but different parameter lists, and the sort is not in the shared base.
  • Operation conflicts: same-named operations with incompatible signatures.
  • Equation conflicts: same-named equations with different term content.

Compatible duplicates (same name, same definition) are silently deduplicated. This allows independent protocol extensions to share common vocabulary without coordination, as long as they agree on the meaning of shared names.

6.10.3 Practical Use: Composing Schema Theories

The panproto-protocols crate uses colimits extensively. For example, the AT Protocol schema theory is built by combining a base graph theory (ThGraph with Vertex, Edge, src, tgt) with a constraint theory (ThConstraint with Constraint, target) over their shared ThVertex. The colimit produces a theory with all three sorts and all three operations, ready to serve as the schema theory for the AT Protocol.

6.11 Type-Checking

The typecheck module provides verification that terms and equations are well-typed with respect to a theory’s operation signatures. Type-checking catches malformed equations, unsound morphisms, and invalid term constructions at construction time rather than silently producing wrong results downstream.

6.11.1 typecheck_term

The core function typecheck_term infers the output sort of a term given a variable typing context and a theory. It walks the term tree recursively:

  • For Term::Var(x): returns the sort from the context, or GatError::UnboundVariable.
  • For Term::App { op, args }: looks up op in the theory, recursively typechecks each argument, verifies that each argument’s sort matches the operation’s input signature, and returns the output sort.
pub fn typecheck_term(
    term: &Term,
    ctx: &VarContext,
    theory: &Theory,
) -> Result<Arc<str>, GatError>

The VarContext type is a FxHashMap<Arc<str>, Arc<str>> mapping variable names to their sort names. Callers construct a context manually when type-checking a standalone term, but more commonly the context is inferred automatically by infer_var_sorts.

6.11.2 infer_var_sorts

infer_var_sorts walks both sides of an equation, collecting sort constraints from every operation application site. A variable used as argument i of operation op must have the sort declared for that input position. If the same variable appears at two positions requiring different sorts, the function returns GatError::ConflictingVarSort.

pub fn infer_var_sorts(eq: &Equation, theory: &Theory) -> Result<VarContext, GatError>

This function is the entry point for automatic variable sort inference, used by typecheck_equation, check_model, and free_model. It eliminates the need for explicit sort annotations on equation variables.

6.11.3 typecheck_equation and typecheck_theory

typecheck_equation combines inference and checking: it infers variable sorts from the equation structure, typechecks both sides, and verifies they produce the same output sort. If the two sides have different sorts, it returns GatError::EquationSortMismatch.

typecheck_theory iterates over all equations in a theory and typechecks each one, returning the first error encountered. This is the function called during VCS validation to ensure that a protocol’s schema theory is internally consistent.

pub fn typecheck_theory(theory: &Theory) -> Result<(), GatError>
Tip

Type-checking is fast – it is linear in the size of the term tree for each equation. Even theories with hundreds of equations typecheck in microseconds. The VCS pipeline runs typecheck_theory on every schema add and schema commit without measurable overhead.

The type-checking functions produce specific error variants that carry enough context for actionable diagnostics:

6.12 Model Checking

The check_model module verifies that a Model (a concrete instance that satisfies a theory’s requirements — like a Rust struct implementing a trait — here specifically an interpretation of a theory in Set) satisfies all equations of its theory by enumerating variable assignments from carrier sets and evaluating both sides.

6.12.1 Model and ModelValue

A Model provides an interpretation of a theory: each sort maps to a carrier set (a Vec<ModelValue>), and each operation maps to a Rust closure that takes ModelValue arguments and returns a ModelValue. ModelValue is an enum with Int(i64) and Str(String) variants – enough for the finite models that panproto needs for equation verification.

6.12.2 check_model and check_model_with_options

pub fn check_model(model: &Model, theory: &Theory) -> Result<Vec<EquationViolation>, GatError>

pub fn check_model_with_options(
    model: &Model,
    theory: &Theory,
    options: &CheckModelOptions,
) -> Result<Vec<EquationViolation>, GatError>

For each equation, the checker infers variable sorts, builds the cartesian product of the carrier sets for those sorts, and evaluates both sides of the equation under every assignment. If lhs_value != rhs_value, the assignment is recorded as an EquationViolation.

6.12.3 EquationViolation

Each violation captures the equation name, the variable assignment that triggered it, and the values that the two sides evaluated to:

pub struct EquationViolation {
    pub equation: Arc<str>,
    pub assignment: FxHashMap<Arc<str>, ModelValue>,
    pub lhs_value: ModelValue,
    pub rhs_value: ModelValue,
}

This gives enough information for the CLI to print a precise diagnostic: “equation ‘assoc’ violated when a=2, b=3, c=1: LHS=6, RHS=4”.

6.12.4 CheckModelOptions

The CheckModelOptions struct provides safety bounds for enumeration. The max_assignments field (default: 10,000) caps the total number of variable assignments evaluated per equation. An equation with 3 variables over a carrier of 100 elements would require 1,000,000 assignments – the limit prevents this from running for minutes.

When the limit is exceeded, the function returns a GatError::ModelError rather than silently skipping the check. The VCS layer reports this as a non-blocking warning.

6.13 Pullbacks (Finding Shared Structure)

The pullback function computes the categorical pullback of two theories over a common codomain. Where colimits (pushouts) merge theories together, pullbacks find the common structure that two theories share when mapped into a common target.

6.13.1 Mathematical Background

Given morphisms m1: T1 -> C and m2: T2 -> C, the pullback theory P contains:

  • A sort for each pair (s1 in T1, s2 in T2) where m1(s1) = m2(s2) and s1, s2 have the same arity.
  • An operation for each pair (op1 in T1, op2 in T2) where m1(op1) = m2(op2) and their input and output sort pairs exist in the pullback.
  • An equation for each pair (eq1 in T1, eq2 in T2) where m1(eq1) = m2(eq2) (i.e., both equations map to the same codomain equation).

6.13.2 PullbackResult

pub struct PullbackResult {
    pub theory: Theory,
    pub proj1: TheoryMorphism,
    pub proj2: TheoryMorphism,
}

The result includes projection morphisms proj1: P -> T1 and proj2: P -> T2 satisfying the universal property: m1 . proj1 = m2 . proj2. Both projections are validated by check_morphism in the test suite.

6.13.3 Pullback Naming

When both sides contribute an element with the same name, the pullback element keeps that name. When the names differ, the pullback element is named "name1=name2". For example, pulling back theories where V1 and V2 both map to Vertex in the codomain produces a pullback sort named V1=V2.

6.13.4 Practical Use: Merge Overlap Detection

The VCS merge algorithm uses pullbacks to detect structural overlap between two branches. When merging branch A and branch B, the three-way merge computes the pullback of the two branch theories over the merge base theory. Non-empty pullback elements indicate shared structure that must be reconciled, while empty pullback means the branches modified disjoint parts of the schema.

6.14 Natural Transformations (Maps Between Mappings)

A natural transformation is a morphism between two theory morphisms. Given morphisms F, G: T1 -> T2, a natural transformation alpha: F => G assigns to each sort S in T1 a term that maps values of sort F(S) to values of sort G(S), subject to naturality conditions.

6.14.1 NaturalTransformation

pub struct NaturalTransformation {
    pub name: Arc<str>,
    pub source: Arc<str>,
    pub target: Arc<str>,
    pub components: HashMap<Arc<str>, Term>,
}

Each component is a Term with a single free variable "x" representing the input value. For the identity natural transformation, every component is Term::Var("x").

6.14.2 check_natural_transformation

pub fn check_natural_transformation(
    nt: &NaturalTransformation,
    f: &TheoryMorphism,
    g: &TheoryMorphism,
    domain: &Theory,
    codomain: &Theory,
) -> Result<(), GatError>

The checker verifies four conditions:

  1. Domain/codomain agreement: F and G must have the same domain and codomain theories.
  2. Component completeness: every sort in the domain must have a component.
  3. Component validity: all operations used in component terms must exist in the codomain theory.
  4. Naturality: for each operation op: S1 -> S2 in the domain, the naturality square must commute: alpha_S2[x := F(op)(x)] == G(op)(alpha_S1(x))

The naturality check is performed symbolically by substituting and comparing term trees. For n-ary operations, the check uses variables x0, x1, ... for each input.

6.14.3 Vertical Composition (Chaining Transforms)

vertical_compose chains two natural transformations: given alpha: F => G and beta: G => H, it produces beta . alpha: F => H. For each sort S, the composed component is beta_S[x := alpha_S(x)].

The function checks that alpha.target == beta.source (the intermediate morphism must match) and returns GatError::NatTransComposeMismatch if they disagree.

6.14.4 Horizontal Composition (Combining Independent Transforms)

horizontal_compose composes natural transformations along different morphism pairs: given alpha: F => G (between T1 and T2) and beta: H => K (between T2 and T3), it produces beta * alpha: H.F => K.G.

For each sort S in the domain, the composed component is beta_{G(S)}[x := H(alpha_S)], where G(S) is the sort that G maps S to, and H(alpha_S) is the result of applying H’s operation renaming to alpha’s component at S.

Note

Horizontal composition is used in the VCS merge pipeline when composing migration paths through different branch histories. The compose_path function in panproto-vcs assembles a chain of natural transformations to verify that the composed migration path is coherent.

6.15 Free Models (Minimal Valid Instances)

The free_model module constructs the free (initial) model of a theory – the minimal model that satisfies all equations. This is useful for generating test instances, computing protocol skeletons, and understanding the structure of a theory.

6.15.1 How Free Model Construction Works

The construction proceeds in three phases:

  1. Term generation: enumerate all closed terms (terms with no free variables) up to a configurable depth bound, organized by sort. Nullary operations (constants) provide the seed; non-nullary operations are applied to all combinations of existing terms of the correct sorts.

  2. Quotient by equations: use a union-find data structure to identify terms that must be equal according to the theory’s equations. For each equation, enumerate all valid variable substitutions (using the generated terms as values) and union the global indices of the LHS and RHS results.

  3. Build model: select one representative per equivalence class as the carrier set elements, and define operations by symbolic term application.

6.15.2 FreeModelConfig

pub struct FreeModelConfig {
    pub max_depth: usize,           // Default: 3
    pub max_terms_per_sort: usize,  // Default: 1000
}

The depth bound controls how deep the term enumeration goes. For a theory with a successor operation, depth 3 generates zero, succ(zero), succ(succ(zero)), and succ(succ(succ(zero))). The per-sort limit prevents combinatorial explosion in theories with binary operations.

6.15.3 Practical Examples

  • Pointed set (one sort, one constant): free model has exactly 1 element.
  • Two constants, no equations: free model has 2 elements.
  • Two constants with a = b equation: free model has 1 element (the equation collapses them).
  • Monoid with identity laws: at depth 1, the free model collapses mul(unit, x) and mul(x, unit) with x, leaving only the identity element.
  • Graph theory (no constants): free model has empty carrier sets for both Vertex and Edge, since there are no closed terms.

6.15.4 CLI Integration

The schema scaffold command uses free_model to generate skeleton instances. The --depth and --max-terms flags map directly to FreeModelConfig fields.

6.16 Quotient Theories (Identifying Equivalent Elements)

The quotient function simplifies a theory by identifying (merging) sorts and/or operations. This is the dual of theory extension: where extension adds new structure, quotienting collapses existing structure by declaring that distinct elements are the same.

6.16.1 How Quotienting Works

Given a theory and a list of identification pairs (a, b), the function:

  1. Classifies each pair as a sort identification or operation identification by checking whether the names exist as sorts or operations in the theory.
  2. Builds a union-find for sorts and a union-find for operations, computing the transitive closure of the identifications.
  3. Verifies compatibility: identified sorts must have the same arity, and identified operations must have the same signature after sort renaming.
  4. Rebuilds the theory with one representative per equivalence class. Sort parameters, operation input sorts, and equation terms are all renamed through the sort and operation rename maps.
  5. Deduplicates equations that become identical after renaming.

6.16.2 Error Conditions

  • GatError::QuotientIncompatible: the identified elements have incompatible structure (different arities or signatures).
  • GatError::SortNotFound / GatError::OpNotFound: a name referenced during rebuilding does not exist.

6.16.3 Practical Use

Quotient theories are used when composing protocol fragments that have structurally equivalent but differently-named elements. For example, if protocol A has sort Node and protocol B has sort Vertex, and both have the same role in the combined theory, quotienting by (Node, Vertex) produces a single sort (the alphabetically-first representative, Node).

The alphabetical representative selection is deterministic and stable, ensuring that quotienting the same theory with the same identifications always produces the same result.

Endofunctors compose via TheoryEndofunctor::compose, producing a new endofunctor whose transform is Compose(self.transform, other.transform). The precondition of the composed endofunctor conservatively requires the first endofunctor’s precondition and defers the second’s to runtime checking.

6.17.2 Factorization

File: crates/panproto-gat/src/factorize.rs

The factorize function decomposes a theory morphism \(F : T_1 \to T_2\) into a sequence of elementary TheoryEndofunctor steps. The decomposition follows a fixed phase ordering that maintains well-formedness at every intermediate step:

  1. Drops (equations, then operations, then sorts) — remove elements not in the codomain.
  2. Renames (sorts, then operations) — apply sort-map and op-map entries where old != new.
  3. Adds (sorts topologically sorted by dependencies, then operations, then equations) — add elements in the codomain but not in the (renamed) domain.

The Factorization struct holds the ordered step sequence. validate_factorization applies the steps to the domain and checks that the result matches the codomain, providing an end-to-end correctness guarantee.

This factorization is the foundation of the automated lens generation pipeline (Chapter 20): each elementary step maps directly to a protolens constructor, and the full step sequence becomes a ProtolensChain.

# panproto-gat -- The GAT Engine {#sec-gat-engine}

`panproto-gat` is Level 0 of the panproto crate hierarchy: the mathematical foundation on which everything else is built. This crate implements generalized algebraic theories (GATs) as defined by @cartmell1986, providing the data structures and algorithms for sorts, operations, equations, theory morphisms, and colimits.

This chapter walks through every major type and function in the crate, explaining both the Rust implementation and the mathematical concepts it encodes.

## Sorts and Sort Parameters

A _sort_ is a type in a GAT. Sorts may be _simple_ (zero parameters) or _dependent_ (parameterized by terms of other sorts). The distinction between simple and dependent sorts is what separates GATs from ordinary algebraic theories in the sense of @lawvere1963: in an ordinary algebraic theory, all sorts are simple.

### SortParam

A `SortParam` names a dependency of a sort on a term of another sort. For example, in the theory of categories, the hom-sort `Hom(a: Ob, b: Ob)` has two parameters, each of sort `Ob`.

### Sort

A `Sort` combines a name with zero or more `SortParam` entries. Simple sorts have an empty parameter list.

```{.rust include="../../crates/panproto-gat/src/sort.rs" start-line=1 end-line=38}
```

The `arity()` method returns the number of parameters -- this is the key invariant that theory morphisms must preserve.

::: {.callout-tip}
In panproto's schema theories, most sorts are simple (`Vertex`, `Edge`, `Constraint`). Dependent sorts appear when a schema theory needs to express constraints that depend on a vertex (e.g., `Constraint(v: Vertex)`), or when modeling categories where morphisms depend on their source and target objects.
:::

### Concrete Examples

Here are some sorts you will encounter in panproto protocol definitions:

- `Vertex` -- simple sort, arity 0. The vertices of a schema graph.
- `Edge` -- simple sort, arity 0. Directed edges between vertices.
- `Hom(a: Ob, b: Ob)` -- dependent sort, arity 2. Morphisms in a category.
- `Constraint(v: Vertex)` -- dependent sort, arity 1. Constraints attached to a specific vertex.

## Operations

An _operation_ (also called a term constructor) is a function in a GAT. Each operation has a name, typed inputs, and a typed output. Operations with zero inputs are constants (nullary constructors).

```{.rust include="../../crates/panproto-gat/src/op.rs" start-line=1 end-line=23}
```

The `inputs` field stores `(param_name, sort_name)` pairs. This is sufficient for first-order operations. Higher-order operations (operations whose inputs are themselves operations) are not directly supported -- panproto does not need them for its protocol modeling use case.

### Operation Signatures in Practice

In a graph schema theory, you would see operations like:

- `src: (e: Edge) -> Vertex` -- the source vertex of an edge.
- `tgt: (e: Edge) -> Vertex` -- the target vertex of an edge.
- `unit: () -> Carrier` -- a nullary operation (constant) in a monoid theory.

The type signature of an operation is the pair `(input_sorts, output_sort)`. Theory morphisms must preserve these signatures up to the sort mapping.

## Equations

Equations (axioms) constrain the behavior of operations. An equation consists of a name and two `Term` trees that must be considered equal. panproto uses a simple term language:

- `Term::Var(name)` -- a variable reference.
- `Term::App(op_name, args)` -- application of an operation to arguments.

Equations are not computationally evaluated; they serve as proof obligations that theory morphisms must preserve. When `panproto-gat` checks a morphism, it verifies that for each equation `lhs = rhs` in the domain theory, the mapped equation `F(lhs) = F(rhs)` exists in the codomain theory.

## Sort Kinds

Sorts are classified by their `SortKind`, which distinguishes structural sorts from value-level sorts:

- **`Structural`** (default): standard schema-level sorts like `Vertex`, `Edge`, `Constraint`.
- **`Val(kind)`**: value sorts carrying data of a specific `ValueKind` (Bool, Int, Float, Str, Bytes, Token, Null, Any).
- **`Coercion { from, to }`**: coercion sorts representing directed morphisms between value kinds.
- **`Merger(kind)`**: merger sorts that combine values of a specific kind.

Value sorts bridge the schema level and the instance level. A sort like `Val(Str)` classifies string values; `Coercion { from: Str, to: Int }` classifies functions that convert strings to integers (e.g., `parse_int`).

## Directed Equations

A `DirectedEquation` extends the standard `Equation` with a computation direction. Where an equation `lhs = rhs` is an undirected assertion, a directed equation `lhs ⟶ rhs` specifies a rewrite rule: when the migration engine encounters a value matching `lhs`, it rewrites to `rhs` using the implementation term.

Each directed equation carries:

- `impl_term`: a `panproto_expr::Expr` — the computable implementation of the rewrite.
- `inverse`: an optional `panproto_expr::Expr` for the backward (put) direction.

The expression language is defined in the `panproto-expr` crate — a pure functional language with lambda calculus, pattern matching, records, lists, and ~50 built-in operations.

## Conflict Policies

A `ConflictPolicy` specifies how merge conflicts are resolved for values of a given kind:

- `KeepLeft` — always prefer the left branch.
- `KeepRight` — always prefer the right branch.
- `Fail` — report a conflict (default behavior).
- `Custom(expr)` — evaluate a `panproto_expr::Expr` to decide.

## Theory

The `Theory` struct is the central type of `panproto-gat`. It groups sorts, operations, equations, directed equations, and conflict policies under a name, with optional inheritance via `extends`.

```{.rust include="../../crates/panproto-gat/src/theory.rs" start-line=10 end-line=44}
```

### Theory Inheritance

The `extends` field enables compositional theory construction. When theory B extends theory A, B inherits all of A's sorts, operations, and equations. The `resolve_theory` function computes the transitive closure of the inheritance chain, merging declarations and detecting cycles.

Diamond inheritance is handled correctly: if theories B and C both extend A, and theory D extends both B and C, the shared declarations from A appear exactly once in the resolved theory D.

For the conceptual introduction to GATs, see the [tutorial chapter 3](https://panproto.dev/tutorial/chapters/03-what-is-a-theory.html).

## Theory Morphisms

A _theory morphism_ is a structure-preserving map from one theory to another. It maps sorts to sorts and operations to operations, subject to three preservation conditions:

1. **Arity preservation**: if sort S has arity n, then F(S) must also have arity n.
2. **Signature preservation**: if operation `op: (a: A, b: B) -> C`, then `F(op): (a: F(A), b: F(B)) -> F(C)`.
3. **Equation preservation**: if `lhs = rhs` is an equation in the domain, then `F(lhs) = F(rhs)` must be an equation in the codomain.

```{.rust include="../../crates/panproto-gat/src/morphism.rs" start-line=8 end-line=24}
```

The `check_morphism` function validates all three conditions. This is the critical correctness check for schema migrations: a migration specification is valid if and only if it induces a valid theory morphism between the source and target schema theories.

### Why Morphisms Matter for Migration

When you migrate data from schema v1 to schema v2, the migration is mathematically a theory morphism from v2's schema theory to v1's schema theory (note the direction reversal -- this is the contravariance of restriction functors). The `CompiledMigration` in `panproto-inst` is the executable form of this morphism, and the `wtype_restrict` pipeline is the functor it induces on instances.

This is the fundamental guarantee that makes panproto migrations principled: because the restrict operation is functorial, composing two migrations and restricting is the same as restricting twice [@spivak2012]. There are no "migration bugs" that arise from order-of-operations issues.

### Inducing Schema Renames

Theory morphisms induce schema-level renames via `induce_schema_renames()`. Sort-map entries where `old ≠ new` become `NameSite::VertexKind` renames (since sorts map to vertex kinds at the schema level), and op-map entries where `old ≠ new` become `NameSite::EdgeKind` renames. This is the top of the _morphism tower_ described in @sec-naming-identity-section.

## Identity and Naming

The `ident` module provides three types that separate identity from display name:

- **`ScopeTag`**: An opaque scope tag distinguishing different naming contexts. Generated via `ScopeTag::fresh()`.
- **`Ident`**: A first-class identifier where equality uses `(scope, index)` only, not the display name. Follows `GATlab` [@lynch2024].
- **`Name`**: An interned string handle (`Arc<str>`) with a pointer-equality fast path. Drop-in replacement for `String` in hot-path types.

The `NameSite` enum enumerates the nine naming sites (edge labels, vertex IDs, vertex kinds, edge kinds, NSIDs, constraint sorts, instance anchors, theory names, sort names). `SiteRename` is a site-qualified rename operation used in schema morphism provenance and VCS commit metadata.

See @sec-naming-identity-section for the full contributor guide.

### Conflict Detection

The colimit function detects three kinds of conflicts:

- **Sort conflicts**: T1 and T2 both declare a sort with the same name but different parameter lists, and the sort is not in the shared base.
- **Operation conflicts**: same-named operations with incompatible signatures.
- **Equation conflicts**: same-named equations with different term content.

Compatible duplicates (same name, same definition) are silently deduplicated. This allows independent protocol extensions to share common vocabulary without coordination, as long as they agree on the meaning of shared names.

### Practical Use: Composing Schema Theories

The `panproto-protocols` crate uses colimits extensively. For example, the AT Protocol schema theory is built by combining a base graph theory (`ThGraph` with Vertex, Edge, src, tgt) with a constraint theory (`ThConstraint` with Constraint, target) over their shared `ThVertex`. The colimit produces a theory with all three sorts and all three operations, ready to serve as the schema theory for the AT Protocol.

## Type-Checking

The `typecheck` module provides verification that terms and equations are well-typed with respect to a theory's operation signatures. Type-checking catches malformed equations, unsound morphisms, and invalid term constructions at construction time rather than silently producing wrong results downstream.

### typecheck_term

The core function `typecheck_term` infers the output sort of a term given a variable typing context and a theory. It walks the term tree recursively:

- For `Term::Var(x)`: returns the sort from the context, or `GatError::UnboundVariable`.
- For `Term::App { op, args }`: looks up `op` in the theory, recursively typechecks each argument, verifies that each argument's sort matches the operation's input signature, and returns the output sort.

```rust
pub fn typecheck_term(
    term: &Term,
    ctx: &VarContext,
    theory: &Theory,
) -> Result<Arc<str>, GatError>
```

The `VarContext` type is a `FxHashMap<Arc<str>, Arc<str>>` mapping variable names to their sort names. Callers construct a context manually when type-checking a standalone term, but more commonly the context is inferred automatically by `infer_var_sorts`.

### infer_var_sorts

`infer_var_sorts` walks both sides of an equation, collecting sort constraints from every operation application site. A variable used as argument `i` of operation `op` must have the sort declared for that input position. If the same variable appears at two positions requiring different sorts, the function returns `GatError::ConflictingVarSort`.

```rust
pub fn infer_var_sorts(eq: &Equation, theory: &Theory) -> Result<VarContext, GatError>
```

This function is the entry point for automatic variable sort inference, used by `typecheck_equation`, `check_model`, and `free_model`. It eliminates the need for explicit sort annotations on equation variables.

### typecheck_equation and typecheck_theory

`typecheck_equation` combines inference and checking: it infers variable sorts from the equation structure, typechecks both sides, and verifies they produce the same output sort. If the two sides have different sorts, it returns `GatError::EquationSortMismatch`.

`typecheck_theory` iterates over all equations in a theory and typechecks each one, returning the first error encountered. This is the function called during VCS validation to ensure that a protocol's schema theory is internally consistent.

```rust
pub fn typecheck_theory(theory: &Theory) -> Result<(), GatError>
```

::: {.callout-tip}
Type-checking is fast -- it is linear in the size of the term tree for each equation. Even theories with hundreds of equations typecheck in microseconds. The VCS pipeline runs `typecheck_theory` on every `schema add` and `schema commit` without measurable overhead.
:::

## Model Checking

The `check_model` module verifies that a `Model` (a concrete instance that satisfies a theory's requirements --- like a Rust struct implementing a trait --- here specifically an interpretation of a theory in Set) satisfies all equations of its theory by enumerating variable assignments from carrier sets and evaluating both sides.

### Model and ModelValue

A `Model` provides an interpretation of a theory: each sort maps to a carrier set (a `Vec<ModelValue>`), and each operation maps to a Rust closure that takes `ModelValue` arguments and returns a `ModelValue`. `ModelValue` is an enum with `Int(i64)` and `Str(String)` variants -- enough for the finite models that panproto needs for equation verification.

### check_model and check_model_with_options

```rust
pub fn check_model(model: &Model, theory: &Theory) -> Result<Vec<EquationViolation>, GatError>

pub fn check_model_with_options(
    model: &Model,
    theory: &Theory,
    options: &CheckModelOptions,
) -> Result<Vec<EquationViolation>, GatError>
```

For each equation, the checker infers variable sorts, builds the cartesian product of the carrier sets for those sorts, and evaluates both sides of the equation under every assignment. If `lhs_value != rhs_value`, the assignment is recorded as an `EquationViolation`.

### EquationViolation

Each violation captures the equation name, the variable assignment that triggered it, and the values that the two sides evaluated to:

```rust
pub struct EquationViolation {
    pub equation: Arc<str>,
    pub assignment: FxHashMap<Arc<str>, ModelValue>,
    pub lhs_value: ModelValue,
    pub rhs_value: ModelValue,
}
```

This gives enough information for the CLI to print a precise diagnostic: "equation 'assoc' violated when a=2, b=3, c=1: LHS=6, RHS=4".

### CheckModelOptions

The `CheckModelOptions` struct provides safety bounds for enumeration. The `max_assignments` field (default: 10,000) caps the total number of variable assignments evaluated per equation. An equation with 3 variables over a carrier of 100 elements would require 1,000,000 assignments -- the limit prevents this from running for minutes.

When the limit is exceeded, the function returns a `GatError::ModelError` rather than silently skipping the check. The VCS layer reports this as a non-blocking warning.

## Pullbacks (Finding Shared Structure)

The `pullback` function computes the categorical pullback of two theories over a common codomain. Where colimits (pushouts) merge theories together, pullbacks find the common structure that two theories share when mapped into a common target.

### Mathematical Background

Given morphisms `m1: T1 -> C` and `m2: T2 -> C`, the pullback theory `P` contains:

- A sort for each pair `(s1 in T1, s2 in T2)` where `m1(s1) = m2(s2)` and `s1`, `s2` have the same arity.
- An operation for each pair `(op1 in T1, op2 in T2)` where `m1(op1) = m2(op2)` and their input and output sort pairs exist in the pullback.
- An equation for each pair `(eq1 in T1, eq2 in T2)` where `m1(eq1) = m2(eq2)` (i.e., both equations map to the same codomain equation).

### PullbackResult

```rust
pub struct PullbackResult {
    pub theory: Theory,
    pub proj1: TheoryMorphism,
    pub proj2: TheoryMorphism,
}
```

The result includes projection morphisms `proj1: P -> T1` and `proj2: P -> T2` satisfying the universal property: `m1 . proj1 = m2 . proj2`. Both projections are validated by `check_morphism` in the test suite.

### Pullback Naming

When both sides contribute an element with the same name, the pullback element keeps that name. When the names differ, the pullback element is named `"name1=name2"`. For example, pulling back theories where `V1` and `V2` both map to `Vertex` in the codomain produces a pullback sort named `V1=V2`.

### Practical Use: Merge Overlap Detection

The VCS merge algorithm uses pullbacks to detect structural overlap between two branches. When merging branch A and branch B, the three-way merge computes the pullback of the two branch theories over the merge base theory. Non-empty pullback elements indicate shared structure that must be reconciled, while empty pullback means the branches modified disjoint parts of the schema.

## Natural Transformations (Maps Between Mappings)

A _natural transformation_ is a morphism between two theory morphisms. Given morphisms `F, G: T1 -> T2`, a natural transformation `alpha: F => G` assigns to each sort `S` in `T1` a term that maps values of sort `F(S)` to values of sort `G(S)`, subject to naturality conditions.

### NaturalTransformation

```rust
pub struct NaturalTransformation {
    pub name: Arc<str>,
    pub source: Arc<str>,
    pub target: Arc<str>,
    pub components: HashMap<Arc<str>, Term>,
}
```

Each component is a `Term` with a single free variable `"x"` representing the input value. For the identity natural transformation, every component is `Term::Var("x")`.

### check_natural_transformation

```rust
pub fn check_natural_transformation(
    nt: &NaturalTransformation,
    f: &TheoryMorphism,
    g: &TheoryMorphism,
    domain: &Theory,
    codomain: &Theory,
) -> Result<(), GatError>
```

The checker verifies four conditions:

1. **Domain/codomain agreement**: `F` and `G` must have the same domain and codomain theories.
2. **Component completeness**: every sort in the domain must have a component.
3. **Component validity**: all operations used in component terms must exist in the codomain theory.
4. **Naturality**: for each operation `op: S1 -> S2` in the domain, the naturality square must commute:
   `alpha_S2[x := F(op)(x)] == G(op)(alpha_S1(x))`

The naturality check is performed symbolically by substituting and comparing term trees. For n-ary operations, the check uses variables `x0, x1, ...` for each input.

### Vertical Composition (Chaining Transforms)

`vertical_compose` chains two natural transformations: given `alpha: F => G` and `beta: G => H`, it produces `beta . alpha: F => H`. For each sort `S`, the composed component is `beta_S[x := alpha_S(x)]`.

The function checks that `alpha.target == beta.source` (the intermediate morphism must match) and returns `GatError::NatTransComposeMismatch` if they disagree.

### Horizontal Composition (Combining Independent Transforms)

`horizontal_compose` composes natural transformations along different morphism pairs: given `alpha: F => G` (between `T1` and `T2`) and `beta: H => K` (between `T2` and `T3`), it produces `beta * alpha: H.F => K.G`.

For each sort `S` in the domain, the composed component is `beta_{G(S)}[x := H(alpha_S)]`, where `G(S)` is the sort that `G` maps `S` to, and `H(alpha_S)` is the result of applying `H`'s operation renaming to `alpha`'s component at `S`.

::: {.callout-note}
Horizontal composition is used in the VCS merge pipeline when composing migration paths through different branch histories. The `compose_path` function in `panproto-vcs` assembles a chain of natural transformations to verify that the composed migration path is coherent.
:::

## Free Models (Minimal Valid Instances)

The `free_model` module constructs the free (initial) model of a theory -- the minimal model that satisfies all equations. This is useful for generating test instances, computing protocol skeletons, and understanding the structure of a theory.

### How Free Model Construction Works

The construction proceeds in three phases:

1. **Term generation**: enumerate all closed terms (terms with no free variables) up to a configurable depth bound, organized by sort. Nullary operations (constants) provide the seed; non-nullary operations are applied to all combinations of existing terms of the correct sorts.

2. **Quotient by equations**: use a union-find data structure to identify terms that must be equal according to the theory's equations. For each equation, enumerate all valid variable substitutions (using the generated terms as values) and union the global indices of the LHS and RHS results.

3. **Build model**: select one representative per equivalence class as the carrier set elements, and define operations by symbolic term application.

### FreeModelConfig

```rust
pub struct FreeModelConfig {
    pub max_depth: usize,           // Default: 3
    pub max_terms_per_sort: usize,  // Default: 1000
}
```

The depth bound controls how deep the term enumeration goes. For a theory with a successor operation, depth 3 generates `zero`, `succ(zero)`, `succ(succ(zero))`, and `succ(succ(succ(zero)))`. The per-sort limit prevents combinatorial explosion in theories with binary operations.

### Practical Examples

- **Pointed set** (one sort, one constant): free model has exactly 1 element.
- **Two constants, no equations**: free model has 2 elements.
- **Two constants with `a = b` equation**: free model has 1 element (the equation collapses them).
- **Monoid with identity laws**: at depth 1, the free model collapses `mul(unit, x)` and `mul(x, unit)` with `x`, leaving only the identity element.
- **Graph theory** (no constants): free model has empty carrier sets for both Vertex and Edge, since there are no closed terms.

### CLI Integration

The `schema scaffold` command uses `free_model` to generate skeleton instances. The `--depth` and `--max-terms` flags map directly to `FreeModelConfig` fields.

## Quotient Theories (Identifying Equivalent Elements)

The `quotient` function simplifies a theory by identifying (merging) sorts and/or operations. This is the dual of theory extension: where extension adds new structure, quotienting collapses existing structure by declaring that distinct elements are the same.

### How Quotienting Works

Given a theory and a list of identification pairs `(a, b)`, the function:

1. Classifies each pair as a sort identification or operation identification by checking whether the names exist as sorts or operations in the theory.
2. Builds a union-find for sorts and a union-find for operations, computing the transitive closure of the identifications.
3. Verifies compatibility: identified sorts must have the same arity, and identified operations must have the same signature after sort renaming.
4. Rebuilds the theory with one representative per equivalence class. Sort parameters, operation input sorts, and equation terms are all renamed through the sort and operation rename maps.
5. Deduplicates equations that become identical after renaming.

### Error Conditions

- `GatError::QuotientIncompatible`: the identified elements have incompatible structure (different arities or signatures).
- `GatError::SortNotFound` / `GatError::OpNotFound`: a name referenced during rebuilding does not exist.

### Practical Use

Quotient theories are used when composing protocol fragments that have structurally equivalent but differently-named elements. For example, if protocol A has sort `Node` and protocol B has sort `Vertex`, and both have the same role in the combined theory, quotienting by `(Node, Vertex)` produces a single sort (the alphabetically-first representative, `Node`).

The alphabetical representative selection is deterministic and stable, ensuring that quotienting the same theory with the same identifications always produces the same result.

## Schema Transforms and Factorization {#sec-schema-endofunctors}

The `schema_functor` and `factorize` modules extend `panproto-gat` with theory endofunctors and morphism decomposition, providing the GAT-level machinery that the protolens engine (@sec-protolens-engine) builds on.

### Schema Endofunctors

**File**: `crates/panproto-gat/src/schema_functor.rs`

A `TheoryEndofunctor` is a map from theories to theories, pairing a `TheoryConstraint` (a predicate specifying which theories it applies to) with a `TheoryTransform` (the actual transformation). The `TheoryTransform` enum provides eleven variants covering sort/operation/equation addition, removal, renaming, pullback, and sequential composition.

```{.rust include="../../crates/panproto-gat/src/schema_functor.rs" start-line=69 end-line=77}
```

The key design principle is *cascading*: dropping a sort automatically drops all operations that reference it, and dropping an operation automatically drops all equations that reference it. This ensures that `TheoryTransform::apply` always produces a well-formed theory.

Endofunctors compose via `TheoryEndofunctor::compose`, producing a new endofunctor whose transform is `Compose(self.transform, other.transform)`. The precondition of the composed endofunctor conservatively requires the first endofunctor's precondition and defers the second's to runtime checking.

### Factorization

**File**: `crates/panproto-gat/src/factorize.rs`

The `factorize` function decomposes a theory morphism $F : T_1 \to T_2$ into a sequence of elementary `TheoryEndofunctor` steps. The decomposition follows a fixed phase ordering that maintains well-formedness at every intermediate step:

1. **Drops** (equations, then operations, then sorts) --- remove elements not in the codomain.
2. **Renames** (sorts, then operations) --- apply sort-map and op-map entries where `old != new`.
3. **Adds** (sorts topologically sorted by dependencies, then operations, then equations) --- add elements in the codomain but not in the (renamed) domain.

The `Factorization` struct holds the ordered step sequence. `validate_factorization` applies the steps to the domain and checks that the result matches the codomain, providing an end-to-end correctness guarantee.

This factorization is the foundation of the automated lens generation pipeline (@sec-auto-lens-pipeline): each elementary step maps directly to a protolens constructor, and the full step sequence becomes a `ProtolensChain`.
