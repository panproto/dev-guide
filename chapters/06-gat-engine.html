6  panproto-gat: The GAT Engine

A sort is a type in a GAT. An operation is a function between sorts. An equation constrains operations. The Rust representations follow. In the theory of categories, for instance, we have sorts \(\mathrm{Ob}\) and \(\mathrm{Hom}(a: \mathrm{Ob}, b: \mathrm{Ob})\), operations \(\mathrm{id}: \mathrm{Ob} \to \mathrm{Hom}(a, a)\) and \(\mathrm{comp}: \mathrm{Hom}(a, b) \times \mathrm{Hom}(b, c) \to \mathrm{Hom}(a, c)\), and equations for associativity and unitality.

This crate is Level 0 of the panproto hierarchy: the algebraic foundation on which everything else is built. It implements GATs as defined by Cartmell (1986).

6.1 Sorts and Sort Parameters

6.1.1 Math

A sort \(S\) in a GAT has arity \(n \geq 0\). When \(n = 0\), \(S\) is a simple sort. When \(n > 0\), \(S\) is a dependent sort parameterized by terms of other sorts. The distinction between simple and dependent sorts is what separates GATs from ordinary algebraic theories in the sense of Lawvere (1963).

A sort parameter names a dependency of a sort on a term of another sort. For example, \(\mathrm{Hom}(a: \mathrm{Ob}, b: \mathrm{Ob})\) has two parameters, each of sort \(\mathrm{Ob}\).

6.1.2 Rust

use std::sync::Arc;

/// The primitive value kind that a value sort ranges over.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub enum ValueKind {
    /// Boolean values.
    Bool,
    /// Integer values.
    Int,
    /// Floating-point values.
    Float,
    /// String values.
    Str,
    /// Byte-sequence values.
    Bytes,
    /// Opaque token values.
    Token,
    /// Null / unit values.
    Null,
    /// Any value kind (polymorphic).
    Any,
}

/// The kind of a sort, distinguishing structural sorts from value/coercion sorts.
#[derive(Debug, Clone, Default, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub enum SortKind {
    /// Standard structural sort (vertices, edges, constraints).
    #[default]
    Structural,
    /// Value sort: carries data of a specific kind.
    Val(ValueKind),
    /// Coercion sort: a directed morphism between value kinds.
    Coercion {
        /// The source value kind.
        from: ValueKind,
        /// The target value kind.
        to: ValueKind,
    },

The arity() method returns the number of parameters. This is the key invariant that theory morphisms must preserve.

6.1.3 Concrete Examples

Sorts you’ll encounter in panproto protocol definitions:

  • Vertex: simple sort, arity 0. The vertices of a schema graph.
  • Edge: simple sort, arity 0. Directed edges between vertices.
  • Hom(a: Ob, b: Ob): dependent sort, arity 2. Morphisms in a category.
  • Constraint(v: Vertex): dependent sort, arity 1. Constraints attached to a specific vertex.
Tip

In panproto’s schema theories, most sorts are simple (Vertex, Edge, Constraint). Dependent sorts appear when a schema theory needs to express constraints that depend on a vertex (e.g., Constraint(v: Vertex)), or when modeling categories where morphisms depend on their source and target objects.

6.2 Operations

6.2.1 Math

An operation \(f: (a_1: S_1, \ldots, a_n: S_n) \to T\) is a function in a GAT. Each operation has typed inputs and a typed output. Operations with zero inputs are constants (nullary constructors). The type signature of an operation is the pair \((\langle S_1, \ldots, S_n \rangle, T)\). Theory morphisms must preserve these signatures up to the sort mapping.

6.2.2 Rust

use std::sync::Arc;

/// An operation (term constructor) in a GAT.
///
/// Operations are the functions/constructors of a GAT. Each operation
/// has typed inputs and a typed output, where types are sort names.
///
/// # Examples
///
/// - `src: Edge → Vertex` (graph source map)
/// - `add: (a: Int, b: Int) → Int` (integer addition)
/// - `id: (x: Ob) → Hom(x, x)` (identity morphism)
///
/// Based on the formal definition of GAT operations from Cartmell (1986).
#[derive(Debug, Clone, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub struct Operation {
    /// The operation name (e.g., "src", "tgt", "compose").
    pub name: Arc<str>,
    /// Typed inputs as `(param_name, sort_name)` pairs.
    pub inputs: Vec<(Arc<str>, Arc<str>)>,
    /// The output sort name.
    pub output: Arc<str>,
}

The inputs field stores (param_name, sort_name) pairs. This is sufficient for first-order operations. Higher-order operations (operations whose inputs are themselves operations) aren’t directly supported; panproto doesn’t need them for its protocol modeling use case.

6.2.3 Concrete Examples

In a graph schema theory:

  • src: (e: Edge) -> Vertex: the source vertex of an edge.
  • tgt: (e: Edge) -> Vertex: the target vertex of an edge.
  • unit: () -> Carrier: a nullary operation (constant) in a monoid theory.

6.3 Equations

6.3.1 Math

An equation (axiom) in a GAT asserts that two terms are equal: \(\ell = r\) where \(\ell\) and \(r\) are built from variables and operation applications. Equations constrain the behavior of operations. They aren’t computationally evaluated; they serve as proof obligations that theory morphisms must preserve.

6.3.2 Rust

panproto uses a simple term language:

  • Term::Var(name): a variable reference.
  • Term::App(op_name, args): application of an operation to arguments.

When panproto-gat checks a morphism, it verifies that for each equation \(\ell = r\) in the domain theory, the mapped equation \(F(\ell) = F(r)\) exists in the codomain theory.

6.4 Sort Kinds

6.4.1 Math

Sorts are classified by a kind that distinguishes structural sorts from value-level sorts. The classification forms a sum type: \(\mathrm{SortKind} = \mathrm{Structural} \mid \mathrm{Val}(k) \mid \mathrm{Coercion}(k_1, k_2) \mid \mathrm{Merger}(k)\) where \(k, k_1, k_2 \in \mathrm{ValueKind}\).

6.4.2 Rust

  • Structural (default): standard schema-level sorts like Vertex, Edge, Constraint.
  • Val(kind): value sorts carrying data of a specific ValueKind (Bool, Int, Float, Str, Bytes, Token, Null, Any).
  • Coercion { from, to }: coercion sorts representing directed morphisms between value kinds.
  • Merger(kind): merger sorts that combine values of a specific kind.

6.4.3 Concrete Example

Value sorts bridge the schema level and the instance level. A sort like Val(Str) classifies string values. Coercion { from: Str, to: Int } classifies functions that convert strings to integers (e.g., parse_int).

6.5 Directed Equations

6.5.1 Math

A directed equation \(\ell \longrightarrow r\) extends the standard equation with a computation direction. Where an equation \(\ell = r\) is an undirected assertion, a directed equation specifies a rewrite rule: when the migration engine encounters a value matching \(\ell\), it rewrites to \(r\) using the implementation term.

6.5.2 Rust

Each directed equation carries:

  • impl_term: a panproto_expr::Expr, the computable implementation of the rewrite.
  • inverse: an optional panproto_expr::Expr for the backward (put) direction.

The expression language is defined in the panproto-expr crate, a pure functional language with lambda calculus, pattern matching, records, lists, and approximately 50 built-in operations.

6.6 Conflict Policies

A ConflictPolicy specifies how merge conflicts are resolved for values of a given kind:

  • KeepLeft: always prefer the left branch.
  • KeepRight: always prefer the right branch.
  • Fail: report a conflict (default behavior).
  • Custom(expr): evaluate a panproto_expr::Expr to decide.

6.7 Theory

6.7.1 Math

A theory \(\mathcal{T} = (S, \Omega, E)\) consists of a set of sorts \(S\), a set of operations \(\Omega\) (each with typed inputs and output drawn from \(S\)), and a set of equations \(E\) over terms built from \(\Omega\). We extend this with directed equations, conflict policies, and an inheritance relation.

6.7.2 Rust

/// A conflict resolution strategy for merge operations.
#[derive(Debug, Clone, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub enum ConflictStrategy {
    /// Keep the left (ours) value.
    KeepLeft,
    /// Keep the right (theirs) value.
    KeepRight,
    /// Fail on conflict.
    Fail,
    /// Custom resolution via an expression.
    Custom(panproto_expr::Expr),
}

/// A conflict resolution policy for merge operations.
#[derive(Debug, Clone, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub struct ConflictPolicy {
    /// A human-readable name for this policy.
    pub name: Arc<str>,
    /// The value kind this policy applies to.
    pub value_kind: ValueKind,
    /// The strategy to use when values conflict.
    pub strategy: ConflictStrategy,
}

/// A generalized algebraic theory (GAT).
///
/// Theories are named collections of sorts, operations, and equations,
/// with optional inheritance via `extends`. When a theory extends another,
/// it inherits all of the parent's sorts, operations, and equations.

6.7.3 Theory Inheritance

The extends field enables compositional theory construction. When theory B extends theory A, B inherits all of A’s sorts, operations, and equations. The resolve_theory function computes the transitive closure of the inheritance chain, merging declarations and detecting cycles.

Diamond inheritance is handled correctly. If theories B and C both extend A, and theory D extends both B and C, the shared declarations from A appear exactly once in the resolved theory D.

Note

For the conceptual introduction to GATs, see the tutorial chapter 3.

CautionWhen should you use theory inheritance vs. colimits?

Use extends when the child genuinely specializes the parent (adding sorts or operations to an existing vocabulary). Use colimits when composing two independent theory fragments that share a common sub-theory. The rule of thumb: if the shared structure is “accidental” (same name, same role, but developed independently), use a colimit. If it’s intentional (the child is meant to extend the parent), use extends.

6.8 Theory Morphisms

6.8.1 Math

A theory morphism \(F: \mathcal{T}_1 \to \mathcal{T}_2\) is a structure-preserving map. It maps sorts to sorts and operations to operations, subject to three preservation conditions:

  1. Arity preservation: if sort \(S\) has arity \(n\), then \(F(S)\) must also have arity \(n\).
  2. Signature preservation: if operation \(f: (a: A, b: B) \to C\), then \(F(f): (a: F(A), b: F(B)) \to F(C)\).
  3. Equation preservation: if \(\ell = r\) is an equation in \(\mathcal{T}_1\), then \(F(\ell) = F(r)\) must be an equation in \(\mathcal{T}_2\).

6.8.2 Rust


/// A structure-preserving map between two theories.
///
/// Maps sorts to sorts and operations to operations. A valid morphism
/// must preserve sort arities, operation type signatures, and equations.
#[derive(Debug, Clone, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
pub struct TheoryMorphism {
    /// A human-readable name for this morphism.
    pub name: Arc<str>,
    /// The name of the domain theory.
    pub domain: Arc<str>,
    /// The name of the codomain theory.
    pub codomain: Arc<str>,
    /// Mapping from domain sort names to codomain sort names.
    pub sort_map: HashMap<Arc<str>, Arc<str>>,
    /// Mapping from domain operation names to codomain operation names.
    pub op_map: HashMap<Arc<str>, Arc<str>>,

The check_morphism function validates all three conditions. This is the critical correctness check for schema migrations: a migration specification is valid if and only if it induces a valid theory morphism between the source and target schema theories.

6.8.3 Why Morphisms Matter for Migration

When we migrate data from schema v1 to schema v2, the migration is mathematically a theory morphism from v2’s schema theory to v1’s schema theory (note the direction reversal; this is the contravariance of restriction functors). The CompiledMigration in panproto-inst is the executable form of this morphism. The wtype_restrict pipeline is the functor it induces on instances.

This is the fundamental guarantee. Because the restrict operation is functorial, composing two migrations and restricting is the same as restricting twice (Spivak 2012). There are no “migration bugs” that arise from order-of-operations issues.

6.8.4 Inducing Schema Renames

Theory morphisms induce schema-level renames via induce_schema_renames(). Sort-map entries where old ≠ new become NameSite::VertexKind renames (since sorts map to vertex kinds at the schema level). Op-map entries where old ≠ new become NameSite::EdgeKind renames. This is the top of the morphism tower described in Section 18.2.

6.9 Identity and Naming

The ident module provides three types that separate identity from display name:

  • ScopeTag: an opaque scope tag distinguishing different naming contexts, generated via ScopeTag::fresh().
  • Ident: a first-class identifier where equality uses (scope, index) only, not the display name. Follows GATlab (Lynch et al. 2024).
  • Name: an interned string handle (Arc<str>) with a pointer-equality fast path. Drop-in replacement for String in hot-path types.

The NameSite enum enumerates the nine naming sites (edge labels, vertex IDs, vertex kinds, edge kinds, NSIDs, constraint sorts, instance anchors, theory names, sort names). SiteRename is a site-qualified rename operation used in schema morphism provenance and VCS commit metadata.

See Section 18.2 for the full contributor guide.

6.10 Colimits (Merging Theories)

6.10.1 Math

The colimit (pushout) of two theories \(T_1\) and \(T_2\) over a shared base \(S\) is a combined theory \(T_1 +_S T_2\) containing all declarations from both, with the shared declarations identified (not duplicated). Formally, given inclusions \(i_1: S \hookrightarrow T_1\) and \(i_2: S \hookrightarrow T_2\), the pushout is the universal cocone.

6.10.2 Rust

use crate::morphism::TheoryMorphism;
use crate::theory::Theory;

/// Result of a categorical pushout (colimit) computation.
///
/// Contains the pushout theory along with inclusion morphisms from
/// both input theories into the pushout. The cocone condition
/// `j1 ∘ i1 = j2 ∘ i2` is verified at construction time.
#[derive(Debug, Clone)]
pub struct ColimitResult {
    /// The pushout theory.
    pub theory: Theory,
    /// Inclusion morphism from the first theory into the pushout: j1: T1 → P.
    pub inclusion1: TheoryMorphism,
    /// Inclusion morphism from the second theory into the pushout: j2: T2 → P.
    pub inclusion2: TheoryMorphism,
}

6.10.4 Conflict Detection

The colimit function detects three kinds of conflicts:

  • Sort conflicts: \(T_1\) and \(T_2\) both declare a sort with the same name but different parameter lists, and the sort is not in the shared base.
  • Operation conflicts: same-named operations with incompatible signatures.
  • Equation conflicts: same-named equations with different term content.

Compatible duplicates (same name, same definition) are silently deduplicated. This allows independent protocol extensions to share common vocabulary without coordination, as long as they agree on the meaning of shared names.

6.10.5 Concrete Example: Composing Schema Theories

The panproto-protocols crate uses colimits extensively. The AT Protocol schema theory is built by combining a base graph theory (ThGraph with Vertex, Edge, src, tgt) with a constraint theory (ThConstraint with Constraint, target) over their shared ThVertex. The colimit produces a theory with all three sorts and all three operations, ready to serve as the schema theory for the AT Protocol.

CautionWhat happens when two protocol fragments define the same sort name with different arities?

The colimit function returns a GatError::SortConflict. You must either rename one of the sorts before merging or use a quotient to explicitly identify them. There’s no implicit resolution.

6.11 Type-Checking

6.11.1 Math

Type-checking verifies that terms and equations are well-typed with respect to a theory’s operation signatures. For a term \(f(t_1, \ldots, t_n)\) where \(f: (a_1: S_1, \ldots, a_n: S_n) \to T\), type-checking confirms that each \(t_i\) has sort \(S_i\) and infers the result sort \(T\).

6.11.2 Rust

The core function typecheck_term infers the output sort of a term given a variable typing context and a theory. It walks the term tree recursively:

  • For Term::Var(x): returns the sort from the context, or GatError::UnboundVariable.
  • For Term::App { op, args }: looks up op in the theory, recursively typechecks each argument, verifies that each argument’s sort matches the operation’s input signature, and returns the output sort.
pub fn typecheck_term(
    term: &Term,
    ctx: &VarContext,
    theory: &Theory,
) -> Result<Arc<str>, GatError>

The VarContext type is a FxHashMap<Arc<str>, Arc<str>> mapping variable names to their sort names. Callers construct a context manually when type-checking a standalone term, but more commonly the context is inferred automatically by infer_var_sorts.

6.11.3 infer_var_sorts

infer_var_sorts walks both sides of an equation, collecting sort constraints from every operation application site. A variable used as argument \(i\) of operation \(f\) must have the sort declared for that input position. If the same variable appears at two positions requiring different sorts, the function returns GatError::ConflictingVarSort.

pub fn infer_var_sorts(eq: &Equation, theory: &Theory) -> Result<VarContext, GatError>

This function is the entry point for automatic variable sort inference. It’s used by typecheck_equation, check_model, and free_model. It eliminates the need for explicit sort annotations on equation variables.

6.11.4 typecheck_equation and typecheck_theory

typecheck_equation combines inference and checking: it infers variable sorts from the equation structure, typechecks both sides, and verifies they produce the same output sort. If the two sides have different sorts, it returns GatError::EquationSortMismatch.

typecheck_theory iterates over all equations in a theory and typechecks each one, returning the first error encountered. This is the function called during VCS validation to ensure that a protocol’s schema theory is internally consistent.

pub fn typecheck_theory(theory: &Theory) -> Result<(), GatError>
Tip

Type-checking is fast. It’s linear in the size of the term tree for each equation. Even theories with hundreds of equations typecheck in microseconds. The VCS pipeline runs typecheck_theory on every schema add and schema commit without measurable overhead.

6.12 Model Checking

6.12.1 Math

A model \(M\) of a theory \(\mathcal{T}\) is an interpretation: each sort \(S\) maps to a carrier set \(M(S)\), and each operation \(f: S_1 \times \cdots \times S_n \to T\) maps to a function \(M(f): M(S_1) \times \cdots \times M(S_n) \to M(T)\). The model satisfies \(\mathcal{T}\) if every equation \(\ell = r\) evaluates to equal values under all variable assignments drawn from the carrier sets.

6.12.2 Rust

A Model provides an interpretation of a theory: each sort maps to a carrier set (a Vec<ModelValue>), and each operation maps to a Rust closure. ModelValue is an enum with Int(i64) and Str(String) variants, enough for the finite models that panproto needs for equation verification.

pub fn check_model(model: &Model, theory: &Theory) -> Result<Vec<EquationViolation>, GatError>

pub fn check_model_with_options(
    model: &Model,
    theory: &Theory,
    options: &CheckModelOptions,
) -> Result<Vec<EquationViolation>, GatError>

For each equation, the checker infers variable sorts, builds the cartesian product of the carrier sets for those sorts, and evaluates both sides under every assignment. If lhs_value != rhs_value, the assignment is recorded as an EquationViolation.

6.12.3 EquationViolation

Each violation captures the equation name, the variable assignment that triggered it, and the values that the two sides evaluated to:

pub struct EquationViolation {
    pub equation: Arc<str>,
    pub assignment: FxHashMap<Arc<str>, ModelValue>,
    pub lhs_value: ModelValue,
    pub rhs_value: ModelValue,
}

This gives enough information for the CLI to print a precise diagnostic: “equation ‘assoc’ violated when a=2, b=3, c=1: LHS=6, RHS=4”.

6.12.4 CheckModelOptions

The CheckModelOptions struct provides safety bounds for enumeration. The max_assignments field (default: 10,000) caps the total number of variable assignments evaluated per equation. An equation with 3 variables over a carrier of 100 elements would require 1,000,000 assignments. The limit prevents this from running for minutes.

When the limit is exceeded, the function returns a GatError::ModelError rather than silently skipping the check. The VCS layer reports this as a non-blocking warning.

6.13 Pullbacks (Finding Shared Structure)

6.13.1 Math

Given morphisms \(m_1: T_1 \to C\) and \(m_2: T_2 \to C\), the pullback theory \(P\) contains:

  • A sort for each pair \((s_1 \in T_1, s_2 \in T_2)\) where \(m_1(s_1) = m_2(s_2)\) and \(s_1\), \(s_2\) have the same arity.
  • An operation for each pair \((f_1 \in T_1, f_2 \in T_2)\) where \(m_1(f_1) = m_2(f_2)\) and their input and output sort pairs exist in the pullback.
  • An equation for each pair \((e_1 \in T_1, e_2 \in T_2)\) where \(m_1(e_1) = m_2(e_2)\).

6.13.2 Rust: PullbackResult

pub struct PullbackResult {
    pub theory: Theory,
    pub proj1: TheoryMorphism,
    pub proj2: TheoryMorphism,
}

The result includes projection morphisms \(\pi_1: P \to T_1\) and \(\pi_2: P \to T_2\) satisfying the universal property: \(m_1 \circ \pi_1 = m_2 \circ \pi_2\). Both projections are validated by check_morphism in the test suite.

6.13.3 Pullback Naming

When both sides contribute an element with the same name, the pullback element keeps that name. When the names differ, the pullback element is named "name1=name2". For example, pulling back theories where V1 and V2 both map to Vertex in the codomain produces a pullback sort named V1=V2.

6.13.4 Concrete Example: Merge Overlap Detection

The VCS merge algorithm uses pullbacks to detect structural overlap between two branches. When merging branch A and branch B, the three-way merge computes the pullback of the two branch theories over the merge base theory. Non-empty pullback elements indicate shared structure that must be reconciled. Empty pullback means the branches modified disjoint parts of the schema.

6.14 Natural Transformations (Maps Between Mappings)

6.14.1 Math

Given morphisms \(F, G: \mathcal{T}_1 \to \mathcal{T}_2\), a natural transformation \(\alpha: F \Rightarrow G\) assigns to each sort \(S\) in \(\mathcal{T}_1\) a term \(\alpha_S\) that maps values of sort \(F(S)\) to values of sort \(G(S)\), subject to naturality: for each operation \(f: S_1 \to S_2\), we have \(\alpha_{S_2} \circ F(f) = G(f) \circ \alpha_{S_1}\).

6.14.2 Rust

pub struct NaturalTransformation {
    pub name: Arc<str>,
    pub source: Arc<str>,
    pub target: Arc<str>,
    pub components: HashMap<Arc<str>, Term>,
}

Each component is a Term with a single free variable "x" representing the input value. For the identity natural transformation, every component is Term::Var("x").

6.14.3 check_natural_transformation

pub fn check_natural_transformation(
    nt: &NaturalTransformation,
    f: &TheoryMorphism,
    g: &TheoryMorphism,
    domain: &Theory,
    codomain: &Theory,
) -> Result<(), GatError>

The checker verifies four conditions:

  1. Domain/codomain agreement: \(F\) and \(G\) must have the same domain and codomain theories.
  2. Component completeness: every sort in the domain must have a component.
  3. Component validity: all operations used in component terms must exist in the codomain theory.
  4. Naturality: for each operation \(f: S_1 \to S_2\) in the domain, the naturality square must commute: \(\alpha_{S_2}[x := F(f)(x)] = G(f)(\alpha_{S_1}(x))\).

The naturality check is performed symbolically by substituting and comparing term trees. For n-ary operations, the check uses variables x0, x1, ... for each input.

6.14.4 Vertical Composition (Chaining Transforms)

vertical_compose chains two natural transformations: given \(\alpha: F \Rightarrow G\) and \(\beta: G \Rightarrow H\), it produces \(\beta \circ \alpha: F \Rightarrow H\). For each sort \(S\), the composed component is \(\beta_S[x := \alpha_S(x)]\).

The function checks that alpha.target == beta.source (the intermediate morphism must match) and returns GatError::NatTransComposeMismatch if they disagree.

6.14.5 Horizontal Composition (Combining Independent Transforms)

horizontal_compose composes natural transformations along different morphism pairs: given \(\alpha: F \Rightarrow G\) (between \(T_1\) and \(T_2\)) and \(\beta: H \Rightarrow K\) (between \(T_2\) and \(T_3\)), it produces \(\beta * \alpha: H \circ F \Rightarrow K \circ G\).

For each sort \(S\) in the domain, the composed component is \(\beta_{G(S)}[x := H(\alpha_S)]\), where \(G(S)\) is the sort that \(G\) maps \(S\) to, and \(H(\alpha_S)\) is the result of applying \(H\)’s operation renaming to \(\alpha\)’s component at \(S\).

Note

Horizontal composition is used in the VCS merge pipeline when composing migration paths through different branch histories. The compose_path function in panproto-vcs assembles a chain of natural transformations to verify that the composed migration path is coherent.

6.15 Free Models (Minimal Valid Instances)

6.15.1 Math

The free (initial) model of a theory \(\mathcal{T}\) is the minimal model satisfying all equations. Its carrier sets are equivalence classes of closed terms (terms with no free variables), quotiented by the equations.

6.15.2 Rust

The construction proceeds in three phases:

  1. Term generation: enumerate all closed terms up to a configurable depth bound, organized by sort. Nullary operations (constants) provide the seed. Non-nullary operations are applied to all combinations of existing terms of the correct sorts.

  2. Quotient by equations: use a union-find data structure to identify terms that must be equal according to the theory’s equations. For each equation, enumerate all valid variable substitutions (using the generated terms as values) and union the global indices of the LHS and RHS results.

  3. Build model: select one representative per equivalence class as the carrier set elements, and define operations by symbolic term application.

6.15.3 FreeModelConfig

pub struct FreeModelConfig {
    pub max_depth: usize,           // Default: 3
    pub max_terms_per_sort: usize,  // Default: 1000
}

The depth bound controls how deep the term enumeration goes. For a theory with a successor operation, depth 3 generates zero, succ(zero), succ(succ(zero)), and succ(succ(succ(zero))). The per-sort limit prevents combinatorial explosion in theories with binary operations.

6.15.4 Concrete Examples

  • Pointed set (one sort, one constant): free model has exactly 1 element.
  • Two constants, no equations: free model has 2 elements.
  • Two constants with \(a = b\) equation: free model has 1 element (the equation collapses them).
  • Monoid with identity laws: at depth 1, the free model collapses mul(unit, x) and mul(x, unit) with x, leaving only the identity element.
  • Graph theory (no constants): free model has empty carrier sets for both Vertex and Edge, since there are no closed terms.

6.15.5 CLI Integration

The schema scaffold command uses free_model to generate skeleton instances. The --depth and --max-terms flags map directly to FreeModelConfig fields.

6.16 Quotient Theories (Identifying Equivalent Elements)

6.16.1 Math

A quotient of a theory \(\mathcal{T}\) by a set of identification pairs \(\{(a_i, b_i)\}\) produces a new theory \(\mathcal{T}/{\sim}\) where each identified pair becomes the same element. This is the dual of theory extension: where extension adds new structure, quotienting collapses existing structure.

6.16.2 Rust

Given a theory and a list of identification pairs (a, b), the function:

  1. Classifies each pair as a sort identification or operation identification.
  2. Builds a union-find for sorts and a union-find for operations, computing the transitive closure.
  3. Verifies compatibility: identified sorts must have the same arity; identified operations must have the same signature after sort renaming.
  4. Rebuilds the theory with one representative per equivalence class. Sort parameters, operation input sorts, and equation terms are all renamed through the sort and operation rename maps.
  5. Deduplicates equations that become identical after renaming.

6.16.3 Error Conditions

  • GatError::QuotientIncompatible: the identified elements have incompatible structure (different arities or signatures).
  • GatError::SortNotFound / GatError::OpNotFound: a name referenced during rebuilding doesn’t exist.

6.16.4 Concrete Example

Quotient theories are used when composing protocol fragments that have structurally equivalent but differently-named elements. If protocol A has sort Node and protocol B has sort Vertex, and both have the same role in the combined theory, quotienting by (Node, Vertex) produces a single sort. The alphabetically-first representative (Node) is selected deterministically.

6.17 Schema Transforms and Factorization

The schema_functor and factorize modules extend panproto-gat with theory endofunctors and morphism decomposition. These provide the GAT-level machinery that the protolens engine (Chapter 19) builds on.

6.17.1 Math

A theory endofunctor \(\Phi\) maps theories to theories. It pairs a precondition (a predicate specifying which theories it applies to) with a transformation. The factorization of a morphism \(F: T_1 \to T_2\) decomposes \(F\) into a sequence of elementary endofunctors, following a fixed phase ordering:

  1. Drops (equations, then operations, then sorts): remove elements not in the codomain.
  2. Renames (sorts, then operations): apply the sort-map and op-map.
  3. Adds (sorts topologically sorted by dependencies, then operations, then equations): add elements in the codomain but not in the (renamed) domain.

CautionWhy does the factorization follow a fixed phase ordering?

Drops must come before renames because renaming a sort that will be dropped wastes work and can create name collisions. Renames must come before adds because a newly-added sort might have the same name as a sort being renamed away. Violating this ordering can produce intermediate theories that are ill-formed.

# panproto-gat: The GAT Engine {#sec-gat-engine}

A sort is a type in a GAT. An operation is a function between sorts. An equation constrains operations. The Rust representations follow. In the theory of categories, for instance, we have sorts $\mathrm{Ob}$ and $\mathrm{Hom}(a: \mathrm{Ob}, b: \mathrm{Ob})$, operations $\mathrm{id}: \mathrm{Ob} \to \mathrm{Hom}(a, a)$ and $\mathrm{comp}: \mathrm{Hom}(a, b) \times \mathrm{Hom}(b, c) \to \mathrm{Hom}(a, c)$, and equations for associativity and unitality.

This crate is Level 0 of the panproto hierarchy: the algebraic foundation on which everything else is built. It implements GATs as defined by @cartmell1986.

## Sorts and Sort Parameters

### Math

A sort $S$ in a GAT has arity $n \geq 0$. When $n = 0$, $S$ is a _simple sort_. When $n > 0$, $S$ is a _dependent sort_ parameterized by terms of other sorts. The distinction between simple and dependent sorts is what separates GATs from ordinary algebraic theories in the sense of @lawvere1963.

A sort parameter names a dependency of a sort on a term of another sort. For example, $\mathrm{Hom}(a: \mathrm{Ob}, b: \mathrm{Ob})$ has two parameters, each of sort $\mathrm{Ob}$.

### Rust

```{.rust include="../../crates/panproto-gat/src/sort.rs" start-line=1 end-line=38}
```

The `arity()` method returns the number of parameters. This is the key invariant that theory morphisms must preserve.

### Concrete Examples

Sorts you'll encounter in panproto protocol definitions:

- `Vertex`: simple sort, arity 0. The vertices of a schema graph.
- `Edge`: simple sort, arity 0. Directed edges between vertices.
- `Hom(a: Ob, b: Ob)`: dependent sort, arity 2. Morphisms in a category.
- `Constraint(v: Vertex)`: dependent sort, arity 1. Constraints attached to a specific vertex.

::: {.callout-tip}
In panproto's schema theories, most sorts are simple (`Vertex`, `Edge`, `Constraint`). Dependent sorts appear when a schema theory needs to express constraints that depend on a vertex (e.g., `Constraint(v: Vertex)`), or when modeling categories where morphisms depend on their source and target objects.
:::

## Operations

### Math

An operation $f: (a_1: S_1, \ldots, a_n: S_n) \to T$ is a function in a GAT. Each operation has typed inputs and a typed output. Operations with zero inputs are constants (nullary constructors). The type signature of an operation is the pair $(\langle S_1, \ldots, S_n \rangle, T)$. Theory morphisms must preserve these signatures up to the sort mapping.

### Rust

```{.rust include="../../crates/panproto-gat/src/op.rs" start-line=1 end-line=23}
```

The `inputs` field stores `(param_name, sort_name)` pairs. This is sufficient for first-order operations. Higher-order operations (operations whose inputs are themselves operations) aren't directly supported; panproto doesn't need them for its protocol modeling use case.

### Concrete Examples

In a graph schema theory:

- `src: (e: Edge) -> Vertex`: the source vertex of an edge.
- `tgt: (e: Edge) -> Vertex`: the target vertex of an edge.
- `unit: () -> Carrier`: a nullary operation (constant) in a monoid theory.

## Equations

### Math

An equation (axiom) in a GAT asserts that two terms are equal: $\ell = r$ where $\ell$ and $r$ are built from variables and operation applications. Equations constrain the behavior of operations. They aren't computationally evaluated; they serve as proof obligations that theory morphisms must preserve.

### Rust

panproto uses a simple term language:

- `Term::Var(name)`: a variable reference.
- `Term::App(op_name, args)`: application of an operation to arguments.

When `panproto-gat` checks a morphism, it verifies that for each equation $\ell = r$ in the domain theory, the mapped equation $F(\ell) = F(r)$ exists in the codomain theory.

## Sort Kinds

### Math

Sorts are classified by a _kind_ that distinguishes structural sorts from value-level sorts. The classification forms a sum type: $\mathrm{SortKind} = \mathrm{Structural} \mid \mathrm{Val}(k) \mid \mathrm{Coercion}(k_1, k_2) \mid \mathrm{Merger}(k)$ where $k, k_1, k_2 \in \mathrm{ValueKind}$.

### Rust

- **`Structural`** (default): standard schema-level sorts like `Vertex`, `Edge`, `Constraint`.
- **`Val(kind)`**: value sorts carrying data of a specific `ValueKind` (Bool, Int, Float, Str, Bytes, Token, Null, Any).
- **`Coercion { from, to }`**: coercion sorts representing directed morphisms between value kinds.
- **`Merger(kind)`**: merger sorts that combine values of a specific kind.

### Concrete Example

Value sorts bridge the schema level and the instance level. A sort like `Val(Str)` classifies string values. `Coercion { from: Str, to: Int }` classifies functions that convert strings to integers (e.g., `parse_int`).

## Directed Equations

### Math

A directed equation $\ell \longrightarrow r$ extends the standard equation with a computation direction. Where an equation $\ell = r$ is an undirected assertion, a directed equation specifies a rewrite rule: when the migration engine encounters a value matching $\ell$, it rewrites to $r$ using the implementation term.

### Rust

Each directed equation carries:

- `impl_term`: a `panproto_expr::Expr`, the computable implementation of the rewrite.
- `inverse`: an optional `panproto_expr::Expr` for the backward (put) direction.

The expression language is defined in the `panproto-expr` crate, a pure functional language with lambda calculus, pattern matching, records, lists, and approximately 50 built-in operations.

## Conflict Policies

A `ConflictPolicy` specifies how merge conflicts are resolved for values of a given kind:

- `KeepLeft`: always prefer the left branch.
- `KeepRight`: always prefer the right branch.
- `Fail`: report a conflict (default behavior).
- `Custom(expr)`: evaluate a `panproto_expr::Expr` to decide.

## Theory

### Math

A theory $\mathcal{T} = (S, \Omega, E)$ consists of a set of sorts $S$, a set of operations $\Omega$ (each with typed inputs and output drawn from $S$), and a set of equations $E$ over terms built from $\Omega$. We extend this with directed equations, conflict policies, and an inheritance relation.


### Theory Inheritance

The `extends` field enables compositional theory construction. When theory B extends theory A, B inherits all of A's sorts, operations, and equations. The `resolve_theory` function computes the transitive closure of the inheritance chain, merging declarations and detecting cycles.

Diamond inheritance is handled correctly. If theories B and C both extend A, and theory D extends both B and C, the shared declarations from A appear exactly once in the resolved theory D.

::: {.callout-note}
For the conceptual introduction to GATs, see the [tutorial chapter 3](https://panproto.dev/tutorial/chapters/03-what-is-a-theory.html).
:::

::: {.callout-caution}
## When should you use theory inheritance vs. colimits?
Use `extends` when the child genuinely specializes the parent (adding sorts or operations to an existing vocabulary). Use colimits when composing two independent theory fragments that share a common sub-theory. The rule of thumb: if the shared structure is "accidental" (same name, same role, but developed independently), use a colimit. If it's intentional (the child is meant to extend the parent), use `extends`.
:::

## Theory Morphisms

### Math

A theory morphism $F: \mathcal{T}_1 \to \mathcal{T}_2$ is a structure-preserving map. It maps sorts to sorts and operations to operations, subject to three preservation conditions:

1. **Arity preservation**: if sort $S$ has arity $n$, then $F(S)$ must also have arity $n$.
2. **Signature preservation**: if operation $f: (a: A, b: B) \to C$, then $F(f): (a: F(A), b: F(B)) \to F(C)$.
3. **Equation preservation**: if $\ell = r$ is an equation in $\mathcal{T}_1$, then $F(\ell) = F(r)$ must be an equation in $\mathcal{T}_2$.

### Rust

```{.rust include="../../crates/panproto-gat/src/morphism.rs" start-line=8 end-line=24}
```

The `check_morphism` function validates all three conditions. This is the critical correctness check for schema migrations: a migration specification is valid if and only if it induces a valid theory morphism between the source and target schema theories.

### Why Morphisms Matter for Migration

When we migrate data from schema v1 to schema v2, the migration is mathematically a theory morphism from v2's schema theory to v1's schema theory (note the direction reversal; this is the contravariance of restriction functors). The `CompiledMigration` in `panproto-inst` is the executable form of this morphism. The `wtype_restrict` pipeline is the functor it induces on instances.

This is the fundamental guarantee. Because the restrict operation is functorial, composing two migrations and restricting is the same as restricting twice [@spivak2012]. There are no "migration bugs" that arise from order-of-operations issues.

### Inducing Schema Renames

Theory morphisms induce schema-level renames via `induce_schema_renames()`. Sort-map entries where `old ≠ new` become `NameSite::VertexKind` renames (since sorts map to vertex kinds at the schema level). Op-map entries where `old ≠ new` become `NameSite::EdgeKind` renames. This is the top of the _morphism tower_ described in @sec-naming-identity-section.

## Identity and Naming

The `ident` module provides three types that separate identity from display name:

- **`ScopeTag`**: an opaque scope tag distinguishing different naming contexts, generated via `ScopeTag::fresh()`.
- **`Ident`**: a first-class identifier where equality uses `(scope, index)` only, not the display name. Follows `GATlab` [@lynch2024].
- **`Name`**: an interned string handle (`Arc<str>`) with a pointer-equality fast path. Drop-in replacement for `String` in hot-path types.

The `NameSite` enum enumerates the nine naming sites (edge labels, vertex IDs, vertex kinds, edge kinds, NSIDs, constraint sorts, instance anchors, theory names, sort names). `SiteRename` is a site-qualified rename operation used in schema morphism provenance and VCS commit metadata.

See @sec-naming-identity-section for the full contributor guide.

## Colimits (Merging Theories)

### Math

The colimit (pushout) of two theories $T_1$ and $T_2$ over a shared base $S$ is a combined theory $T_1 +_S T_2$ containing all declarations from both, with the shared declarations identified (not duplicated). Formally, given inclusions $i_1: S \hookrightarrow T_1$ and $i_2: S \hookrightarrow T_2$, the pushout is the universal cocone.

### Conflict Detection

The colimit function detects three kinds of conflicts:

- **Sort conflicts**: $T_1$ and $T_2$ both declare a sort with the same name but different parameter lists, and the sort is not in the shared base.
- **Operation conflicts**: same-named operations with incompatible signatures.
- **Equation conflicts**: same-named equations with different term content.

Compatible duplicates (same name, same definition) are silently deduplicated. This allows independent protocol extensions to share common vocabulary without coordination, as long as they agree on the meaning of shared names.

### Concrete Example: Composing Schema Theories

The `panproto-protocols` crate uses colimits extensively. The AT Protocol schema theory is built by combining a base graph theory (`ThGraph` with Vertex, Edge, src, tgt) with a constraint theory (`ThConstraint` with Constraint, target) over their shared `ThVertex`. The colimit produces a theory with all three sorts and all three operations, ready to serve as the schema theory for the AT Protocol.

::: {.callout-caution}
## What happens when two protocol fragments define the same sort name with different arities?
The colimit function returns a `GatError::SortConflict`. You must either rename one of the sorts before merging or use a quotient to explicitly identify them. There's no implicit resolution.
:::

## Type-Checking

### Math

Type-checking verifies that terms and equations are well-typed with respect to a theory's operation signatures. For a term $f(t_1, \ldots, t_n)$ where $f: (a_1: S_1, \ldots, a_n: S_n) \to T$, type-checking confirms that each $t_i$ has sort $S_i$ and infers the result sort $T$.

### Rust

The core function `typecheck_term` infers the output sort of a term given a variable typing context and a theory. It walks the term tree recursively:

- For `Term::Var(x)`: returns the sort from the context, or `GatError::UnboundVariable`.
- For `Term::App { op, args }`: looks up `op` in the theory, recursively typechecks each argument, verifies that each argument's sort matches the operation's input signature, and returns the output sort.

```rust
pub fn typecheck_term(
    term: &Term,
    ctx: &VarContext,
    theory: &Theory,
) -> Result<Arc<str>, GatError>
```

The `VarContext` type is a `FxHashMap<Arc<str>, Arc<str>>` mapping variable names to their sort names. Callers construct a context manually when type-checking a standalone term, but more commonly the context is inferred automatically by `infer_var_sorts`.

### infer_var_sorts

`infer_var_sorts` walks both sides of an equation, collecting sort constraints from every operation application site. A variable used as argument $i$ of operation $f$ must have the sort declared for that input position. If the same variable appears at two positions requiring different sorts, the function returns `GatError::ConflictingVarSort`.

```rust
pub fn infer_var_sorts(eq: &Equation, theory: &Theory) -> Result<VarContext, GatError>
```

This function is the entry point for automatic variable sort inference. It's used by `typecheck_equation`, `check_model`, and `free_model`. It eliminates the need for explicit sort annotations on equation variables.

### typecheck_equation and typecheck_theory

`typecheck_equation` combines inference and checking: it infers variable sorts from the equation structure, typechecks both sides, and verifies they produce the same output sort. If the two sides have different sorts, it returns `GatError::EquationSortMismatch`.

`typecheck_theory` iterates over all equations in a theory and typechecks each one, returning the first error encountered. This is the function called during VCS validation to ensure that a protocol's schema theory is internally consistent.

```rust
pub fn typecheck_theory(theory: &Theory) -> Result<(), GatError>
```

::: {.callout-tip}
Type-checking is fast. It's linear in the size of the term tree for each equation. Even theories with hundreds of equations typecheck in microseconds. The VCS pipeline runs `typecheck_theory` on every `schema add` and `schema commit` without measurable overhead.
:::

## Model Checking

### Math

A model $M$ of a theory $\mathcal{T}$ is an interpretation: each sort $S$ maps to a carrier set $M(S)$, and each operation $f: S_1 \times \cdots \times S_n \to T$ maps to a function $M(f): M(S_1) \times \cdots \times M(S_n) \to M(T)$. The model satisfies $\mathcal{T}$ if every equation $\ell = r$ evaluates to equal values under all variable assignments drawn from the carrier sets.

### Rust

A `Model` provides an interpretation of a theory: each sort maps to a carrier set (a `Vec<ModelValue>`), and each operation maps to a Rust closure. `ModelValue` is an enum with `Int(i64)` and `Str(String)` variants, enough for the finite models that panproto needs for equation verification.

```rust
pub fn check_model(model: &Model, theory: &Theory) -> Result<Vec<EquationViolation>, GatError>

pub fn check_model_with_options(
    model: &Model,
    theory: &Theory,
    options: &CheckModelOptions,
) -> Result<Vec<EquationViolation>, GatError>
```

For each equation, the checker infers variable sorts, builds the cartesian product of the carrier sets for those sorts, and evaluates both sides under every assignment. If `lhs_value != rhs_value`, the assignment is recorded as an `EquationViolation`.

### EquationViolation

Each violation captures the equation name, the variable assignment that triggered it, and the values that the two sides evaluated to:

```rust
pub struct EquationViolation {
    pub equation: Arc<str>,
    pub assignment: FxHashMap<Arc<str>, ModelValue>,
    pub lhs_value: ModelValue,
    pub rhs_value: ModelValue,
}
```

This gives enough information for the CLI to print a precise diagnostic: "equation 'assoc' violated when a=2, b=3, c=1: LHS=6, RHS=4".

### CheckModelOptions

The `CheckModelOptions` struct provides safety bounds for enumeration. The `max_assignments` field (default: 10,000) caps the total number of variable assignments evaluated per equation. An equation with 3 variables over a carrier of 100 elements would require 1,000,000 assignments. The limit prevents this from running for minutes.

When the limit is exceeded, the function returns a `GatError::ModelError` rather than silently skipping the check. The VCS layer reports this as a non-blocking warning.

## Pullbacks (Finding Shared Structure)

### Math

Given morphisms $m_1: T_1 \to C$ and $m_2: T_2 \to C$, the pullback theory $P$ contains:

- A sort for each pair $(s_1 \in T_1, s_2 \in T_2)$ where $m_1(s_1) = m_2(s_2)$ and $s_1$, $s_2$ have the same arity.
- An operation for each pair $(f_1 \in T_1, f_2 \in T_2)$ where $m_1(f_1) = m_2(f_2)$ and their input and output sort pairs exist in the pullback.
- An equation for each pair $(e_1 \in T_1, e_2 \in T_2)$ where $m_1(e_1) = m_2(e_2)$.

### Rust: PullbackResult

```rust
pub struct PullbackResult {

    pub proj2: TheoryMorphism,
}
```

The result includes projection morphisms $\pi_1: P \to T_1$ and $\pi_2: P \to T_2$ satisfying the universal property: $m_1 \circ \pi_1 = m_2 \circ \pi_2$. Both projections are validated by `check_morphism` in the test suite.

### Pullback Naming

When both sides contribute an element with the same name, the pullback element keeps that name. When the names differ, the pullback element is named `"name1=name2"`. For example, pulling back theories where `V1` and `V2` both map to `Vertex` in the codomain produces a pullback sort named `V1=V2`.

### Concrete Example: Merge Overlap Detection

The VCS merge algorithm uses pullbacks to detect structural overlap between two branches. When merging branch A and branch B, the three-way merge computes the pullback of the two branch theories over the merge base theory. Non-empty pullback elements indicate shared structure that must be reconciled. Empty pullback means the branches modified disjoint parts of the schema.

## Natural Transformations (Maps Between Mappings)

### Math

Given morphisms $F, G: \mathcal{T}_1 \to \mathcal{T}_2$, a natural transformation $\alpha: F \Rightarrow G$ assigns to each sort $S$ in $\mathcal{T}_1$ a term $\alpha_S$ that maps values of sort $F(S)$ to values of sort $G(S)$, subject to naturality: for each operation $f: S_1 \to S_2$, we have $\alpha_{S_2} \circ F(f) = G(f) \circ \alpha_{S_1}$.

### Rust

```rust
pub struct NaturalTransformation {
    pub name: Arc<str>,
    pub source: Arc<str>,
    pub target: Arc<str>,
    pub components: HashMap<Arc<str>, Term>,
}
```

Each component is a `Term` with a single free variable `"x"` representing the input value. For the identity natural transformation, every component is `Term::Var("x")`.

### check_natural_transformation

```rust
pub fn check_natural_transformation(
    nt: &NaturalTransformation,
    f: &TheoryMorphism,
    g: &TheoryMorphism,
    domain: &Theory,
    codomain: &Theory,
) -> Result<(), GatError>
```

The checker verifies four conditions:

1. **Domain/codomain agreement**: $F$ and $G$ must have the same domain and codomain theories.
2. **Component completeness**: every sort in the domain must have a component.
3. **Component validity**: all operations used in component terms must exist in the codomain theory.
4. **Naturality**: for each operation $f: S_1 \to S_2$ in the domain, the naturality square must commute: $\alpha_{S_2}[x := F(f)(x)] = G(f)(\alpha_{S_1}(x))$.

The naturality check is performed symbolically by substituting and comparing term trees. For n-ary operations, the check uses variables `x0, x1, ...` for each input.

### Vertical Composition (Chaining Transforms)

`vertical_compose` chains two natural transformations: given $\alpha: F \Rightarrow G$ and $\beta: G \Rightarrow H$, it produces $\beta \circ \alpha: F \Rightarrow H$. For each sort $S$, the composed component is $\beta_S[x := \alpha_S(x)]$.

The function checks that `alpha.target == beta.source` (the intermediate morphism must match) and returns `GatError::NatTransComposeMismatch` if they disagree.

### Horizontal Composition (Combining Independent Transforms)

`horizontal_compose` composes natural transformations along different morphism pairs: given $\alpha: F \Rightarrow G$ (between $T_1$ and $T_2$) and $\beta: H \Rightarrow K$ (between $T_2$ and $T_3$), it produces $\beta * \alpha: H \circ F \Rightarrow K \circ G$.

For each sort $S$ in the domain, the composed component is $\beta_{G(S)}[x := H(\alpha_S)]$, where $G(S)$ is the sort that $G$ maps $S$ to, and $H(\alpha_S)$ is the result of applying $H$'s operation renaming to $\alpha$'s component at $S$.

::: {.callout-note}
Horizontal composition is used in the VCS merge pipeline when composing migration paths through different branch histories. The `compose_path` function in `panproto-vcs` assembles a chain of natural transformations to verify that the composed migration path is coherent.
:::

## Free Models (Minimal Valid Instances)

### Math

The free (initial) model of a theory $\mathcal{T}$ is the minimal model satisfying all equations. Its carrier sets are equivalence classes of closed terms (terms with no free variables), quotiented by the equations.

### Rust

The construction proceeds in three phases:

1. **Term generation**: enumerate all closed terms up to a configurable depth bound, organized by sort. Nullary operations (constants) provide the seed. Non-nullary operations are applied to all combinations of existing terms of the correct sorts.

2. **Quotient by equations**: use a union-find data structure to identify terms that must be equal according to the theory's equations. For each equation, enumerate all valid variable substitutions (using the generated terms as values) and union the global indices of the LHS and RHS results.

3. **Build model**: select one representative per equivalence class as the carrier set elements, and define operations by symbolic term application.

### FreeModelConfig

```rust
pub struct FreeModelConfig {
    pub max_depth: usize,           // Default: 3
    pub max_terms_per_sort: usize,  // Default: 1000
}
```

The depth bound controls how deep the term enumeration goes. For a theory with a successor operation, depth 3 generates `zero`, `succ(zero)`, `succ(succ(zero))`, and `succ(succ(succ(zero)))`. The per-sort limit prevents combinatorial explosion in theories with binary operations.

### Concrete Examples

- **Pointed set** (one sort, one constant): free model has exactly 1 element.
- **Two constants, no equations**: free model has 2 elements.
- **Two constants with $a = b$ equation**: free model has 1 element (the equation collapses them).
- **Monoid with identity laws**: at depth 1, the free model collapses `mul(unit, x)` and `mul(x, unit)` with `x`, leaving only the identity element.
- **Graph theory** (no constants): free model has empty carrier sets for both Vertex and Edge, since there are no closed terms.

### CLI Integration

The `schema scaffold` command uses `free_model` to generate skeleton instances. The `--depth` and `--max-terms` flags map directly to `FreeModelConfig` fields.

## Quotient Theories (Identifying Equivalent Elements)

### Math

A quotient of a theory $\mathcal{T}$ by a set of identification pairs $\{(a_i, b_i)\}$ produces a new theory $\mathcal{T}/{\sim}$ where each identified pair becomes the same element. This is the dual of theory extension: where extension adds new structure, quotienting collapses existing structure.

### Rust

Given a theory and a list of identification pairs `(a, b)`, the function:

1. Classifies each pair as a sort identification or operation identification.
2. Builds a union-find for sorts and a union-find for operations, computing the transitive closure.
3. Verifies compatibility: identified sorts must have the same arity; identified operations must have the same signature after sort renaming.
4. Rebuilds the theory with one representative per equivalence class. Sort parameters, operation input sorts, and equation terms are all renamed through the sort and operation rename maps.
5. Deduplicates equations that become identical after renaming.

### Error Conditions

- `GatError::QuotientIncompatible`: the identified elements have incompatible structure (different arities or signatures).
- `GatError::SortNotFound` / `GatError::OpNotFound`: a name referenced during rebuilding doesn't exist.

### Concrete Example

Quotient theories are used when composing protocol fragments that have structurally equivalent but differently-named elements. If protocol A has sort `Node` and protocol B has sort `Vertex`, and both have the same role in the combined theory, quotienting by `(Node, Vertex)` produces a single sort. The alphabetically-first representative (`Node`) is selected deterministically.

## Schema Transforms and Factorization {#sec-schema-endofunctors}

The `schema_functor` and `factorize` modules extend `panproto-gat` with theory endofunctors and morphism decomposition. These provide the GAT-level machinery that the protolens engine (@sec-protolens-engine) builds on.

### Math

A theory endofunctor $\Phi$ maps theories to theories. It pairs a precondition (a predicate specifying which theories it applies to) with a transformation. The factorization of a morphism $F: T_1 \to T_2$ decomposes $F$ into a sequence of elementary endofunctors, following a fixed phase ordering:

1. **Drops** (equations, then operations, then sorts): remove elements not in the codomain.
2. **Renames** (sorts, then operations): apply the sort-map and op-map.
3. **Adds** (sorts topologically sorted by dependencies, then operations, then equations): add elements in the codomain but not in the (renamed) domain.

### Rust: TheoryTransform

**File**: `crates/panproto-gat/src/schema_functor.rs`

```{.rust include="../../crates/panproto-gat/src/schema_functor.rs" start-line=69 end-line=77}
```

The key design principle is _cascading_: dropping a sort automatically drops all operations that reference it. Dropping an operation automatically drops all equations that reference it. This ensures that `TheoryTransform::apply` always produces a well-formed theory.

Endofunctors compose via `TheoryEndofunctor::compose`, producing a new endofunctor whose transform is `Compose(self.transform, other.transform)`. The precondition of the composed endofunctor conservatively requires the first endofunctor's precondition and defers the second's to runtime checking.

### Rust: Factorization

**File**: `crates/panproto-gat/src/factorize.rs`

The `Factorization` struct holds the ordered step sequence. `validate_factorization` applies the steps to the domain and checks that the result matches the codomain, providing an end-to-end correctness guarantee.

This factorization is the foundation of the automated lens generation pipeline (@sec-auto-lens-pipeline): each elementary step maps directly to a protolens constructor, and the full step sequence becomes a `ProtolensChain`.

::: {.callout-caution}
## Why does the factorization follow a fixed phase ordering?
Drops must come before renames because renaming a sort that will be dropped wastes work and can create name collisions. Renames must come before adds because a newly-added sort might have the same name as a sort being renamed away. Violating this ordering can produce intermediate theories that are ill-formed.
