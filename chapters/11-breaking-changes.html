11  panproto-check: Breaking Change Detection

The check pipeline is a three-stage function composition: \(\mathrm{report} \circ \mathrm{classify} \circ \mathrm{diff}\). Given two schemas \(S_1\) and \(S_2\) and a protocol \(P\), diff computes the structural delta, classify assigns severity based on \(P\)’s edge rules and constraint sorts, and report renders the result. For example, removing a vertex is always breaking; adding a vertex is always non-breaking; tightening a constraint is breaking only when the protocol recognizes that constraint sort.

11.1 Structural Diffing

11.1.1 SchemaDiff

The SchemaDiff struct records every structural difference between two schema revisions:

/// A structural diff between two schemas.
///
/// Each field captures a specific category of change between the old
/// and new schema revisions.
#[derive(Clone, Debug, Default, PartialEq, Eq, Serialize, Deserialize)]
pub struct SchemaDiff {
    // --- Vertices ---
    /// Vertex IDs present in the new schema but absent from the old.
    pub added_vertices: Vec<String>,
    /// Vertex IDs present in the old schema but absent from the new.
    pub removed_vertices: Vec<String>,
    /// Vertices whose `kind` changed between old and new.
    pub kind_changes: Vec<KindChange>,

    // --- Edges ---
    /// Edges present in the new schema but absent from the old.
    pub added_edges: Vec<Edge>,
    /// Edges present in the old schema but absent from the new.
    pub removed_edges: Vec<Edge>,

    // --- Constraints ---
    /// Constraints that changed between old and new, keyed by vertex ID.
    pub modified_constraints: HashMap<String, ConstraintDiff>,

    // --- Hyper-edges ---
    /// Hyper-edge IDs added in the new schema.
    pub added_hyper_edges: Vec<String>,
    /// Hyper-edge IDs removed from the old schema.
    pub removed_hyper_edges: Vec<String>,
    /// Hyper-edges whose kind, signature, or parent label changed.
    pub modified_hyper_edges: Vec<HyperEdgeChange>,

    // --- Required edges ---
    /// Per-vertex: required edges added in the new schema.
    pub added_required: HashMap<String, Vec<Edge>>,
    /// Per-vertex: required edges removed from the old schema.
    pub removed_required: HashMap<String, Vec<Edge>>,

    // --- NSIDs ---
    /// Vertex-to-NSID mappings added in the new schema.
    pub added_nsids: HashMap<String, String>,
    /// Vertex IDs whose NSID mapping was removed.
    pub removed_nsids: Vec<String>,
    /// NSID mappings that changed: `(vertex_id, old_nsid, new_nsid)`.
    pub changed_nsids: Vec<(String, String, String)>,

    // --- Variants ---
    /// Variants added in the new schema.
    pub added_variants: Vec<Variant>,
    /// Variants removed from the old schema.
    pub removed_variants: Vec<Variant>,
    /// Variants whose tag changed (same ID, different tag).
    pub modified_variants: Vec<VariantChange>,

    // --- Orderings ---
    /// Edge ordering changes: `(edge, old_position, new_position)`.
    pub order_changes: Vec<(Edge, Option<u32>, Option<u32>)>,

    // --- Recursion points ---
    /// Recursion points added in the new schema.
    pub added_recursion_points: Vec<RecursionPoint>,
    /// Recursion points removed from the old schema.
    pub removed_recursion_points: Vec<RecursionPoint>,
    /// Recursion points whose target vertex changed.
    pub modified_recursion_points: Vec<RecursionPointChange>,

    // --- Usage modes ---
    /// Usage mode changes: `(edge, old_mode, new_mode)`.
    pub usage_mode_changes: Vec<(Edge, UsageMode, UsageMode)>,

    // --- Spans ---
    /// Span IDs added in the new schema.
    pub added_spans: Vec<String>,
    /// Span IDs removed from the old schema.
    pub removed_spans: Vec<String>,
    /// Spans whose left or right vertex changed.
    pub modified_spans: Vec<SpanChange>,

    // --- Nominal ---
    /// Nominal flag changes: `(vertex_id, old_value, new_value)`.
    pub nominal_changes: Vec<(String, bool, bool)>,
}

The struct captures changes across all 13 schema fields: vertices, edges, constraints, hyper-edges, required edges, NSIDs, variants, orderings, recursion points, usage modes, spans, and nominal flags. Each category is a simple collection; the diff doesn’t assign severity to any change.

11.1.2 Constraint and Kind Changes

Constraint changes are captured at a finer granularity through ConstraintDiff, ConstraintChange, and KindChange:

/// Describes how constraints on a single vertex changed.
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub struct ConstraintDiff {
    /// Constraints added in the new schema.
    pub added: Vec<Constraint>,
    /// Constraints removed from the old schema.
    pub removed: Vec<Constraint>,
    /// Constraints whose value changed.
    pub changed: Vec<ConstraintChange>,
}

/// A single constraint that changed its value.
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub struct ConstraintChange {
    /// The constraint sort (e.g., `"maxLength"`).
    pub sort: String,
    /// The value in the old schema.
    pub old_value: String,
    /// The value in the new schema.
    pub new_value: String,
}

/// Records a vertex whose kind changed between schema versions.
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub struct KindChange {
    /// The vertex ID.
    pub vertex_id: String,
    /// The kind in the old schema.
    pub old_kind: String,
    /// The kind in the new schema.
    pub new_kind: String,
}

ConstraintDiff records three sub-categories for each vertex: constraints that were added (present in new but not old), constraints that were removed (present in old but not new), and constraints whose value changed (present in both but with different values). The ConstraintChange struct captures the sort, old value, and new value for each changed constraint, providing enough information for the classifier to determine the direction of change.

KindChange records vertices whose kind field changed between versions (e.g., a vertex that was "string" in the old schema but "integer" in the new schema).

11.1.3 Extended Change Types

The diff also tracks fine-grained changes to hyper-edges, variants, recursion points, and spans:

/// Records changes to a hyper-edge's kind, signature, or parent label.
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub struct HyperEdgeChange {
    /// The hyper-edge ID.
    pub id: String,
    /// Kind change: `(old_kind, new_kind)`, or `None` if unchanged.
    pub kind_change: Option<(String, String)>,
    /// Signature labels added: label → `vertex_id`.
    pub signature_added: HashMap<String, String>,
    /// Signature labels removed: label → `vertex_id`.
    pub signature_removed: HashMap<String, String>,
    /// Signature labels whose vertex changed: label → (`old_vid`, `new_vid`).
    pub signature_changed: HashMap<String, (String, String)>,
    /// Parent label change: `(old, new)`, or `None` if unchanged.
    pub parent_label_change: Option<(String, String)>,
}

/// Records a variant whose tag changed between schema versions.
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub struct VariantChange {
    /// The variant ID.
    pub id: String,
    /// The parent coproduct vertex ID.
    pub parent_vertex: String,
    /// The old tag.
    pub old_tag: Option<String>,
    /// The new tag.
    pub new_tag: Option<String>,
}

/// Records a recursion point whose target vertex changed.
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub struct RecursionPointChange {
    /// The fixpoint marker vertex ID.
    pub mu_id: String,
    /// The old target vertex.
    pub old_target: String,
    /// The new target vertex.
    pub new_target: String,
}

/// Records a span whose left or right vertex changed.
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub struct SpanChange {
    /// The span ID.
    pub id: String,
    /// Left vertex change: `(old, new)`, or `None` if unchanged.
    pub left_change: Option<(String, String)>,
    /// Right vertex change: `(old, new)`, or `None` if unchanged.
    pub right_change: Option<(String, String)>,
}

The function compares all schema fields using set operations. Output is deterministic: added and removed lists are sorted for consistent comparison in tests and CI.

Note

The diff operates on panproto’s graph-level schema representation, not on any protocol-specific format. A SchemaDiff between two JSON Schema versions and a SchemaDiff between two Protobuf versions have the same type and structure. Protocol-specific semantics only enter at the classification stage.

11.1.5 What diff Does Not Do

The diff is purely structural. It doesn’t:

  • Determine whether a change is breaking or non-breaking (that’s classify’s job).
  • Track renames (a rename appears as a removal plus addition of different vertices/edges).
  • Detect semantic equivalences (two schemas that are structurally different but semantically equivalent will produce a non-empty diff).

If you need rename detection, use the migration engine (Chapter 9) to compute an explicit mapping, then use classify on the migration’s implied diff.

11.2 Protocol-Aware Classification

Classification is where protocol semantics enter. The classify function takes a SchemaDiff and a Protocol and determines the severity of each change based on the protocol’s edge rules and constraint sorts.

11.2.1 Breaking Changes

Change When Breaking Rationale
Vertex removal Always Consumers depending on the vertex will fail
Edge removal When the edge kind matches a protocol edge rule Protocol-governed edges are part of the structural contract
Kind change Always Existing data may not conform to the new kind
Constraint tightening When the sort is recognized by the protocol Existing data may violate the tighter constraint
Constraint addition When the sort is recognized by the protocol Existing data may violate the new constraint

Vertex removal is unconditionally breaking because any consumer relying on that vertex (reading a field, following an edge, anchoring a node) will fail.

Edge removal is protocol-dependent. The protocol’s edge_rules declare which edge kinds are structurally significant. For example, JSON Schema protocols declare "prop" edges as governed; SQL protocols declare "fk" (foreign key) edges. Removing a governed edge is breaking; removing an ungoverned edge (e.g., a metadata annotation) is not.

Kind changes are unconditionally breaking. Changing a vertex from "string" to "integer" means that existing string data can’t be interpreted as integers without an explicit coercion.

11.2.2 Non-Breaking Changes

Change Rationale
Vertex addition New consumers can use it; existing consumers ignore it
Edge addition Additive change; existing consumers are unaffected
Constraint relaxation Existing data remains valid under looser constraints
Constraint removal Equivalent to maximally relaxing the constraint

Vertex and edge additions are the bread and butter of backward-compatible evolution. Adding an optional field to a JSON Schema, a new column to a SQL table, or a new message type to a Protobuf definition: all of these are non-breaking because existing consumers simply ignore the additions.

Constraint relaxation is non-breaking because existing data was valid under the tighter constraint and therefore remains valid under the looser one. Raising maxLength from 300 to 3000, for example, doesn’t invalidate any existing strings.

CautionWhat about edge removal for an ungoverned edge kind?

If the protocol doesn’t have an EdgeRule for the removed edge’s kind, the removal is classified as non-breaking. This is intentional: ungoverned edges are considered metadata or annotations, not structural contracts. If you want all edge removals to be breaking, add a catch-all edge rule to your protocol.

11.2.3 Constraint Direction

For numeric constraints, the classification engine determines direction automatically:

  • Upper bounds (maxLength, maxSize, maximum, maxGraphemes): a smaller new value is tightening (breaking); a larger new value is relaxing (non-breaking).
  • Lower bounds (minLength, minimum): a larger new value is tightening (breaking); a smaller new value is relaxing (non-breaking).
  • Other constraints: any value change is treated as tightening (breaking), since the engine can’t infer the direction for arbitrary constraint sorts without additional metadata.
Tip

If your protocol defines custom constraint sorts with known directionality, you can extend the is_constraint_tightened function in classify.rs to handle them. Unrecognized sorts default to “any change is breaking,” which is the conservative choice.

11.2.4 Compatibility Levels

The CompatReport assigns one of three implicit compatibility levels based on its contents:

  • Fully compatible: the breaking and non_breaking lists are both empty. The schemas are structurally identical.
  • Backward compatible: the breaking list is empty but non_breaking is not. Existing consumers are safe; new consumers can use the additions. The compatible field is true.
  • Breaking: the breaking list is non-empty. At least one change is incompatible. The compatible field is false.
Important

Classification is protocol-dependent. The same SchemaDiff can produce different CompatReport results for different protocols. For example, removing an edge with kind "fk" is breaking under the SQL protocol (which has an edge rule for foreign keys) but might be non-breaking under a protocol that doesn’t govern that edge kind. Always pass the correct protocol to classify.

11.3 Reporting

The report module converts a CompatReport into output suitable for humans or machines.

11.3.1 Text Reports

report_text produces a plain-text summary suitable for terminal display:

  1. A compatibility verdict: "COMPATIBLE: No breaking changes detected." or "INCOMPATIBLE: Breaking changes detected."
  2. An itemized list of breaking changes (if any), with a count header and numbered entries.
  3. An itemized list of non-breaking changes (if any), with a count header and numbered entries.
  4. If both lists are empty: "No changes detected."

This is what the schema check CLI command displays by default.

11.3.2 JSON Reports

report_json produces a serde_json::Value with three fields:

  • compatible (bool): whether the migration is backward-compatible.
  • breaking (array): each entry describes a breaking change with its type and details.
  • non_breaking (array): each entry describes a non-breaking change with its type and details.

This format is consumed by CI integrations (e.g., GitHub Actions annotations), the TypeScript SDK, and any tooling that needs machine-readable compatibility verdicts.

11.3.3 Choosing a Report Format

Use report_text for developer-facing output (CLI, pull request comments). Use report_json for machine-facing output (CI gates, automated migration generators, monitoring dashboards).

CautionHow do you integrate the JSON report with GitHub Actions?

Parse the JSON output and create inline annotations using the GitHub Checks API. The breaking array entries contain vertex IDs and edge descriptions that can be mapped back to source locations if your schema files have line-number metadata. See the CI chapter (Chapter 17) for a working example.

11.4 Relationship to Migrations and Lenses

Breaking change detection is complementary to the migration and lens systems:

  • Before migration: run diff + classify to determine whether a migration is safe. If the report says “breaking,” the developer knows to provide a migration with resolvers, supply default values, or use a lens with complement tracking.
  • With lenses: the RemoveField combinator in panproto-lens always produces a non-empty complement. The check crate can confirm that the corresponding schema change is indeed breaking, validating that the lens is necessary and correctly scoped.
  • In CI: the schema check CLI command runs this pipeline and exits with a non-zero status on breaking changes, enabling automated gatekeeping. Pull requests that introduce breaking changes without an accompanying migration can be flagged automatically.
  • In documentation: the text report can be included in changelog entries or release notes to communicate the nature of schema changes to downstream consumers.

11.5 Integration with CI

The schema check CLI command wraps this pipeline:

schema check --old schema-v1.json --new schema-v2.json --protocol atproto

Exit codes:

  • 0: fully compatible or backward compatible.
  • 1: breaking changes detected.
  • 2: error (e.g., malformed schema, unknown protocol).

11.6 Performance Considerations

The entire pipeline is designed to be fast enough for interactive use:

  • diff is \(O(|V_{\mathrm{old}}| + |V_{\mathrm{new}}| + |E_{\mathrm{old}}| + |E_{\mathrm{new}}| + |C_{\mathrm{old}}| + |C_{\mathrm{new}}|)\) where \(V\), \(E\), and \(C\) are vertex, edge, and constraint counts. For typical schemas with hundreds of vertices, this completes in microseconds.
  • classify is \(O(|\mathrm{diff}|)\), iterating once over each change category.
  • report_text and report_json are \(O(|\mathrm{report}|)\).

The pipeline can be run on every commit in CI without measurable impact on build times.

11.7 Error Handling

The panproto-check crate defines a CheckError type for failures that prevent the pipeline from completing:

  • Schema deserialization failures (malformed input).
  • Protocol lookup failures (unknown protocol name).

A CompatReport with breaking changes is not an error. It’s a successful analysis that found incompatibilities. The error types cover only cases where the analysis itself can’t be performed.

11.8 Module Map

Module Exports Purpose
diff SchemaDiff, ConstraintDiff, ConstraintChange, KindChange, diff Protocol-agnostic structural diffing
classify CompatReport, BreakingChange, NonBreakingChange, classify Protocol-aware severity classification
report report_text, report_json Human-readable and machine-readable output
error CheckError Error types for pipeline failures

All public types are re-exported from the crate root.

11.9 Testing Strategy

The test suite validates each pipeline stage:

  • Diff tests verify that specific structural changes (vertex addition/removal, edge addition/removal, constraint modification, kind change) produce the expected SchemaDiff entries. An important invariant: diffing identical schemas produces an empty diff.
  • Classification tests verify that breaking and non-breaking changes are correctly assigned based on protocol edge rules and constraint sorts. Tests cover edge cases like removing an edge whose kind isn’t governed by any edge rule (should be non-breaking) and tightening a constraint whose sort isn’t recognized by the protocol (should be ignored).
  • Report tests verify that text and JSON output formats are correctly structured and contain the expected information.

Parse the JSON output and create inline annotations using the GitHub Checks API. The `breaking` array entries contain vertex IDs and edge descriptions that can be mapped back to source locations if your schema files have line-number metadata. See the CI chapter (Chapter 17) for a working example.

## Relationship to Migrations and Lenses

Breaking change detection is complementary to the migration and lens systems:

- **Before migration**: run `diff` + `classify` to determine whether a migration is safe. If the report says "breaking," the developer knows to provide a migration with resolvers, supply default values, or use a lens with complement tracking.
- **With lenses**: the `RemoveField` combinator in `panproto-lens` always produces a non-empty complement. The check crate can confirm that the corresponding schema change is indeed breaking, validating that the lens is necessary and correctly scoped.
- **In CI**: the `schema check` CLI command runs this pipeline and exits with a non-zero status on breaking changes, enabling automated gatekeeping. Pull requests that introduce breaking changes without an accompanying migration can be flagged automatically.
- **In documentation**: the text report can be included in changelog entries or release notes to communicate the nature of schema changes to downstream consumers.

## Integration with CI

The `schema check` CLI command wraps this pipeline:

```bash
schema check --old schema-v1.json --new schema-v2.json --protocol atproto
```

Exit codes:

- `0`: fully compatible or backward compatible.
- `1`: breaking changes detected.
- `2`: error (e.g., malformed schema, unknown protocol).

## Performance Considerations

The entire pipeline is designed to be fast enough for interactive use:

- **`diff`** is $O(|V_{\mathrm{old}}| + |V_{\mathrm{new}}| + |E_{\mathrm{old}}| + |E_{\mathrm{new}}| + |C_{\mathrm{old}}| + |C_{\mathrm{new}}|)$ where $V$, $E$, and $C$ are vertex, edge, and constraint counts. For typical schemas with hundreds of vertices, this completes in microseconds.
- **`classify`** is $O(|\mathrm{diff}|)$, iterating once over each change category.
- **`report_text`** and **`report_json`** are $O(|\mathrm{report}|)$.

The pipeline can be run on every commit in CI without measurable impact on build times.

## Error Handling

The `panproto-check` crate defines a `CheckError` type for failures that prevent the pipeline from completing:

- Schema deserialization failures (malformed input).
- Protocol lookup failures (unknown protocol name).

A `CompatReport` with breaking changes is _not_ an error. It's a successful analysis that found incompatibilities. The error types cover only cases where the analysis itself can't be performed.

## Module Map

| Module | Exports | Purpose |
|---|---|---|
| `diff` | `SchemaDiff`, `ConstraintDiff`, `ConstraintChange`, `KindChange`, `diff` | Protocol-agnostic structural diffing |
| `classify` | `CompatReport`, `BreakingChange`, `NonBreakingChange`, `classify` | Protocol-aware severity classification |
| `report` | `report_text`, `report_json` | Human-readable and machine-readable output |
| `error` | `CheckError` | Error types for pipeline failures |

All public types are re-exported from the crate root.

## Testing Strategy

The test suite validates each pipeline stage:

- **Diff tests** verify that specific structural changes (vertex addition/removal, edge addition/removal, constraint modification, kind change) produce the expected `SchemaDiff` entries. An important invariant: diffing identical schemas produces an empty diff.
- **Classification tests** verify that breaking and non-breaking changes are correctly assigned based on protocol edge rules and constraint sorts. Tests cover edge cases like removing an edge whose kind isn't governed by any edge rule (should be non-breaking) and tightening a constraint whose sort isn't recognized by the protocol (should be ignored).
- **Report tests** verify that text and JSON output formats are correctly structured and contain the expected information.
