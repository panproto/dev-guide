12  panproto-protocols: Protocol Definitions

The panproto-protocols crate defines 60+ protocol specifications across 9 categories. Each protocol is built by composing reusable building-block theories via structured merges.1 We’ll walk through the theory composition pattern, the building blocks, and how to add new protocols.

12.1 Protocol Categories

panproto organizes protocols into nine categories:

Category Examples Count
Serialization Protobuf, Avro, Thrift, Cap’n Proto, FlatBuffers, ASN.1, Bond, MsgPack 8
Data Schema JSON Schema, XML/XSD, CSV/Table Schema, YAML, TOML, CDDL, INI, BSON 8
API GraphQL, OpenAPI, AsyncAPI, RAML, JSON:API 5
Database SQL, MongoDB, Cassandra, DynamoDB, Neo4j, Redis 6
Type System TypeScript, Python, Rust, Java, Go, Swift, Kotlin, C# 8
Web/Document ATProto, HTML, CSS, DOCX, ODF, Markdown, JSX, Vue, Svelte 9
Data Science Parquet, Arrow, DataFrame 3
Domain GeoJSON, FHIR, RSS/Atom, vCard/iCal, EDI X12, SWIFT MT 6
Config HCL, K8s CRD, Docker Compose, CloudFormation, Ansible 5

Despite this diversity, every protocol follows the same pattern: compose building-block theories via structured merge, define edge rules and vertex kinds, and implement a parser/emitter pair. Factoring common structure into shared theories avoids the combinatorial explosion that plagues conventional interop tools.

12.2 Building-Block Theories

The theories module defines the reusable GATs that protocols compose to form their schema and instance theories. Each function returns a standalone Theory value that can be registered in a theory registry and referenced by name.

12.2.1 Graph Theories

The three graph theories form the foundation for schema representation:

th_graph is the most fundamental building block: a directed graph with Vertex and Edge sorts (a type in a formal theory, not a sorting operation) and src/tgt operations. Nearly every protocol uses this theory as its base, since schemas are fundamentally graphs. Vertices represent types or fields; edges represent relationships between them.

th_simple_graph adds an equation enforcing edge uniqueness by endpoints. This is appropriate for protocols like Protobuf where each field name must be unique within a message. There cannot be two distinct edges with the same source and target.

th_hypergraph introduces multi-arity connections via HyperEdge and Label sorts. A hyper-edge connects multiple vertices through labeled positions, modeling constructs like SQL table definitions (where a table relates multiple columns via labeled positions like “column_name” and “column_type”) and multi-argument function signatures.

The constraint theory demonstrates the dependent sort pattern:

The Constraint sort is parameterized by a vertex via SortParam::new("v", "Vertex"). This is a GAT-level encoding of the fact that constraints are attached to specific schema elements. A maxLength constraint on a string field is a different sort instance than a maxLength constraint on a different field, even though they share the same sort name.

This dependent sort pattern has practical consequences throughout panproto:

  • The migration engine’s existence checker (Chapter 9) can scope constraint compatibility checks to individual vertices.
  • The breaking change detector (Chapter 11) can attribute constraint tightening to the specific vertex affected.
  • The lens system (Chapter 10) can track per-vertex constraint changes in the complement.
Note

The dependent sort Constraint(v: Vertex) is a GAT-level encoding that ensures constraint-related operations are automatically scoped to the correct vertex, without requiring runtime dispatch or downcasting.2

TipWhy dependent sorts instead of flat constraints?

If constraints were a flat sort (not parameterized by vertex), the existence checker couldn’t distinguish “constraint C tightened on vertex A” from “constraint C tightened on vertex B.” The dependent sort makes the scoping automatic.

12.2.3 Instance Theories

Beyond the schema theories, panproto defines instance theories that model how data is structured within a schema:

  • th_wtype: W-type trees with Node, Arc, and Value sorts. Operations include anchor (mapping nodes to schema vertices), arc_src/arc_tgt (tree edges), arc_edge (labeling arcs with schema edges), and node_value (payload). Used by JSON-like, XML-like, and document protocols.

  • th_functor: set-valued functors with Table, Row, and ForeignKey sorts. Operations include table_vertex (mapping tables to schema vertices) and fk_src/fk_tgt (foreign key endpoints). Used by relational and columnar protocols.

  • th_flat: flat key-value structures with Node, Field, and Value sorts. Operations include field_node (owning node), field_value (payload), and node_anchor (schema vertex). Used by IDL protocols like Protobuf where data has a flat field structure without deep nesting.

  • th_meta: metadata extension adding Discriminator and ExtraField sorts. Composes with th_wtype via structured merge over the shared Node sort. Used by document protocols with mixed content models (XML, HTML) where elements can have both structured children and unstructured attributes.

12.3 The Structured Merge Pattern

Each protocol is defined by composing building-block theories via structured merge, an operation that glues theories together along shared sorts.3 The shared sorts act as the “meeting point” where two theories agree on what a concept means.

For example, ATProto’s schema theory is the structured merge of ThGraph, ThConstraint, and ThMulti, all sharing the Vertex sort:

The structured merge ensures that the Vertex sort in ThGraph and the Vertex sort in ThConstraint are identified: they refer to the same concept. The resulting schema theory has all the sorts and operations from both components, with shared sorts unified. This is not a mere union; the merge preserves the equations and operation signatures from both theories, ensuring that the composed theory is well-formed.

12.3.1 Why This Construction?

The structured merge precisely controls which sorts are identified and which remain distinct. Alternative composition strategies either identify too much (forcing all sorts to be shared) or too little (sharing nothing).4

In practice, this means:

  • ThGraph and ThConstraint share Vertex but not Edge and Constraint. Edges and constraints are distinct concepts that happen to be attached to the same vertex sort.
  • ThGraph and ThMulti share both Vertex and Edge. Multi-edge labels are an extension of the same edge concept.
  • ThWType and ThMeta share Node. Metadata is an extension of the same node concept.

12.4 Theory Groups

Protocols cluster into six theory groups based on their schema/instance theory pair. Each group has a corresponding register_* helper function that handles the merge computation:

Group Schema Theory Instance Theory Registration Helper
A ThGraph + ThConstraint + ThMulti ThWType register_constrained_multigraph_wtype
B ThHypergraph + ThConstraint ThFunctor register_hypergraph_functor
C ThSimpleGraph + ThConstraint ThFlat register_simple_graph_flat
D ThGraph + ThConstraint + ThMulti + ThInterface ThWType register_typed_graph_wtype
E ThGraph + ThConstraint + ThMulti ThWType + ThMeta register_multigraph_wtype_meta

12.4.1 Group A: Constrained Multigraph + W-Type

The largest group, covering protocols with tree-shaped data and potentially parallel edges. ATProto, JSON Schema, OpenAPI, AsyncAPI, MongoDB, YAML Schema, TOML Schema, and many more belong here. The schema theory combines directed graphs (for type relationships), constraints (for validation rules like maxLength), and multi-edge support (for fields that share the same type). Instances are W-type trees.

12.4.2 Group B: Hypergraph + Functor

Covers relational and columnar protocols: SQL, Cassandra, DynamoDB, Parquet, Arrow, DataFrame, CSV/Table Schema. The schema theory uses hypergraphs because tables are multi-arity relations (a table “connects” its columns). Instances are set-valued mappings; each table maps to a set of rows.5

12.4.3 Group C: Simple Graph + Flat

Covers IDL and serialization protocols: Protobuf, Avro, Thrift, Cap’n Proto, FlatBuffers, ASN.1, Bond, HCL, Redis. These protocols have flat field structures (no deep nesting within a single message/record) and enforce field uniqueness, hence the simple graph. Instances are flat key-value structures.

12.4.4 Group D: Typed Graph + W-Type with Interfaces

Covers type systems and API schemas with inheritance: GraphQL, TypeScript, Python, Rust, Java, Go, Swift, Kotlin, C#. Adds the ThInterface theory to Group A, modeling constructs like GraphQL interfaces, TypeScript interfaces, and Java abstract classes.

12.4.5 Group E: Multigraph + W-Type + Metadata

Covers document formats with mixed content: XML/XSD, HTML, CSS, DOCX, ODF, Neo4j. Combines Group A’s schema theory with an instance theory that adds metadata support (discriminators and extra fields) for elements that have both structured children and unstructured attributes.

Tip

When adding a new protocol, first determine which theory group it belongs to. If it fits an existing group, you can reuse the corresponding register_* function and only need to define the protocol’s edge rules, vertex kinds, and parser/emitter.

12.5 The Protocol Interface

Each protocol module exports a protocol() function that returns a Protocol struct. The Protocol specifies:

  • name: the protocol identifier (e.g., "atproto", "json_schema", "sql").
  • schema_theory and instance_theory: names referencing entries in the theory registry, populated by the register_* helpers.
  • edge_rules: which edge kinds are structurally significant and what source/target vertex kinds they connect. Used by panproto-check to classify breaking changes (Chapter 11).
  • obj_kinds: vertex kinds that represent composite objects (as opposed to scalar values like "string" or "integer"). Used during schema parsing and by the lens system.
  • constraint_sorts: which constraint sorts this protocol recognizes (e.g., "maxLength", "pattern", "maxGraphemes"). Used by both the migration engine’s existence checker and the breaking change detector.

The theory names in the Protocol struct reference entries in the theory registry, which is populated by the register_* helpers. This indirection keeps protocol definitions declarative: a protocol does not construct its own theories; it names them.

12.6 The Two-Level Presentation Architecture

panproto’s two-parameter architecture (schema theory + instance theory) is mirrored by two crates that provide the presentation layer (the parse/emit layer that converts native format syntax to panproto’s internal representation):6

12.6.1 Schema-Level Presentations (panproto-protocols)

Every protocol implements a bidirectional schema-level conversion:

  • Parser: parse_*(input: &str) -> Result<Schema, ProtocolError> converts a native format specification (.proto file, JSON Schema, SQL DDL, GraphQL SDL) into panproto’s Schema representation, a model (a concrete instance that satisfies a theory’s requirements, like a Rust struct implementing a trait) of the protocol’s schema theory \(T_P^S\).
  • Emitter: emit_*(schema: &Schema) -> Result<String, ProtocolError> converts a Schema back into the native format specification.

Examples:

  • parse_json_schema(json_str) parses a JSON Schema document into a Schema.
  • emit_protobuf(schema) emits a .proto file from a Schema.
  • parse_sql(ddl_str) parses SQL DDL statements into a Schema.

12.6.2 Instance-Level Presentations (panproto-io)

The panproto-io crate provides the complementary instance-level conversions for all 76 protocols:

  • Parser: parse_wtype(protocol, schema, &bytes) -> Result<WInstance, ParseInstanceError> converts raw data (an HTML document, a CoNLL-U file, a Protobuf wire message) into a WInstance or FInstance, a model of the protocol’s instance theory \(T_P^I\).
  • Emitter: emit_wtype(protocol, schema, &instance) -> Result<Vec<u8>, EmitInstanceError> converts an instance back to raw format bytes.

The choice of WInstance vs FInstance is determined by the protocol’s instance_theory field, not by implementation preference.

12.6.3 The Complete Pipeline

Together, these two levels of presentation complete the data migration pipeline:

raw spec ──protocols parse──→ Schema v1 ──mig compile──→ CompiledMigration
raw data ──io parse──→ Instance v1 ──inst restrict──→ Instance v2 ──io emit──→ raw data

The commutativity guarantee from Spivak 2012 ensures that parse, restrict, and emit compose correctly across all 76 protocols. Both levels support round-trip testing: parse(emit(parse(input))) should produce a structurally equivalent result.

NoteWhat does “commutativity” buy us here?

It means we don’t need separate integration tests for every (source protocol, target protocol) pair. Correctness of the individual presentations plus the algebraic guarantee gives us correctness of the full pipeline.

12.6.4 Parsing Strategy

Parsers typically follow a two-phase approach:

  1. Syntactic parsing: use a format-specific parser (e.g., serde_json for JSON Schema, tree-sitter for Protobuf) to produce an AST or intermediate representation.
  2. Semantic translation: walk the AST, creating vertices for types and fields, edges for relationships, constraints for validation rules, and hyper-edges for multi-arity relations.

12.6.5 Emitting Strategy

Emitters reverse the process:

  1. Topology traversal: walk the schema graph, identifying root vertices, following edges in topological order.
  2. Format construction: generate the native syntax for each vertex (type declaration, field definition, constraint annotation) and edge (relationship, containment).

12.7 Adding a New Protocol

To add a new protocol to panproto, follow these five steps:

12.7.1 Step 1: Identify Component Theories

Determine which building-block theories your protocol needs. Most protocols fit one of the five existing groups. Ask yourself:

  • Does the protocol have directed edges between fields/types? Use ThGraph.
  • Does the protocol support parallel edges (e.g., multiple fields with the same type)? Add ThMulti.
  • Does the protocol have constraints (min/max length, required fields, patterns)? Add ThConstraint.
  • Does the protocol have multi-arity relations (e.g., SQL foreign keys, table definitions)? Use ThHypergraph instead of ThGraph.
  • Does the protocol support interfaces or inheritance? Add ThInterface.
  • Does the protocol have flat field structures with no deep nesting? Use ThFlat as the instance theory.
  • Does the protocol have tree-structured data? Use ThWType.
  • Does the protocol have mixed content (structured + unstructured)? Add ThMeta to ThWType.

12.7.2 Step 2: Compose via Structured Merge

If your protocol fits an existing group, call the corresponding register_* function with your theory names:

// In your protocol module's register function:
theories::register_constrained_multigraph_wtype(
    registry,
    "ThMyProtocolSchema",
    "ThMyProtocolInstance",
);

If your protocol needs a novel combination, compose the theories manually using panproto_gat::colimit(), specifying shared sorts at each step. Follow the pattern established by the existing register_* functions.

12.7.3 Step 3: Define Edge Rules and Vertex Kinds

Create a protocol() function in your module:

pub fn protocol() -> Protocol {
    Protocol {
        name: "my_protocol".into(),
        schema_theory: "ThMyProtocolSchema".into(),
        instance_theory: "ThMyProtocolInstance".into(),
        edge_rules: vec![/* ... */],
        obj_kinds: vec!["object".into(), "message".into()],
        constraint_sorts: vec!["maxLength".into()],
    }
}

12.7.4 Step 4: Implement Schema Presentation (panproto-protocols)

If your protocol has a schema definition language (e.g., .proto, DDL, SDL), write parse_my_protocol in the panproto-protocols crate to convert native format specifications into a Schema. Write emit_my_protocol to convert back.

Some protocols (like CoNLL-U) have no separate schema language; the schema is implicit in the format specification. These protocols only need Step 4b.

12.7.5 Step 4b: Implement Instance Presentation (panproto-io)

Write an instance codec in panproto-io by implementing the InstanceParser and InstanceEmitter traits. For most protocols, this means choosing the right shared pathway:

  • JSON data: use JsonCodec::new("my_protocol"), which delegates to the SIMD simd-json pathway.
  • XML data: use XmlCodec::new("my_protocol"), which delegates to the quick-xml pull parser.
  • Tab/delimited data: use TabularCodec::tsv("my_protocol", "table_name"), which delegates to the memchr SIMD pathway.
  • Custom format: implement InstanceParser and InstanceEmitter directly using a dedicated parsing library.

Register the codec in the appropriate category module’s register_all function and add it to the default_registry test.

12.7.6 Step 5: Add Integration Tests

Add tests at both levels:

  • Schema round-trip (panproto-protocols): parse a native specification, emit it back, verify the result matches.
  • Instance round-trip (panproto-io): parse native data, emit it back, re-parse, verify structural equality. Use real-world fixture data, not synthetic examples.
  • Migration integration: verify that parse → restrict → emit produces valid output.
  • Breaking-change tests: verify classification logic for your protocol’s edge rules.
Warning

When composing theories via structured merge, the shared theory must contain exactly the sorts that should be identified. If you share too few sorts, the merge will create duplicate sorts that should be unified. If you share too many sorts, you may over-constrain the composition. Review the existing register_* functions for examples of correct sharing patterns.

12.7.7 Worked Example: A Hypothetical “TaskML” Protocol

Suppose we’re defining a protocol for a task management markup language with:

  • Tasks have fields (name, description, priority).
  • Tasks can have subtasks (nesting).
  • Fields have validation constraints (maxLength on name, enum values for priority).
  • No parallel edges (each field name is unique within a task).

Step 1: This maps to Group C (simple graph + flat). The simple graph enforces field uniqueness; the flat instance theory models the non-nested field structure. But subtasks imply nesting, which requires W-type trees. So this is actually Group A (constrained multigraph + W-type), minus the multi-edge support since field names are unique. We could use Group A as-is (the multi-edge capability is unused but harmless) or define a custom composition of ThSimpleGraph + ThConstraint with ThWType.

Step 2: Using Group A for simplicity:

theories::register_constrained_multigraph_wtype(
    registry,
    "ThTaskMLSchema",
    "ThTaskMLInstance",
);

Step 3: Define the protocol:

pub fn protocol() -> Protocol {
    Protocol {
        name: "taskml".into(),
        schema_theory: "ThTaskMLSchema".into(),
        instance_theory: "ThTaskMLInstance".into(),
        edge_rules: vec![EdgeRule {
            edge_kind: "field".into(),
            src_kinds: vec!["task".into()],
            tgt_kinds: vec!["string".into(), "enum".into(), "integer".into()],
        }],
        obj_kinds: vec!["task".into()],
        constraint_sorts: vec!["maxLength".into(), "enumValues".into()],
    }
}

Steps 4 and 5: Implement parse_taskml and emit_taskml, then write round-trip and migration tests.

TipHow do I know if my protocol needs ThHypergraph vs ThGraph?

If your protocol’s core relations are binary (one source, one target), use ThGraph. If they’re \(n\)-ary (like SQL foreign keys connecting multiple columns), use ThHypergraph. The edge structure of your format’s native syntax usually makes this obvious.

12.8 Theoretical Foundations

The structured merge pattern is backed by a formal guarantee: the merge preserves the equations and well-formedness conditions of both component theories. If ThGraph guarantees that every edge has a source and target vertex, and ThConstraint guarantees that every constraint is attached to a vertex, then the merged theory inherits both guarantees.7

The panproto-protocols and panproto-io crates together implement two levels of the data pipeline: schema presentations map format specifications to abstract schema models, and instance presentations map format data to abstract instance models. This two-level factorization mirrors panproto’s two-parameter architecture and ensures that the full pipeline (from raw bytes through migration to target format) composes correctly by construction.

Despite this diversity, every protocol follows the same pattern: compose building-block theories via structured merge, define edge rules and vertex kinds, and implement a parser/emitter pair. Factoring common structure into shared theories avoids the combinatorial explosion that plagues conventional interop tools.

## Building-Block Theories

The `theories` module defines the reusable GATs that protocols compose to form their schema and instance theories. Each function returns a standalone `Theory` value that can be registered in a theory registry and referenced by name.

### Graph Theories

The three graph theories form the foundation for schema representation:

```{.rust include="../../crates/panproto-protocols/src/theories.rs" start-line=55 end-line=127}
```

**`th_graph`** is the most fundamental building block: a directed graph with `Vertex` and `Edge` sorts (a type in a formal theory, not a sorting operation) and `src`/`tgt` operations. Nearly every protocol uses this theory as its base, since schemas are fundamentally graphs. Vertices represent types or fields; edges represent relationships between them.

**`th_simple_graph`** adds an equation enforcing edge uniqueness by endpoints. This is appropriate for protocols like Protobuf where each field name must be unique within a message. There cannot be two distinct edges with the same source and target.

**`th_hypergraph`** introduces multi-arity connections via `HyperEdge` and `Label` sorts. A hyper-edge connects multiple vertices through labeled positions, modeling constructs like SQL table definitions (where a table relates multiple columns via labeled positions like "column_name" and "column_type") and multi-argument function signatures.

### Constraint Theory

The constraint theory demonstrates the dependent sort pattern:

```{.rust include="../../crates/panproto-protocols/src/theories.rs" start-line=129 end-line=144}
```

The `Constraint` sort is parameterized by a vertex via `SortParam::new("v", "Vertex")`. This is a GAT-level encoding of the fact that constraints are attached to specific schema elements. A `maxLength` constraint on a string field is a different sort instance than a `maxLength` constraint on a different field, even though they share the same sort name.

This dependent sort pattern has practical consequences throughout panproto:

- The migration engine's existence checker (Chapter 9) can scope constraint compatibility checks to individual vertices.
- The breaking change detector (Chapter 11) can attribute constraint tightening to the specific vertex affected.
- The lens system (Chapter 10) can track per-vertex constraint changes in the complement.

::: {.callout-note}
The dependent sort `Constraint(v: Vertex)` is a GAT-level encoding that ensures constraint-related operations are automatically scoped to the correct vertex, without requiring runtime dispatch or downcasting.[^fibered]

[^fibered]: The dependent sort `Constraint(v: Vertex)` means "the set of valid constraints depends on which vertex you are looking at." In the categorical literature, this is called a *fibered sort* [@cartmell1986], a sort whose elements are indexed by elements of another sort.
:::

::: {.callout-tip}
## Why dependent sorts instead of flat constraints?

If constraints were a flat sort (not parameterized by vertex), the existence checker couldn't distinguish "constraint C tightened on vertex A" from "constraint C tightened on vertex B." The dependent sort makes the scoping automatic.
:::

### Instance Theories

Beyond the schema theories, panproto defines instance theories that model how data is structured within a schema:

- **`th_wtype`**: W-type trees with `Node`, `Arc`, and `Value` sorts. Operations include `anchor` (mapping nodes to schema vertices), `arc_src`/`arc_tgt` (tree edges), `arc_edge` (labeling arcs with schema edges), and `node_value` (payload). Used by JSON-like, XML-like, and document protocols.

- **`th_functor`**: set-valued functors with `Table`, `Row`, and `ForeignKey` sorts. Operations include `table_vertex` (mapping tables to schema vertices) and `fk_src`/`fk_tgt` (foreign key endpoints). Used by relational and columnar protocols.

- **`th_flat`**: flat key-value structures with `Node`, `Field`, and `Value` sorts. Operations include `field_node` (owning node), `field_value` (payload), and `node_anchor` (schema vertex). Used by IDL protocols like Protobuf where data has a flat field structure without deep nesting.

- **`th_meta`**: metadata extension adding `Discriminator` and `ExtraField` sorts. Composes with `th_wtype` via structured merge over the shared `Node` sort. Used by document protocols with mixed content models (XML, HTML) where elements can have both structured children and unstructured attributes.

## The Structured Merge Pattern

Each protocol is defined by composing building-block theories via structured merge, an operation that glues theories together along shared sorts.[^colimit-pattern] The shared sorts act as the "meeting point" where two theories agree on what a concept means.

[^colimit-pattern]: The structured merge glues theories together along shared sorts, producing a combined theory where shared elements are unified. Formally, this is a *colimit* in the category of GATs. See @sec-gat-engine for the full definition.

For example, ATProto's schema theory is the structured merge of `ThGraph`, `ThConstraint`, and `ThMulti`, all sharing the `Vertex` sort:

```{mermaid}
graph TD
    ThGraph["ThGraph<br/>(Vertex, Edge, src, tgt)"]
    ThConstraint["ThConstraint<br/>(Vertex, Constraint)"]
    ThMulti["ThMulti<br/>(Vertex, Edge, EdgeLabel)"]
    SharedV["Shared: Vertex"]
    SharedVE["Shared: Vertex, Edge"]

    ThGraph -->|merge over| SharedV
    ThConstraint -->|merge over| SharedV
    SharedV --> GC["ThGraph + ThConstraint"]
    GC -->|merge over| SharedVE
    ThMulti -->|merge over| SharedVE
    SharedVE --> Schema["ATProto Schema Theory"]

    ThWType["ThWType<br/>(Node, Arc, Value)"] --> Instance["ATProto Instance Theory"]
```

The structured merge ensures that the `Vertex` sort in `ThGraph` and the `Vertex` sort in `ThConstraint` are identified: they refer to the same concept. The resulting schema theory has all the sorts and operations from both components, with shared sorts unified. This is not a mere union; the merge preserves the equations and operation signatures from both theories, ensuring that the composed theory is well-formed.

### Why This Construction?

The structured merge precisely controls which sorts are identified and which remain distinct. Alternative composition strategies either identify too much (forcing all sorts to be shared) or too little (sharing nothing).[^colimit-why]

[^colimit-why]: The structured merge precisely controls which concepts are shared and which remain independent. In category theory, this construction is a *colimit* (specifically a *pushout*), the universal way to combine theories while identifying shared structure.

In practice, this means:

- `ThGraph` and `ThConstraint` share `Vertex` but not `Edge` and `Constraint`. Edges and constraints are distinct concepts that happen to be attached to the same vertex sort.
- `ThGraph` and `ThMulti` share both `Vertex` and `Edge`. Multi-edge labels are an extension of the same edge concept.
- `ThWType` and `ThMeta` share `Node`. Metadata is an extension of the same node concept.

## Theory Groups

Protocols cluster into six theory groups based on their schema/instance theory pair. Each group has a corresponding `register_*` helper function that handles the merge computation:

| Group | Schema Theory | Instance Theory | Registration Helper |
|---|---|---|---|
| A | ThGraph + ThConstraint + ThMulti | ThWType | `register_constrained_multigraph_wtype` |
| B | ThHypergraph + ThConstraint | ThFunctor | `register_hypergraph_functor` |
| C | ThSimpleGraph + ThConstraint | ThFlat | `register_simple_graph_flat` |
| D | ThGraph + ThConstraint + ThMulti + ThInterface | ThWType | `register_typed_graph_wtype` |
| E | ThGraph + ThConstraint + ThMulti | ThWType + ThMeta | `register_multigraph_wtype_meta` |

### Group A: Constrained Multigraph + W-Type

The largest group, covering protocols with tree-shaped data and potentially parallel edges. ATProto, JSON Schema, OpenAPI, AsyncAPI, MongoDB, YAML Schema, TOML Schema, and many more belong here. The schema theory combines directed graphs (for type relationships), constraints (for validation rules like `maxLength`), and multi-edge support (for fields that share the same type). Instances are W-type trees.

### Group B: Hypergraph + Functor

Covers relational and columnar protocols: SQL, Cassandra, DynamoDB, Parquet, Arrow, DataFrame, CSV/Table Schema. The schema theory uses hypergraphs because tables are multi-arity relations (a table "connects" its columns). Instances are set-valued mappings; each table maps to a set of rows.[^functor-group-b]

[^functor-group-b]: Each table in a relational instance corresponds to a schema vertex, and each row is an element of that table. In the formal literature, these table-shaped mappings are called *set-valued functors* from the schema category to **Set**.

### Group C: Simple Graph + Flat

Covers IDL and serialization protocols: Protobuf, Avro, Thrift, Cap'n Proto, FlatBuffers, ASN.1, Bond, HCL, Redis. These protocols have flat field structures (no deep nesting within a single message/record) and enforce field uniqueness, hence the simple graph. Instances are flat key-value structures.

### Group D: Typed Graph + W-Type with Interfaces

Covers type systems and API schemas with inheritance: GraphQL, TypeScript, Python, Rust, Java, Go, Swift, Kotlin, C#. Adds the `ThInterface` theory to Group A, modeling constructs like GraphQL interfaces, TypeScript interfaces, and Java abstract classes.

### Group E: Multigraph + W-Type + Metadata

Covers document formats with mixed content: XML/XSD, HTML, CSS, DOCX, ODF, Neo4j. Combines Group A's schema theory with an instance theory that adds metadata support (discriminators and extra fields) for elements that have both structured children and unstructured attributes.

::: {.callout-tip}
When adding a new protocol, first determine which theory group it belongs to. If it fits an existing group, you can reuse the corresponding `register_*` function and only need to define the protocol's edge rules, vertex kinds, and parser/emitter.
:::

## The Protocol Interface

Each protocol module exports a `protocol()` function that returns a `Protocol` struct. The `Protocol` specifies:

- **`name`**: the protocol identifier (e.g., `"atproto"`, `"json_schema"`, `"sql"`).
- **`schema_theory`** and **`instance_theory`**: names referencing entries in the theory registry, populated by the `register_*` helpers.
- **`edge_rules`**: which edge kinds are structurally significant and what source/target vertex kinds they connect. Used by `panproto-check` to classify breaking changes (Chapter 11).
- **`obj_kinds`**: vertex kinds that represent composite objects (as opposed to scalar values like `"string"` or `"integer"`). Used during schema parsing and by the lens system.
- **`constraint_sorts`**: which constraint sorts this protocol recognizes (e.g., `"maxLength"`, `"pattern"`, `"maxGraphemes"`). Used by both the migration engine's existence checker and the breaking change detector.

The theory names in the `Protocol` struct reference entries in the theory registry, which is populated by the `register_*` helpers. This indirection keeps protocol definitions declarative: a protocol does not construct its own theories; it names them.

## The Two-Level Presentation Architecture

panproto's two-parameter architecture (schema theory + instance theory) is mirrored by two crates that provide the **presentation** layer (the parse/emit layer that converts native format syntax to panproto's internal representation):[^presentation-functor]

[^presentation-functor]: Each presentation layer converts between a native format (like `.proto` files or SQL DDL) and panproto's internal representation. In the formal literature, this conversion is a *functor* mapping syntactic objects to their algebraic models.

### Schema-Level Presentations (`panproto-protocols`)

Every protocol implements a bidirectional schema-level conversion:

- **Parser**: `parse_*(input: &str) -> Result<Schema, ProtocolError>` converts a native format _specification_ (`.proto` file, JSON Schema, SQL DDL, GraphQL SDL) into panproto's `Schema` representation, a model (a concrete instance that satisfies a theory's requirements, like a Rust struct implementing a trait) of the protocol's schema theory $T_P^S$.
- **Emitter**: `emit_*(schema: &Schema) -> Result<String, ProtocolError>` converts a `Schema` back into the native format specification.

Examples:

- `parse_json_schema(json_str)` parses a JSON Schema document into a `Schema`.
- `emit_protobuf(schema)` emits a `.proto` file from a `Schema`.
- `parse_sql(ddl_str)` parses SQL DDL statements into a `Schema`.

### Instance-Level Presentations (`panproto-io`)

The `panproto-io` crate provides the complementary **instance-level** conversions for all 76 protocols:

- **Parser**: `parse_wtype(protocol, schema, &bytes) -> Result<WInstance, ParseInstanceError>` converts raw _data_ (an HTML document, a CoNLL-U file, a Protobuf wire message) into a `WInstance` or `FInstance`, a model of the protocol's instance theory $T_P^I$.
- **Emitter**: `emit_wtype(protocol, schema, &instance) -> Result<Vec<u8>, EmitInstanceError>` converts an instance back to raw format bytes.

The choice of `WInstance` vs `FInstance` is determined by the protocol's `instance_theory` field, not by implementation preference.

### The Complete Pipeline

Together, these two levels of presentation complete the data migration pipeline:

```
raw spec ──protocols parse──→ Schema v1 ──mig compile──→ CompiledMigration
raw data ──io parse──→ Instance v1 ──inst restrict──→ Instance v2 ──io emit──→ raw data
```

The commutativity guarantee from Spivak 2012 ensures that parse, restrict, and emit compose correctly across all 76 protocols. Both levels support round-trip testing: `parse(emit(parse(input)))` should produce a structurally equivalent result.

::: {.callout-note}
## What does "commutativity" buy us here?

It means we don't need separate integration tests for every (source protocol, target protocol) pair. Correctness of the individual presentations plus the algebraic guarantee gives us correctness of the full pipeline.
:::

### Parsing Strategy

Parsers typically follow a two-phase approach:

1. **Syntactic parsing**: use a format-specific parser (e.g., `serde_json` for JSON Schema, `tree-sitter` for Protobuf) to produce an AST or intermediate representation.
2. **Semantic translation**: walk the AST, creating vertices for types and fields, edges for relationships, constraints for validation rules, and hyper-edges for multi-arity relations.

### Emitting Strategy

Emitters reverse the process:

1. **Topology traversal**: walk the schema graph, identifying root vertices, following edges in topological order.
2. **Format construction**: generate the native syntax for each vertex (type declaration, field definition, constraint annotation) and edge (relationship, containment).

## Adding a New Protocol

To add a new protocol to panproto, follow these five steps:

### Step 1: Identify Component Theories

Determine which building-block theories your protocol needs. Most protocols fit one of the five existing groups. Ask yourself:

- Does the protocol have directed edges between fields/types? Use `ThGraph`.
- Does the protocol support parallel edges (e.g., multiple fields with the same type)? Add `ThMulti`.
- Does the protocol have constraints (min/max length, required fields, patterns)? Add `ThConstraint`.
- Does the protocol have multi-arity relations (e.g., SQL foreign keys, table definitions)? Use `ThHypergraph` instead of `ThGraph`.
- Does the protocol support interfaces or inheritance? Add `ThInterface`.
- Does the protocol have flat field structures with no deep nesting? Use `ThFlat` as the instance theory.
- Does the protocol have tree-structured data? Use `ThWType`.
- Does the protocol have mixed content (structured + unstructured)? Add `ThMeta` to `ThWType`.

### Step 2: Compose via Structured Merge

If your protocol fits an existing group, call the corresponding `register_*` function with your theory names:

```rust
// In your protocol module's register function:
theories::register_constrained_multigraph_wtype(
    registry,
    "ThMyProtocolSchema",
    "ThMyProtocolInstance",
);
```

If your protocol needs a novel combination, compose the theories manually using `panproto_gat::colimit()`, specifying shared sorts at each step. Follow the pattern established by the existing `register_*` functions.

### Step 3: Define Edge Rules and Vertex Kinds

Create a `protocol()` function in your module:

```rust
pub fn protocol() -> Protocol {
    Protocol {
        name: "my_protocol".into(),
        schema_theory: "ThMyProtocolSchema".into(),
        instance_theory: "ThMyProtocolInstance".into(),
        edge_rules: vec![/* ... */],
        obj_kinds: vec!["object".into(), "message".into()],
        constraint_sorts: vec!["maxLength".into()],
    }
}
```

### Step 4: Implement Schema Presentation (`panproto-protocols`)

If your protocol has a schema definition language (e.g., `.proto`, DDL, SDL), write `parse_my_protocol` in the `panproto-protocols` crate to convert native format specifications into a `Schema`. Write `emit_my_protocol` to convert back.

Some protocols (like CoNLL-U) have no separate schema language; the schema is implicit in the format specification. These protocols only need Step 4b.

### Step 4b: Implement Instance Presentation (`panproto-io`)

Write an instance codec in `panproto-io` by implementing the `InstanceParser` and `InstanceEmitter` traits. For most protocols, this means choosing the right shared pathway:

- **JSON data**: use `JsonCodec::new("my_protocol")`, which delegates to the SIMD `simd-json` pathway.
- **XML data**: use `XmlCodec::new("my_protocol")`, which delegates to the `quick-xml` pull parser.
- **Tab/delimited data**: use `TabularCodec::tsv("my_protocol", "table_name")`, which delegates to the `memchr` SIMD pathway.
- **Custom format**: implement `InstanceParser` and `InstanceEmitter` directly using a dedicated parsing library.

Register the codec in the appropriate category module's `register_all` function and add it to the `default_registry` test.

### Step 5: Add Integration Tests

Add tests at both levels:

- **Schema round-trip** (`panproto-protocols`): parse a native specification, emit it back, verify the result matches.
- **Instance round-trip** (`panproto-io`): parse native data, emit it back, re-parse, verify structural equality. Use real-world fixture data, not synthetic examples.
- **Migration integration**: verify that `parse → restrict → emit` produces valid output.
- **Breaking-change tests**: verify classification logic for your protocol's edge rules.

::: {.callout-warning}
When composing theories via structured merge, the shared theory must contain exactly the sorts that should be identified. If you share too few sorts, the merge will create duplicate sorts that should be unified. If you share too many sorts, you may over-constrain the composition. Review the existing `register_*` functions for examples of correct sharing patterns.
:::

### Worked Example: A Hypothetical "TaskML" Protocol

Suppose we're defining a protocol for a task management markup language with:

- Tasks have fields (name, description, priority).
- Tasks can have subtasks (nesting).
- Fields have validation constraints (maxLength on name, enum values for priority).
- No parallel edges (each field name is unique within a task).

**Step 1**: This maps to Group C (simple graph + flat). The simple graph enforces field uniqueness; the flat instance theory models the non-nested field structure. But subtasks imply nesting, which requires W-type trees. So this is actually Group A (constrained multigraph + W-type), minus the multi-edge support since field names are unique. We could use Group A as-is (the multi-edge capability is unused but harmless) or define a custom composition of `ThSimpleGraph + ThConstraint` with `ThWType`.

**Step 2**: Using Group A for simplicity:

```rust
theories::register_constrained_multigraph_wtype(
    registry,
    "ThTaskMLSchema",
    "ThTaskMLInstance",
);
```

**Step 3**: Define the protocol:

```rust
pub fn protocol() -> Protocol {
    Protocol {
        name: "taskml".into(),
        schema_theory: "ThTaskMLSchema".into(),
        instance_theory: "ThTaskMLInstance".into(),
        edge_rules: vec![EdgeRule {
            edge_kind: "field".into(),
            src_kinds: vec!["task".into()],
            tgt_kinds: vec!["string".into(), "enum".into(), "integer".into()],
        }],
        obj_kinds: vec!["task".into()],
        constraint_sorts: vec!["maxLength".into(), "enumValues".into()],
    }
}
```

**Steps 4 and 5**: Implement `parse_taskml` and `emit_taskml`, then write round-trip and migration tests.

::: {.callout-tip}
## How do I know if my protocol needs ThHypergraph vs ThGraph?

If your protocol's core relations are binary (one source, one target), use `ThGraph`. If they're $n$-ary (like SQL foreign keys connecting multiple columns), use `ThHypergraph`. The edge structure of your format's native syntax usually makes this obvious.
:::

## Theoretical Foundations

The structured merge pattern is backed by a formal guarantee: the merge preserves the equations and well-formedness conditions of both component theories. If `ThGraph` guarantees that every edge has a source and target vertex, and `ThConstraint` guarantees that every constraint is attached to a vertex, then the merged theory inherits both guarantees.[^theory-morphism]

[^theory-morphism]: The structured merge combines building-block theories into a single theory, identifying shared sorts so they appear exactly once. Each building-block is a GAT, and the merge identifies sorts along the shared sub-theory. Formally, this is a colimit (pushout) in the category of GATs [@cartmell1986]. The protocol architecture follows @lynch2024, where data interchange protocols are presentations of algebraic theories and interoperability is mediated by structure-preserving maps (theory morphisms) between them. The `panproto-protocols` and `panproto-io` crates together implement both levels of the conversion pipeline described in Lynch 2024.

The `panproto-protocols` and `panproto-io` crates together implement two levels of the data pipeline: schema presentations map format specifications to abstract schema models, and instance presentations map format data to abstract instance models. This two-level factorization mirrors panproto's two-parameter architecture and ensures that the full pipeline (from raw bytes through migration to target format) composes correctly by construction.

For a hands-on guide to building your own protocol, see [tutorial chapter 10](https://panproto.dev/tutorial/chapters/10-building-your-own-protocol.html).
