17  VCS Engine

The panproto-vcs crate implements schematic version control: schemas are content-addressed objects stored in a commit DAG, with branches, three-way merge, and data lifting through history.

17.1 Module Layout

crates/panproto-vcs/src/

  error.rs         -- VcsError (thiserror, non_exhaustive)
  hash.rs          -- canonical serialization + blake3 content addressing
  object.rs        -- ObjectId, CommitObject, Object enum
  store.rs         -- Store trait, HeadState, ReflogEntry
  fs_store.rs      -- FsStore (.panproto/ on disk)
  mem_store.rs     -- MemStore (HashMap-backed, for tests + WASM)
  refs.rs          -- branches, tags, resolve_ref
  reflog.rs        -- append-only audit log of ref changes
  index.rs         -- staging area + validation pipeline
  dag.rs           -- merge base, path finding, log walk, compose_path
  merge.rs         -- three-way schema merge + conflict detection
  rebase.rs        -- replay commits onto a new base
  cherry_pick.rs   -- apply single commit's migration
  reset.rs         -- soft/mixed/hard HEAD reset
  stash.rs         -- save/restore working state
  bisect.rs        -- binary search for breaking commit

17.2 Object Store

ObjectId is a blake3 hash of the canonical MessagePack serialization. Canonical forms use BTreeMap instead of HashMap and exclude precomputed indices, which are derived data. Four object types live in the store:

  • Object::Schema(Schema): a schema snapshot
  • Object::Migration { src, tgt, mapping }: a morphism between two schemas
  • Object::Commit(CommitObject): a point in the DAG
  • Object::Tag(TagObject): an annotated tag storing target object ID, tagger name, timestamp, and message
pub struct CommitObject {
    pub schema_id: ObjectId,
    pub parents: Vec<ObjectId>,       // 0=root, 1=normal, 2=merge
    pub migration_id: Option<ObjectId>,
    pub protocol: String,
    pub author: String,
    pub timestamp: u64,
    pub message: String,
}

17.3 Merge Options

MergeOptions controls merge behavior:

pub struct MergeOptions {
    pub no_commit: bool,   // stage result without committing
    pub ff_only: bool,     // fail if not fast-forwardable
    pub no_ff: bool,       // force merge commit even on fast-forward
    pub squash: bool,      // collapse into a single non-merge commit
    pub message: Option<String>,  // custom commit message
}

17.4 Cherry-Pick and GC Options

CherryPickOptions controls cherry-pick behavior: no_commit stages without committing, record_origin appends the source commit ID to the message.

GcOptions has a single dry_run field. When true, gc::gc reports unreachable objects without deleting them.

17.5 DAG Algorithms

  • Merge base: two-frontier BFS, alternately expanding from each side
  • Path finding: BFS backwards from target, reconstruct via parent map
  • Log walk: max-heap by timestamp for topological-chronological order
  • Compose path: chain migrations via panproto_mig::compose
  • Is ancestor: BFS reachability check

17.6 Repository Operations

Beyond the core commit flow, repo.rs exposes several additional operations:

  • Repository::amend() – creates a commit with the same parent(s) as the current HEAD but updated schema, migration, and message. The previous HEAD commit becomes unreachable (collectible by gc).
  • refs::force_delete_branch – deletes a branch regardless of merge status
  • refs::rename_branch – renames a branch ref, updating HEAD if the current branch is renamed
  • refs::create_and_checkout_branch – creates a branch at HEAD and switches to it atomically
  • refs::create_annotated_tag – creates a TagObject in the store and points a tag ref at it
  • refs::create_tag_force – overwrites an existing tag ref
  • stash::stash_apply – re-applies the top stash entry without removing it from the stack
  • stash::stash_show – returns the schema diff between the stash entry and its parent
  • stash::stash_clear – drops all stash entries and removes the refs/stash ref

17.7 Error Variants

VcsError is #[non_exhaustive] and includes:

  • ObjectNotFound, RefNotFound, BranchNotFound, TagNotFound
  • BranchAlreadyExists, TagAlreadyExists
  • NotOnBranch, DetachedHead, IndexEmpty, IndexNotEmpty
  • MergeConflict, RebaseConflict, CherryPickConflict
  • FastForwardRequired, NothingToAmend
  • StashEmpty, InvalidRef, CorruptObject
  • RemoteNotSupported

17.8 Three-Way Merge (Combining Diverged Branches)

Given schemas base, ours, theirs, the merge computes a combined result:1

  1. diff(base, ours) and diff(base, theirs) – the extended diff covers all 13 schema fields (vertices, edges, constraints, hyper-edges, required, NSIDs, variants, orderings, recursion points, usage modes, spans, nominal)
  2. For every element in every field, classify its fate on each side: unchanged, added, removed, or modified
  3. Apply the merge rule: one-sided changes accepted; identical changes deduplicated; incompatible changes flagged as typed conflicts with the base value retained
  4. No tie-breaking – the merge is commutative: merge(base, A, B) == merge(base, B, A)
  5. Rebuild precomputed indices
  6. Derive migrations from ours->merged and theirs->merged via auto_mig::derive_migration

17.9 Rename Detection

The rename_detect module provides heuristic rename detection between schema versions. When auto_mig::derive_migration encounters a vertex that is absent in one schema version and a structurally similar vertex appears in the other, the rename detector can identify this as a rename rather than a delete-and-add.

17.9.1 Algorithm

For vertex rename detection (detect_vertex_renames):

  1. Collect removed vertices (in old, not in new) and added vertices (in new, not in old)
  2. For each (removed, added) pair, compute structural similarity:
    • Same vertex kind: +0.3
    • Same outgoing edge names: +0.3
    • Same incoming edge names: +0.2
    • Edit distance of vertex ID names <= 3: +0.2
  3. Greedy bipartite matching: take highest-scoring pairs first
  4. Return pairs >= threshold

For edge label rename detection (detect_edge_renames), the algorithm compares removed and added edges between the same surviving vertex pair, scoring by name edit distance normalized to string length.

17.9.2 CommitObject Integration

Detected renames are stored in CommitObject.renames as Vec<SiteRename>. This field is #[serde(default, skip_serializing_if = "Vec::is_empty")], so old commits without renames deserialize correctly.

See Section 18.2 for the full naming and identity architecture.

17.10 Related Work and Differentiation

The term “schematic version control” distinguishes panproto from “semantic version control” (AST-aware diffing tools for source code) and from schema registries (which store schema versions linearly without branching or merge).

17.10.1 Semantic/Structural Version Control for Source Code

These tools bring structure-awareness to source code via AST/CST parsing. They understand syntax but not the semantics of the data being versioned.

17.10.1.1 sem and weave (Ataraxy Labs)

sem is a Rust CLI that provides entity-level diff, blame, graph, and impact analysis for code across 16 languages via tree-sitter. It uses structural hashing to distinguish cosmetic changes from real logic changes. weave is its companion merge driver, which resolves conflicts at the entity level and achieves 31/31 clean merges on a benchmark where git manages 15/31.

Both tools sit on top of git. They do not replace git’s object store or DAG; they provide a smarter view of git’s existing data. sem parses source code into entity trees (functions, classes, interfaces). panproto-vcs operates on a different kind of object: mathematical schema structure expressed as GAT models with sorts (types in a formal theory, not sorting operations), operations, and equations. weave’s merge works by entity-level pattern matching. panproto-vcs’s merge computes the combined result over a shared base theory.2

17.10.1.2 Cow (Tikhon Jelvis)

Cow is a Haskell project that diffs and merges parse trees instead of text. It uses a modification of the Wagner-Fischer algorithm to compute tree edit distance, and can detect moves and group related changes.

Cow works on generic parse trees and is language-agnostic. panproto-vcs works on typed schema graphs where the types (sorts, operations, equations) constrain what counts as a valid change. Cow’s merge minimizes edit distance; panproto-vcs computes a merge with universal-property guarantees that edit-distance approaches lack.

17.10.1.3 difftastic (Wilfred Hughes)

difftastic is a Rust structural diff tool using tree-sitter for 30+ languages. It produces human-readable diffs that understand syntax and ignore whitespace, but it is purely a diff tool. It cannot merge, has no VCS, and has no schema awareness.

17.10.1.4 Mergiraf

Mergiraf is a Rust syntax-aware three-way merge tool for git using tree-sitter, supporting 33 languages. When the AST merge fails, it falls back to line-based merging.

The fallback is the important difference. Mergiraf is heuristic: when its structured approach fails, it silently degrades to a less-informed algorithm. panproto-vcs’s merge always produces a result. Conflicts are reported explicitly, never papered over.

17.10.1.5 SemanticMerge, Spork, JDime, jFSTMerge, IntelliMerge

SemanticMerge (Plastic SCM / Unity) is a commercial structure-based merge for C#, Java, VB.NET, and C. Spork is a Java-only AST merge using the 3DM-Merge algorithm. JDime, jFSTMerge, and IntelliMerge are academic structured merge tools for Java.

All of these are language-specific merge tools for source code. panproto-vcs is protocol-parametric: the same merge algorithm works across 77 data formats because it operates on the abstract schema graph.

17.10.2 Theoretical Proposals

17.10.2.1 Structure-aware VCS via observational bridge types (Topos Institute 2024)

This proposal uses logical relations and bridge types from HoTT to formally specify what constitutes a valid structural change. It applies parametricity to version control. There is no implementation.

This is the closest work in mathematical spirit. Both panproto-vcs and the Topos proposal use type-theoretic foundations to define valid changes. panproto-vcs uses GATs (Cartmell 1986); the Topos proposal uses bridge types from cubical type theory. The two approaches may be compatible: bridge types could serve as an alternative formalization of migration validity within panproto’s framework.

17.10.2.2 A Categorical Theory of Patches (Mimram and Di Giusto 2013)

Mimram and Di Giusto define a category where objects are files and morphisms are patches, with merge defined as a shared-structure merge.3 They investigate the free completion of this category under finite merges and give an explicit description where objects are finite sets labeled by lines equipped with a transitive relation.

panproto-vcs implements this idea for schemas rather than text files. The objects in Mimram and Di Giusto’s category are sequences of lines, a flat structure. panproto’s objects are models of GATs: typed graphs with protocol-specific sorts, operations, equations, and constraints. The richer structure gives the merge more to work with.

17.10.2.3 DVCS for Structured Data (Schürmann 2020)

Schurmann’s manifesto calls for version control over relational algebra and algebraic datatypes. It identifies the gap but provides no implementation. panproto-vcs realizes this vision.

17.10.3 Algebraic/Categorical VCS

17.10.3.1 Darcs

Darcs is a Haskell VCS where patches are algebraic objects with inverse and commutation operations. Merge works by finding a reordering of patches that lets both sides apply cleanly.

Darcs’s patches are syntactic: line insertions and deletions that can commute when they touch different lines. panproto’s migrations are semantic: vertex and edge maps that compose associatively and merge over shared structure. Darcs has no notion of typed structure.4

17.10.3.2 Pijul

Pijul is a Rust DVCS based on Mimram and Di Giusto (2013)’s categorical patch theory. Merge is the shared-structure merge in a category of files and patches, giving commutativity and associativity.5

Pijul’s objects are byte-chunk graphs, essentially ordered sequences of text blocks. panproto’s are models of GATs with protocol-specific theories, constraints, and edge rules. panproto’s merge has access to type information that lets it distinguish “both branches renamed vertex X” (conflict) from “both branches added unrelated vertices” (clean merge). Pijul cannot make this distinction. Additionally, panproto can migrate instance data through the DAG via lifting, which requires understanding the semantics of the objects being versioned.

17.10.4 Schema Migration Tools

These tools manage schema evolution but are not version control systems. They lack content-addressing, commit DAGs, branching, and structural merge.

17.10.4.1 Cambria (Litt et al. 2021)

Cambria is a JS/TS library for JSON schema evolution via bidirectional lenses, using combinators (rename, add, remove, hoist, wrap). It targets local-first and CRDT-based collaboration.

panproto subsumes Cambria’s combinator model in panproto-lens. Cambria operates only on JSON; panproto works across 76 protocols. Cambria has no VCS layer. Cambria’s convert combinator breaks the lens laws, as acknowledged by Litt et al. (2021); panproto preserves round-trip fidelity via explicit complement tracking.

17.10.4.2 buf (Buf Technologies)

Buf is a Go tool for Protobuf schema linting, breaking change detection, and schema registry. It integrates with CI/CD to enforce compatibility.

Buf is a registry and linter for a single protocol. It relies on git for version control, cannot migrate instance data, has no merge algorithm, and does not work across protocol boundaries.

17.10.4.3 Atlas (Ariga)

Atlas is a Go tool for declarative database schema management, supporting several SQL dialects. It generates migration SQL scripts from declarative schema definitions.

Atlas is database-specific and generates imperative migration scripts. It relies on git for version history. panproto-vcs is protocol-parametric and provides its own version control with structural merge.

17.10.4.4 Prisma Migrate, Flyway, Liquibase

These tools produce linear migration sequences (SQL scripts). No branching, no merge, no content-addressing.

17.10.5 Schema Registries

Schema registries store schema versions but do not provide version control. Versions are linearly ordered per subject with no branching or merge.

17.10.5.1 Confluent Schema Registry

Confluent’s Schema Registry provides a REST API for Avro, JSON Schema, and Protobuf schemas, with compatibility checking modes (BACKWARD, FORWARD, FULL, etc.) and linear version history per subject. Schemas are identified by sequential version numbers rather than by their structure.

17.10.5.2 SchemaVer, AWS Glue Schema Registry, Apicurio

SchemaVer is a versioning scheme (MODEL-REVISION-ADDITION), analogous to SemVer for schemas. panproto-vcs uses content-addressing: identity is derived from structure, not assigned. AWS Glue Schema Registry and Apicurio are cloud and self-hosted schema registries for event streaming, similar to Confluent.

## Three-Way Merge (Combining Diverged Branches)

Given schemas `base`, `ours`, `theirs`, the merge computes a combined result:^[The merge combines the two branch schemas along their shared base, producing a unified result. Categorically, this is a pushout (colimit) over the shared base.]

1. `diff(base, ours)` and `diff(base, theirs)` -- the extended diff covers all 13 schema fields (vertices, edges, constraints, hyper-edges, required, NSIDs, variants, orderings, recursion points, usage modes, spans, nominal)
2. For every element in every field, classify its fate on each side: unchanged, added, removed, or modified
3. Apply the merge rule: one-sided changes accepted; identical changes deduplicated; incompatible changes flagged as typed conflicts with the base value retained
4. No tie-breaking -- the merge is commutative: `merge(base, A, B) == merge(base, B, A)`
5. Rebuild precomputed indices
6. Derive migrations from ours->merged and theirs->merged via `auto_mig::derive_migration`

## Rename Detection

The `rename_detect` module provides heuristic rename detection between schema versions. When `auto_mig::derive_migration` encounters a vertex that is absent in one schema version and a structurally similar vertex appears in the other, the rename detector can identify this as a rename rather than a delete-and-add.

### Algorithm

For vertex rename detection (`detect_vertex_renames`):

1. Collect removed vertices (in old, not in new) and added vertices (in new, not in old)
2. For each (removed, added) pair, compute structural similarity:
   - Same vertex kind: +0.3
   - Same outgoing edge names: +0.3
   - Same incoming edge names: +0.2
   - Edit distance of vertex ID names <= 3: +0.2
3. Greedy bipartite matching: take highest-scoring pairs first
4. Return pairs >= threshold

For edge label rename detection (`detect_edge_renames`), the algorithm compares removed and added edges between the same surviving vertex pair, scoring by name edit distance normalized to string length.

### CommitObject Integration

Detected renames are stored in `CommitObject.renames` as `Vec<SiteRename>`. This field is `#[serde(default, skip_serializing_if = "Vec::is_empty")]`, so old commits without renames deserialize correctly.

Cow's merge minimizes edit distance; panproto-vcs computes a merge with universal-property guarantees that edit-distance approaches lack.

#### [difftastic](https://github.com/Wilfred/difftastic) (Wilfred Hughes)

difftastic is a Rust structural diff tool using tree-sitter for 30+ languages. It produces human-readable diffs that understand syntax and ignore whitespace, but it is purely a diff tool. It cannot merge, has no VCS, and has no schema awareness.

#### [Mergiraf](https://codeberg.org/mergiraf/mergiraf)

Mergiraf is a Rust syntax-aware three-way merge tool for git using tree-sitter, supporting 33 languages. When the AST merge fails, it falls back to line-based merging.

The fallback is the important difference. Mergiraf is heuristic: when its structured approach fails, it silently degrades to a less-informed algorithm. panproto-vcs's merge always produces a result. Conflicts are reported explicitly, never papered over.

#### SemanticMerge, Spork, JDime, jFSTMerge, IntelliMerge

SemanticMerge (Plastic SCM / Unity) is a commercial structure-based merge for C#, Java, VB.NET, and C. Spork is a Java-only AST merge using the 3DM-Merge algorithm. JDime, jFSTMerge, and IntelliMerge are academic structured merge tools for Java.

All of these are language-specific merge tools for source code. panproto-vcs is protocol-parametric: the same merge algorithm works across 77 data formats because it operates on the abstract schema graph.

### Theoretical Proposals

#### Structure-aware VCS via observational bridge types [@topos2024]

This proposal uses logical relations and bridge types from HoTT to formally specify what constitutes a valid structural change. It applies parametricity to version control. There is no implementation.

This is the closest work in mathematical spirit. Both panproto-vcs and the Topos proposal use type-theoretic foundations to define valid changes. panproto-vcs uses GATs [@cartmell1986]; the Topos proposal uses bridge types from cubical type theory. The two approaches may be compatible: bridge types could serve as an alternative formalization of migration validity within panproto's framework.

#### A Categorical Theory of Patches [@mimram2013]

Mimram and Di Giusto define a category where objects are files and morphisms are patches, with merge defined as a shared-structure merge.^[The merge combines structures along shared elements, producing a universal result. Categorically, this is a pushout (colimit).] They investigate the free completion of this category under finite merges and give an explicit description where objects are finite sets labeled by lines equipped with a transitive relation.

panproto-vcs implements this idea for schemas rather than text files. The objects in Mimram and Di Giusto's category are sequences of lines, a flat structure. panproto's objects are models of GATs: typed graphs with protocol-specific sorts, operations, equations, and constraints. The richer structure gives the merge more to work with.

#### DVCS for Structured Data [@schurmann2020]

Schurmann's manifesto calls for version control over relational algebra and algebraic datatypes. It identifies the gap but provides no implementation. panproto-vcs realizes this vision.

### Algebraic/Categorical VCS

#### [Darcs](http://darcs.net/)

Darcs is a Haskell VCS where patches are algebraic objects with inverse and commutation operations. Merge works by finding a reordering of patches that lets both sides apply cleanly.

Darcs's patches are syntactic: line insertions and deletions that can commute when they touch different lines. panproto's migrations are semantic: vertex and edge maps that compose associatively and merge over shared structure. Darcs has no notion of typed structure.^[Both [Darcs](http://darcs.net/) and [Pijul](https://pijul.org/) use algebraic patch theories, but their objects are untyped text (lines and insertions). panproto's objects are typed schema graphs with sorts, operations, and equations, which allows the merge to exploit type information for conflict detection.]

#### [Pijul](https://pijul.org/)

Pijul is a Rust DVCS based on @mimram2013's categorical patch theory. Merge is the shared-structure merge in a category of files and patches, giving commutativity and associativity.^[Both Pijul and panproto-vcs use pushouts for merge. The difference is in the objects being merged.]

Pijul's objects are byte-chunk graphs, essentially ordered sequences of text blocks. panproto's are models of GATs with protocol-specific theories, constraints, and edge rules. panproto's merge has access to type information that lets it distinguish "both branches renamed vertex X" (conflict) from "both branches added unrelated vertices" (clean merge). Pijul cannot make this distinction. Additionally, panproto can migrate instance data through the DAG via lifting, which requires understanding the semantics of the objects being versioned.

### Schema Migration Tools

These tools manage schema evolution but are not version control systems. They lack content-addressing, commit DAGs, branching, and structural merge.

#### Cambria [@litt2021]

Cambria is a JS/TS library for JSON schema evolution via bidirectional lenses, using combinators (rename, add, remove, hoist, wrap). It targets local-first and CRDT-based collaboration.

panproto subsumes Cambria's combinator model in panproto-lens. Cambria operates only on JSON; panproto works across 76 protocols. Cambria has no VCS layer. Cambria's `convert` combinator breaks the lens laws, as acknowledged by @litt2021; panproto preserves round-trip fidelity via explicit complement tracking.

#### [buf](https://buf.build/) (Buf Technologies)

[Buf](https://buf.build/) is a Go tool for Protobuf schema linting, breaking change detection, and schema registry. It integrates with CI/CD to enforce compatibility.

Buf is a registry and linter for a single protocol. It relies on git for version control, cannot migrate instance data, has no merge algorithm, and does not work across protocol boundaries.

#### [Atlas](https://atlasgo.io/) (Ariga)

Atlas is a Go tool for declarative database schema management, supporting several SQL dialects. It generates migration SQL scripts from declarative schema definitions.

Atlas is database-specific and generates imperative migration scripts. It relies on git for version history. panproto-vcs is protocol-parametric and provides its own version control with structural merge.

#### [Prisma Migrate](https://www.prisma.io/migrate), [Flyway](https://flywaydb.org/), [Liquibase](https://www.liquibase.com/)

These tools produce linear migration sequences (SQL scripts). No branching, no merge, no content-addressing.

### Schema Registries

Schema registries store schema versions but do not provide version control. Versions are linearly ordered per subject with no branching or merge.

#### [Confluent Schema Registry](https://docs.confluent.io/platform/current/schema-registry/)

Confluent's Schema Registry provides a REST API for Avro, JSON Schema, and Protobuf schemas, with compatibility checking modes (BACKWARD, FORWARD, FULL, etc.) and linear version history per subject. Schemas are identified by sequential version numbers rather than by their structure.

#### SchemaVer, AWS Glue Schema Registry, Apicurio

SchemaVer is a versioning scheme (MODEL-REVISION-ADDITION), analogous to SemVer for schemas. panproto-vcs uses content-addressing: identity is derived from structure, not assigned. AWS Glue Schema Registry and Apicurio are cloud and self-hosted schema registries for event streaming, similar to Confluent.
