18  Advanced Internals: Building Blocks, Naming, Auto-Morphisms, and Type Checking

This chapter covers four interconnected subsystems: the 27 building-block GATs that compose into protocol theories, the identity and naming layer, automatic morphism search and the extend/restrict (project data from the target schema onto the source schema, keeping only what the source schema expects)/complete triple, and the type-checking pipeline.

18.1 Building-Block Theories

panproto defines 27 building-block GATs that compose via colimit into protocol-specific schema and instance theories.

18.1.1 Formal Invariants

Every building-block theory satisfies:

  1. Sort closure: every sort (a type in a formal theory, not a sorting operation) name in an operation’s input/output types is declared in the theory’s sorts list.
  2. Equation well-typedness: both sides of every equation have the same output sort.
  3. No tautologies: lhs != rhs syntactically for all equations.

18.1.2 The Property-to-Structure Principle

Several building blocks encode conditional properties as unconditional equations by adding operations and dependent sorts.1

For example, injectivity of coproduct injection is not injection(v1) = injection(v2) -> v1 = v2 (a conditional equation, not expressible in GATs). Instead, ThCoproduct adds a retraction variant_of : Vertex -> Variant with the equation variant_of(injection(v)) = v. The retraction is structure, not a property.

This separation yields a clean boundary: structural properties (transitivity, injectivity, uniqueness, optionality) belong in GAT sorts+equations; validation properties (cardinality, patterns, ranges) belong in ThConstraint.

18.1.3 Theories with Equations

18.1.3.1 ThRecursion — Fold-Unfold

Sorts: Vertex, Mu
Ops:   fold : Vertex → Mu
       unfold : Mu → Vertex
Eq:    unfold(fold(v)) = v

Type check: fold(v) : Mu, unfold(fold(v)) : Vertex, v : Vertex. Both sides: Vertex.

This is the isorecursive fold-unfold law. fold is a section of unfold.

18.1.3.2 ThReflexiveGraph — Identity Edges

Sorts: Vertex, Edge
Ops:   id : Vertex → Edge
       src : Edge → Vertex
       tgt : Edge → Vertex
Eqs:   src(id(v)) = v
       tgt(id(v)) = v

Type check: id(v) : Edge, src(id(v)) : Vertex, v : Vertex. Both sides: Vertex.

Without the equations, this is just ThGraph + an extra operation. The equations make the identity edge load-bearing.

18.1.3.3 ThSymmetricGraph — Involution

Sorts: Vertex, Edge
Ops:   inv : Edge → Edge
       src : Edge → Vertex
       tgt : Edge → Vertex
Eqs:   src(inv(e)) = tgt(e)
       tgt(inv(e)) = src(e)
       inv(inv(e)) = e

Type checks:

  • src(inv(e)) : Vertex, tgt(e) : Vertex
  • inv(inv(e)) : Edge, e : Edge

18.1.3.4 ThCoproduct — Variant Retraction

Sorts: Vertex, Variant, Tag
Ops:   injection : Variant → Vertex
       tag : Variant → Tag
       variant_of : Vertex → Variant
Eq:    variant_of(injection(v)) = v

Type check: injection(v) : Vertex, variant_of(injection(v)) : Variant, v : Variant. Both sides: Variant.

18.1.3.5 ThPartial — Definedness Witness

Sorts: Vertex, Defined
Ops:   defined : Vertex → Defined
       witness : Defined → Vertex
Eq:    defined(witness(d)) = d

Type check: witness(d) : Vertex, defined(witness(d)) : Defined, d : Defined. Both sides: Defined.

18.2 Naming, Identity, and the Morphism Tower

This section covers the panproto-gat identifier abstraction, the unified naming layer, the morphism tower that cascades theory-level renames into data migrations, and the VCS rename detection heuristic.

For the user-facing introduction to naming, see the tutorial chapter on naming.

18.2.1 The Ident Abstraction

File: crates/panproto-gat/src/ident.rs

The Ident type separates stable identity from display name, following GATlab (Lynch et al. 2024):

pub struct Ident {
    pub scope: ScopeTag,   // which theory/schema this belongs to
    pub index: u32,        // positional index within scope
    pub name: Arc<str>,    // human-readable display name
}

PartialEq and Hash use only (scope, index), making comparisons O(1) regardless of name length. Renaming an identifier (via ident.renamed("new_name")) does not change its identity, so HashMap<Ident, _> entries survive renames without rehashing.

ScopeTag is a monotonically increasing u32 assigned via ScopeTag::fresh(). Each Theory or SchemaBuilder gets a unique scope at construction time. ScopeTag::LEGACY (scope 0) is used for deserializing old-format data via Ident::from_legacy().

18.2.2 The Name Type

File: crates/panproto-gat/src/ident.rs

Name wraps Arc<str> with a pointer-equality fast path:

pub struct Name(pub Arc<str>);

Equality checks use Arc::ptr_eq first (single pointer comparison), falling back to string comparison only on pointer mismatch. In the restrict hot path, both sides typically originate from the same schema construction, so the fast path hits frequently.

Key trait implementations:

  • Deref<Target=str> and Borrow<str>: HashMap<Name, V>::get("string_literal") works
  • From<String>, From<&str>: zero-friction conversion from string types
  • Default: empty string
  • serde::Serialize/Deserialize: transparent (serializes as a plain string)

Performance impact on wtype_restrict:

  • surviving_verts.contains(&node.anchor): same hash cost, gains Arc::ptr_eq fast path on equality
  • node.anchor.clone(): Arc::clone (atomic refcount bump) instead of heap allocation

18.2.3 NameSite and SiteRename

File: crates/panproto-gat/src/ident.rs

All 9 naming sites in panproto are unified under the NameSite enum:

pub enum NameSite {
    EdgeLabel, VertexId, VertexKind, EdgeKind,
    Nsid, ConstraintSort, InstanceAnchor,
    TheoryName, SortName,
}

SiteRename { site, old, new } is a site-qualified rename operation. These are stored in SchemaMorphism provenance and CommitObject rename metadata.

18.2.4 TheoryMorphism.induce_schema_renames()

File: crates/panproto-gat/src/morphism.rs

A theory morphism F: T1 -> T2 with sort_map and op_map induces schema-level renames:

  • Each sort_map entry where old != new becomes a NameSite::VertexKind rename
  • Each op_map entry where old != new becomes a NameSite::EdgeKind rename

This is analogous to GATlab’s SimpleTheoryMap, which maps type and term constructors between theories while preserving structural equivalence.

18.2.5 SchemaMorphism

File: crates/panproto-schema/src/morphism.rs

An explicit schema morphism (a map F: S -> T):

pub struct SchemaMorphism {
    pub name: String,
    pub src_protocol: String,
    pub tgt_protocol: String,
    pub vertex_map: HashMap<Name, Name>,
    pub edge_map: HashMap<Edge, Edge>,
    pub renames: Vec<SiteRename>,  // provenance
}

Composition via m1.compose(&m2) chains vertex/edge maps and concatenates renames. This makes the structure of schema transformations explicit, enabling transitive composition of cross-protocol translations.

18.2.6 The Cascade Module

File: crates/panproto-mig/src/cascade.rs

The morphism tower connects three levels:

Theory morphism  ──induce_schema_morphism──→  Schema morphism
                                                    │
                                          induce_data_migration
                                                    │
                                                    ▼
                                            CompiledMigration
  • induce_schema_morphism(theory_morph, src_schema): Applies sort/op renames to produce a SchemaMorphism
  • induce_data_migration(schema_morph, tgt_schema): Lowers to CompiledMigration for the restrict pipeline
  • induce_migration_from_theory(theory_morph, src, tgt): Convenience function chaining both

This implements the action described in Spivak (2012): a schema morphism F induces the pullback on instances.2

18.2.7 New Combinators

File: crates/panproto-lens/src/combinators.rs

Seven new combinator variants complement the original six:

Combinator Site Lossless? Notes
RenameVertex VertexId Yes Cascades to edges, constraints, variants
RenameKind VertexKind Yes Fine-grained (single vertex)
RenameEdgeKind EdgeKind Yes All edges with matching kind
RenameNsid Nsid Yes Single vertex
RenameConstraintSort ConstraintSort Yes All constraints
ApplyTheoryMorphism Multiple Yes Top of the morphism tower
Rename Any Yes Unified dispatcher

Each combinator has a corresponding apply_* function and build_compiled_migration arm. All are lossless (empty complement), so the lens laws hold trivially.

18.2.8 VCS Rename Detection

File: crates/panproto-vcs/src/rename_detect.rs

18.2.8.1 Algorithm

When comparing old and new schemas:

  1. Collect removed vertices (in old, not in new) and added vertices (in new, not in old)
  2. For each (removed, added) pair, compute structural similarity score:
    • Same vertex kind: +0.3
    • Same outgoing edge names: +0.3
    • Same incoming edge names: +0.2
    • Edit distance of names <= 3: +0.2
  3. Greedy bipartite matching: take highest-scoring pairs first
  4. Return pairs >= threshold (default 0.5)

18.2.8.2 Integration

Detected renames are stored in CommitObject.renames as Vec<SiteRename>. The schema diff --detect-renames CLI command exposes rename detection.

18.2.9 Migration Path: String to Name

The core types (Vertex, Edge, Schema, Node, CompiledMigration, Migration) use Name for all identifier and label fields. Constraint.value and Schema.protocol remain String since they are literal data, not identifiers.

The SchemaBuilder API still accepts &str parameters, converting to Name internally. Protocol parsers required no changes.

Performance: Benchmarks show no regression from the String to Name migration. The Arc::clone fast path eliminates heap allocations in the restrict pipeline’s inner loop.

18.3 Automatic Morphisms: Extend, Restrict, and Complete

This section covers the automatic homomorphism search, the extend/restrict/complete operations,3 schema-level pushouts, overlap discovery, and the chase algorithm.

For the user-facing introduction, see the tutorial chapter on automatic migration.

18.3.1 Homomorphism Search

File: crates/panproto-mig/src/hom_search.rs

18.3.1.1 CSP Reduction

The homomorphism problem reduces to a constraint satisfaction problem:

  • Variables: one per source vertex
  • Domains: target vertices with compatible kind
  • Constraints: edge-preserving squares

18.3.1.2 Algorithm

Backtracking search with forward checking:

  1. Variable ordering: MRV heuristic — assign vertices with smallest domains first
  2. Value selection: try each domain value
  3. Forward check: after assignment, verify all unassigned neighbors still have valid domain values via has_compatible_edge
  4. Edge compatibility: requires matching edge kind between mapped vertex pairs; edge names are allowed to differ (name differences only affect quality scoring)
  5. Edge map derivation: once all vertices are assigned, the edge map is determined by finding compatible edges between each pair of mapped vertices, preferring name-matching edges

18.3.1.3 Search Options

Option Effect
monic Injective vertex map (tracks used_targets set)
epic Surjective vertex map (checked at base case)
iso Bijective (monic + epic)
max_results Early termination bound
initial Pre-assigned mappings (domain restricted to singletons)

18.3.1.4 Quality Scoring

compute_quality produces a score in [0.0, 1.0]:

  • 50% weight: average name similarity (1 - edit_distance / max_length) across vertex pairs
  • 50% weight: fraction of edges where source and target have the same label name

18.3.1.5 Performance

Schemas have O(10-100) vertices. The backtracking search with MRV and forward checking handles this scale easily. Worst case is exponential in the number of vertices with identical kinds, but the edge constraints prune aggressively.

18.3.2 Extend (Left Kan Extension)

Files: crates/panproto-inst/src/wtype.rs (wtype_extend), crates/panproto-inst/src/functor.rs (functor_extend)

18.3.2.1 W-Type Extend

wtype_extend maps all source nodes into the target schema:

  1. Remap node anchors via vertex_remap
  2. Remap arc edges via edge_remap
  3. Rebuild tree indices

Unlike wtype_restrict, extend keeps all mappable nodes rather than dropping unmapped ones.

18.3.2.2 Functor Extend

functor_extend (pre-existing) handles the relational case:

  1. Copy tables forward via vertex_remap
  2. Compute coproduct (disjoint union) when multiple sources map to one target
  3. Fill missing columns with Value::Null
  4. Offset FK row indices for concatenated tables

18.3.3 Complete (Right Kan Extension)

File: crates/panproto-inst/src/pi.rs

18.3.3.1 Functor Complete

functor_pi computes the product over fibers:

  1. Build fiber map: for each target vertex, collect source vertices mapping to it
  2. Single-element fibers: copy table directly
  3. Multi-element fibers: Cartesian product of rows with column union
  4. Size gate: reject products exceeding max_product_size

18.3.3.2 W-Type Complete

wtype_pi currently only handles injective migrations (single-element fibers). Multi-element fiber products for trees require Cartesian products of subtrees, which is deferred to a future release.

18.3.4 Schema Pushout

File: crates/panproto-schema/src/colimit.rs

18.3.4.1 SchemaOverlap

Specifies which elements of two schemas are identified:

pub struct SchemaOverlap {
    pub vertex_pairs: Vec<(Name, Name)>,
    pub edge_pairs: Vec<(Edge, Edge)>,
}

18.3.4.2 schema_pushout Algorithm

  1. Merged vertices: all left vertices + non-overlapping right vertices (overlapping right vertices merge with their left counterpart)
  2. Merged edges: all left edges + right edges with remapped vertex references
  3. Constraint/metadata union for overlapping vertices
  4. Build morphisms: left->pushout (identity for left vertices), right->pushout (identity or remapped for right vertices)
  5. Rebuild adjacency indices

18.3.4.3 Universal Property

For any schema C with morphisms from both left and right that agree on the overlap, there exists a unique morphism from the pushout to C.

18.3.5 Overlap Discovery

File: crates/panproto-mig/src/overlap.rs

Uses find_best_morphism with monic: true in both directions (left->right and right->left), taking whichever produces more vertex pairs. The vertex map entries from the best morphism define the overlap pairs.

18.3.6 Chase Algorithm

File: crates/panproto-mig/src/chase.rs

18.3.6.1 Embedded Dependencies

pub struct EmbeddedDependency {
    pub pattern_vertex: Name,
    pub pattern_values: HashMap<String, Value>,
    pub consequence_vertex: Name,
    pub consequence_values: HashMap<String, Value>,
}

18.3.6.2 Algorithm

  1. Clone instance
  2. For each iteration (up to max_iterations):
    1. For each dependency, find pattern matches without consequences
    2. Add missing consequence rows
    3. If nothing changed, return fixpoint
  3. If limit reached, return ChaseError::NonTermination

18.3.6.3 Termination

Guaranteed for acyclic dependencies (no dependency’s consequence can trigger itself). For cyclic dependencies, the iteration bound prevents infinite loops. panproto’s equational theories are purely algebraic, so termination is guaranteed for all standard schema theories.

18.3.7 From Morphisms to Protolenses

The homomorphism search, extend/restrict/complete operations, and overlap discovery described here feed directly into the protolens engine (Chapter 19) and the automated lens generation pipeline (Chapter 20).

18.3.7.1 Pipeline Connection

The key integration point is the factorize function in panproto-gat/src/factorize.rs. Given a TheoryMorphism produced by hom_search, factorize decomposes it into elementary TheoryEndofunctor steps. Each step maps to an elementary protolens constructor, producing a ProtolensChain that can be instantiated at any schema satisfying the preconditions.

This extends the morphism tower from three levels to four:

  1. Theory morphism (this section): structure-preserving map between GATs.
  2. Factorization (Section 20.3): decomposition into elementary transforms.
  3. Protolens chain (Chapter 19): schema-parameterized family of lenses.
  4. Concrete lens (Chapter 10): instantiated lens for a specific schema pair.

18.3.7.2 Overlap and Symmetric Lenses

The overlap discovery algorithm (find_best_morphism with monic: true) is reused by the protolens pipeline to construct symmetric lenses. Given two schemas with partial overlap, the pipeline factorizes the morphisms from each schema to the overlap, constructs protolens chains in both directions, and combines them into a SymmetricLens. See Section 20.5 for details.

18.3.7.3 Performance Implications

Because hom_search is the most expensive step (exponential worst case), the automated lens pipeline caches morphism search results. Once a morphism is found, factorization and protolens construction are both linear in the theory size, so the incremental cost of producing a protolens chain from a cached morphism is negligible.

18.4 The Type-Checking Pipeline

Type-checking in panproto operates at three levels, each building on the one below.

18.4.1 Pipeline Overview

Level 0 – GAT (panproto-gat): The typecheck module provides the primitive type-checking functions. These operate on Term, Equation, and Theory values and return Result<_, GatError>. This level has no knowledge of schemas, migrations, or the VCS.

Level 1 – VCS (panproto-vcs): The gat_validate module wraps the GAT-level functions and collects their results into GatDiagnostics, a structured report of type errors, equation violations, and migration warnings. The index module stores these diagnostics in the staging area, and the repo module consults them during commit.

Level 2 – CLI (panproto-cli): The CLI commands convert VcsError::ValidationFailed into user-facing miette diagnostics with source context, suggestions, and error codes.

18.4.2 Level 0: GAT Type-Checking

The GAT level provides four public functions that form the foundation of the pipeline:

18.4.2.1 typecheck_term

pub fn typecheck_term(
    term: &Term,
    ctx: &VarContext,
    theory: &Theory,
) -> Result<Arc<str>, GatError>

Given a term, a variable typing context, and a theory, infers the output sort of the term. This is the recursive core: for App { op, args }, it looks up op in the theory, typechecks each argument recursively, checks that each argument’s inferred sort matches the operation’s declared input sort, and returns the output sort.

18.4.2.2 infer_var_sorts

pub fn infer_var_sorts(eq: &Equation, theory: &Theory) -> Result<VarContext, GatError>

Infers variable sorts from an equation by walking both sides and collecting sort constraints from operation application sites. This eliminates the need for explicit sort annotations on equation variables — the sorts are determined by the positions where variables appear.

18.4.2.3 typecheck_equation

pub fn typecheck_equation(eq: &Equation, theory: &Theory) -> Result<(), GatError>

Combines infer_var_sorts and typecheck_term: infers the variable context, typechecks both sides, and verifies they produce the same output sort.

18.4.2.4 typecheck_theory

pub fn typecheck_theory(theory: &Theory) -> Result<(), GatError>

Iterates over all equations in a theory and typechecks each one. Returns the first error encountered. This is the function that the VCS layer calls to validate protocol theories.

18.4.2.5 Error Types

All type-checking errors are variants of GatError:

These error types carry enough structured data for downstream layers to produce precise, actionable diagnostics.

18.4.3 Level 1: VCS Integration

The VCS layer integrates type-checking at three points in the schema evolution workflow: staging (schema add), committing (schema commit), and theory validation.

18.4.3.1 GatDiagnostics

The GatDiagnostics struct in panproto-vcs::gat_validate is the bridge between GAT-level errors and VCS-level validation:

pub struct GatDiagnostics {
    pub type_errors: Vec<String>,
    pub equation_errors: Vec<String>,
    pub migration_warnings: Vec<String>,
}
  • type_errors collects results from typecheck_theory. Each entry is the Display output of a GatError.
  • equation_errors collects results from check_model_with_options. Each entry describes a specific equation violation with its variable assignment and the values that disagreed.
  • migration_warnings collects structural warnings from validate_migration — these are non-blocking issues like empty vertex maps or incoherent edge endpoints.

The is_clean() method returns true only when both type_errors and equation_errors are empty. Migration warnings do not block commits.

18.4.3.2 Integration Point: schema add (Repository::add)

When Repository::add stages a schema, it runs GAT validation on the auto-derived migration:

  1. Compute the schema diff between HEAD and the new schema.
  2. Call auto_mig::derive_migration to produce a Migration.
  3. Call gat_validate::validate_migration to check migration structure — the result populates GatDiagnostics.
  4. Store the diagnostics in the StagedSchema::gat_diagnostics field.
  5. If gat_diagnostics.has_errors(), set ValidationStatus::Invalid(diag.all_errors()).

The index file (.panproto/index.json) persists the diagnostics so that schema commit can consult them without re-running validation.

18.4.3.3 Integration Point: schema commit (Repository::commit_with_options)

When Repository::commit_with_options creates a commit, it checks two gates:

  1. ValidationStatus: if ValidationStatus::Invalid, the commit is blocked with VcsError::ValidationFailed.
  2. GatDiagnostics: even if ValidationStatus is Valid, the commit checks gat_diagnostics.has_errors() as a secondary gate. This handles the case where validation status was manually set to Valid but diagnostics were populated later.

Both gates are bypassed when CommitOptions::skip_verify is true. The --skip-verify CLI flag sets this option, providing an escape hatch for work-in-progress schemas.

pub struct CommitOptions {
    pub skip_verify: bool,
}

18.4.3.4 Integration Point: Theory Validation

The validate_theory_equations function provides standalone theory type-checking:

pub fn validate_theory_equations(theory: &Theory) -> GatDiagnostics

This is called by the schema validate and schema typecheck CLI commands for schemas that are not part of a VCS repository. It runs typecheck_theory and collects any errors into GatDiagnostics::type_errors.

The validate_schema_equations function goes further by building a model (a concrete instance that satisfies a theory’s requirements — like a Rust struct implementing a trait) from the schema and running check_model_with_options:

pub fn validate_schema_equations(
    schema: &Schema,
    theory: &Theory,
    model: &Model,
) -> GatDiagnostics

This checks whether the schema’s interpretation of the theory satisfies all equations. The check uses CheckModelOptions::max_assignments = 10_000 to bound enumeration time. If the assignment count exceeds the limit, the error is reported as a non-blocking diagnostic rather than a hard failure.

18.4.4 Level 2: CLI Error Formatting

The CLI layer converts VcsError::ValidationFailed into user-facing diagnostics using the miette crate.

18.4.4.1 Error Flow

GatError::ArgTypeMismatch { op: "src", arg_index: 0, expected: "Edge", got: "Vertex" }
  -> GatDiagnostics { type_errors: ["type mismatch for src argument 0: expected Edge, got Vertex"] }
    -> ValidationStatus::Invalid(["type error: type mismatch for src argument 0: ..."])
      -> VcsError::ValidationFailed { reasons: ["type error: ..."] }
        -> miette::Report with labeled span and suggestion

Each layer adds context:

  • GatError: the raw error with structured fields (op, arg_index, expected, got).
  • GatDiagnostics: the error serialized to a string via Display, prefixed with its category ("type error: " or "equation violation: ").
  • ValidationStatus: the collected error strings from all_errors().
  • VcsError: the error wrapped in a ValidationFailed variant with a reasons field.
  • miette: the error rendered with ANSI colors, source labels, and help text.

18.4.4.2 Equation Violation Formatting

Equation violations receive special formatting in gat_validate::format_violation:

equation 'assoc' violated when a=2, b=3, c=1: LHS=6, RHS=4

This format shows the equation name, the variable assignment that triggered the violation, and the concrete values that the two sides evaluated to. For models with symbolic term representations (as produced by free_model), the values are printed as term strings:

equation 'left_id' violated when a=mul(unit(), unit()): LHS=mul(unit(), mul(unit(), unit())), RHS=mul(unit(), unit())

18.4.5 Cross-Cutting Integration Points

The type-checking pipeline touches several subsystems beyond the primary add/commit path:

18.4.5.1 auto_mig

The auto_mig::derive_migration function in panproto-vcs produces the migration that is subsequently validated. While auto_mig itself does not call type-checking functions, the quality of its output directly affects whether GAT validation passes. A poorly-derived migration (e.g., one that maps vertices to vertices of incompatible kinds) will produce GatDiagnostics with migration warnings.

18.4.5.2 Index Persistence

The StagedSchema struct in panproto-vcs::index stores gat_diagnostics: Option<GatDiagnostics>. This field is serialized to .panproto/index.json via serde. The #[serde(default)] attribute ensures backward compatibility: index files written before the GAT completeness feature was added deserialize with gat_diagnostics: None, which is treated as “no diagnostics available” rather than “diagnostics clean”.

18.4.5.3 Merge and Rebase

The merge and rebase pipelines do not currently run GAT type-checking on the merged result. Three-way merge operates at the schema level (vertices, edges, constraints), and the merged schema is validated structurally. GAT validation is deferred to the next schema add that stages the merged schema for further evolution.

The merge pipeline does use GAT-level operations: it calls pullback to compute structural overlap between branches, and it constructs migration morphisms that are validated by check_morphism. These are correctness checks on the merge algorithm itself, not on the user’s schema.

18.4.6 Design Rationale

18.4.6.1 Why Three Levels

The three-level design separates concerns cleanly:

  • Level 0 is pure mathematics. It knows about terms, sorts, and operations, nothing about files, schemas, or version control. This makes it easy to test in isolation and reuse in non-VCS contexts (e.g., the WASM boundary, the TypeScript SDK).
  • Level 1 is integration logic. It knows how to translate between GAT errors and VCS concepts (staging, commits, validation status). It serializes diagnostics for persistence.
  • Level 2 is presentation. It knows how to render errors for humans with colors, source labels, and suggestions. It uses miette, which is a CLI-only dependency.

18.4.6.2 Why Store Diagnostics in the Index

Storing GatDiagnostics in the index rather than re-computing them at commit time has two benefits:

  1. Performance: type-checking and model checking can be expensive for large theories. Running them once at staging time and persisting the results avoids redundant work.
  2. Auditability: the index file records exactly what diagnostics were produced, even if the user later modifies the schema file. The commit gate checks the stored diagnostics, not the current file state.

18.4.6.3 Why skip_verify Exists

The --skip-verify flag is an explicit escape hatch for situations where the type-checking pipeline produces false positives or where a work-in-progress schema intentionally violates equations. Without this flag, users would be blocked from committing legitimate intermediate states. The flag is named to make its risk visible: you are skipping verification, not disabling a cosmetic check.

18.4.6.4 Why Equation Checking is Bounded

The max_assignments bound in CheckModelOptions prevents combinatorial explosion. An equation with 3 variables over a carrier of 100 elements requires 1,000,000 assignments; with 4 variables it is 100,000,000. The default bound of 10,000 catches most violations while keeping verification fast. Users who need exhaustive checking can raise the bound via schema verify --max-assignments.

# Advanced Internals: Building Blocks, Naming, Auto-Morphisms, and Type Checking {#sec-advanced-internals}

This chapter covers four interconnected subsystems: the 27 building-block GATs that compose into protocol theories, the identity and naming layer, automatic morphism search and the extend/restrict (project data from the target schema onto the source schema, keeping only what the source schema expects)/complete triple, and the type-checking pipeline.

## Building-Block Theories {#sec-building-blocks-section}

panproto defines 27 building-block GATs that compose via colimit into protocol-specific schema and instance theories.

### Formal Invariants

Every building-block theory satisfies:

1. **Sort closure**: every sort (a type in a formal theory, not a sorting operation) name in an operation's input/output types is declared in the theory's sorts list.
2. **Equation well-typedness**: both sides of every equation have the same output sort.
3. **No tautologies**: `lhs != rhs` syntactically for all equations.

### The Property-to-Structure Principle

Several building blocks encode conditional properties as unconditional equations by adding operations and dependent sorts.^[Rather than expressing injectivity as a conditional equation (which GATs cannot express), panproto adds a retraction operation that witnesses the property structurally. This is Cartmell's property-to-structure move [@cartmell1986].]

For example, injectivity of coproduct injection is not `injection(v1) = injection(v2) -> v1 = v2` (a conditional equation, not expressible in GATs). Instead, ThCoproduct adds a retraction `variant_of : Vertex -> Variant` with the equation `variant_of(injection(v)) = v`. The retraction is *structure*, not a *property*.

This separation yields a clean boundary: structural properties (transitivity, injectivity, uniqueness, optionality) belong in GAT sorts+equations; validation properties (cardinality, patterns, ranges) belong in ThConstraint.

### Theories with Equations

#### ThRecursion --- Fold-Unfold

```
Sorts: Vertex, Mu
Ops:   fold : Vertex → Mu
       unfold : Mu → Vertex
Eq:    unfold(fold(v)) = v
```

Type check: `fold(v) : Mu`, `unfold(fold(v)) : Vertex`, `v : Vertex`. Both sides: Vertex.

This is the isorecursive fold-unfold law. `fold` is a section of `unfold`.

#### ThReflexiveGraph --- Identity Edges

```
Sorts: Vertex, Edge
Ops:   id : Vertex → Edge
       src : Edge → Vertex
       tgt : Edge → Vertex

       tgt(id(v)) = v
```

Type check: `id(v) : Edge`, `src(id(v)) : Vertex`, `v : Vertex`. Both sides: Vertex.

Without the equations, this is just ThGraph + an extra operation. The equations make the identity edge load-bearing.

#### ThSymmetricGraph --- Involution

```
Sorts: Vertex, Edge
Ops:   inv : Edge → Edge
       src : Edge → Vertex
       tgt : Edge → Vertex
Eqs:   src(inv(e)) = tgt(e)
       tgt(inv(e)) = src(e)
       inv(inv(e)) = e
```

Type checks:

- `src(inv(e)) : Vertex`, `tgt(e) : Vertex`
- `inv(inv(e)) : Edge`, `e : Edge`

#### ThCoproduct --- Variant Retraction

```
Sorts: Vertex, Variant, Tag
Ops:   injection : Variant → Vertex
       tag : Variant → Tag
       variant_of : Vertex → Variant
Eq:    variant_of(injection(v)) = v
```

Type check: `injection(v) : Vertex`, `variant_of(injection(v)) : Variant`, `v : Variant`. Both sides: Variant.

#### ThPartial --- Definedness Witness

```
Sorts: Vertex, Defined
Ops:   defined : Vertex → Defined
       witness : Defined → Vertex
Eq:    defined(witness(d)) = d
```

Type check: `witness(d) : Vertex`, `defined(witness(d)) : Defined`, `d : Defined`. Both sides: Defined.

## Naming, Identity, and the Morphism Tower {#sec-naming-identity-section}

This section covers the `panproto-gat` identifier abstraction, the unified naming layer, the morphism tower that cascades theory-level renames into data migrations, and the VCS rename detection heuristic.

For the user-facing introduction to naming, see the [tutorial chapter on naming](@sec-naming).

### The `Ident` Abstraction

**File**: `crates/panproto-gat/src/ident.rs`

The `Ident` type separates stable identity from display name, following [GATlab](https://algebraicjulia.github.io/GATlab.jl/) [@lynch2024]:

```rust
pub struct Ident {
    pub scope: ScopeTag,   // which theory/schema this belongs to
    pub index: u32,        // positional index within scope
    pub name: Arc<str>,    // human-readable display name
}
```

`PartialEq` and `Hash` use only `(scope, index)`, making comparisons O(1) regardless of name length. Renaming an identifier (via `ident.renamed("new_name")`) does not change its identity, so `HashMap<Ident, _>` entries survive renames without rehashing.

`ScopeTag` is a monotonically increasing `u32` assigned via `ScopeTag::fresh()`. Each `Theory` or `SchemaBuilder` gets a unique scope at construction time. `ScopeTag::LEGACY` (scope 0) is used for deserializing old-format data via `Ident::from_legacy()`.

### The `Name` Type

**File**: `crates/panproto-gat/src/ident.rs`

`Name` wraps `Arc<str>` with a pointer-equality fast path:

```rust
pub struct Name(pub Arc<str>);
```

Equality checks use `Arc::ptr_eq` first (single pointer comparison), falling back to string comparison only on pointer mismatch. In the restrict hot path, both sides typically originate from the same schema construction, so the fast path hits frequently.

Key trait implementations:

- `Deref<Target=str>` and `Borrow<str>`: `HashMap<Name, V>::get("string_literal")` works
- `From<String>`, `From<&str>`: zero-friction conversion from string types
- `Default`: empty string
- `serde::Serialize/Deserialize`: transparent (serializes as a plain string)

**Performance impact on `wtype_restrict`**:

- `surviving_verts.contains(&node.anchor)`: same hash cost, gains `Arc::ptr_eq` fast path on equality
- `node.anchor.clone()`: `Arc::clone` (atomic refcount bump) instead of heap allocation

### `TheoryMorphism.induce_schema_renames()`

**File**: `crates/panproto-gat/src/morphism.rs`

- Each `sort_map` entry where `old != new` becomes a `NameSite::VertexKind` rename

This is analogous to [GATlab](https://algebraicjulia.github.io/GATlab.jl/)'s `SimpleTheoryMap`, which maps type and term constructors between theories while preserving structural equivalence.

### The Cascade Module

**File**: `crates/panproto-mig/src/cascade.rs`

### VCS Rename Detection

**File**: `crates/panproto-vcs/src/rename_detect.rs`

#### Algorithm

When comparing old and new schemas:

1. Collect removed vertices (in old, not in new) and added vertices (in new, not in old)
2. For each (removed, added) pair, compute structural similarity score:
   - Same vertex kind: +0.3
   - Same outgoing edge names: +0.3
   - Same incoming edge names: +0.2
   - Edit distance of names <= 3: +0.2
3. Greedy bipartite matching: take highest-scoring pairs first
4. Return pairs >= threshold (default 0.5)

#### Integration

Detected renames are stored in `CommitObject.renames` as `Vec<SiteRename>`. The `schema diff --detect-renames` CLI command exposes rename detection.

### Migration Path: `String` to `Name`

The core types (`Vertex`, `Edge`, `Schema`, `Node`, `CompiledMigration`, `Migration`) use `Name` for all identifier and label fields. `Constraint.value` and `Schema.protocol` remain `String` since they are literal data, not identifiers.

The `SchemaBuilder` API still accepts `&str` parameters, converting to `Name` internally. Protocol parsers required no changes.

**Performance**: Benchmarks show no regression from the `String` to `Name` migration. The `Arc::clone` fast path eliminates heap allocations in the restrict pipeline's inner loop.

## Automatic Morphisms: Extend, Restrict, and Complete {#sec-auto-morphisms-section}

This section covers the automatic homomorphism search, the extend/restrict/complete operations,^[Extend pushes data forward along a migration ($\Sigma_F$), restrict pulls data backward ($\Delta_F$), and complete fills in missing data by computing products over fibers ($\Pi_F$). These three operations form an adjoint triple in categorical database theory [@spivak2012].] schema-level pushouts, overlap discovery, and the chase algorithm.

For the user-facing introduction, see the [tutorial chapter on automatic migration](@sec-auto-migration).

### Homomorphism Search

**File**: `crates/panproto-mig/src/hom_search.rs`

#### CSP Reduction

The homomorphism problem reduces to a constraint satisfaction problem:

- **Variables**: one per source vertex
- **Domains**: target vertices with compatible kind
- **Constraints**: edge-preserving squares

#### Algorithm

Backtracking search with forward checking:

1. **Variable ordering**: MRV heuristic --- assign vertices with smallest domains first
2. **Value selection**: try each domain value
3. **Forward check**: after assignment, verify all unassigned neighbors still have valid domain values via `has_compatible_edge`
4. **Edge compatibility**: requires matching edge kind between mapped vertex pairs; edge names are allowed to differ (name differences only affect quality scoring)
5. **Edge map derivation**: once all vertices are assigned, the edge map is determined by finding compatible edges between each pair of mapped vertices, preferring name-matching edges

#### Search Options

| Option | Effect |
|--------|--------|
| `monic` | Injective vertex map (tracks `used_targets` set) |
| `epic` | Surjective vertex map (checked at base case) |
| `iso` | Bijective (monic + epic) |
| `max_results` | Early termination bound |
| `initial` | Pre-assigned mappings (domain restricted to singletons) |

#### Quality Scoring

`compute_quality` produces a score in [0.0, 1.0]:

- 50% weight: average name similarity (1 - edit_distance / max_length) across vertex pairs
- 50% weight: fraction of edges where source and target have the same label name

#### Performance

Schemas have O(10-100) vertices. The backtracking search with MRV and forward checking handles this scale easily. Worst case is exponential in the number of vertices with identical kinds, but the edge constraints prune aggressively.

### Extend (Left Kan Extension)

**Files**: `crates/panproto-inst/src/wtype.rs` (`wtype_extend`), `crates/panproto-inst/src/functor.rs` (`functor_extend`)

#### W-Type Extend

`wtype_extend` maps all source nodes into the target schema:

1. Remap node anchors via `vertex_remap`
2. Remap arc edges via `edge_remap`
3. Rebuild tree indices

Unlike `wtype_restrict`, extend keeps all mappable nodes rather than dropping unmapped ones.

#### Functor Extend

`functor_extend` (pre-existing) handles the relational case:

1. Copy tables forward via `vertex_remap`
2. Compute coproduct (disjoint union) when multiple sources map to one target
3. Fill missing columns with `Value::Null`
4. Offset FK row indices for concatenated tables

### Complete (Right Kan Extension)

**File**: `crates/panproto-inst/src/pi.rs`

#### Functor Complete

`functor_pi` computes the product over fibers:

1. Build fiber map: for each target vertex, collect source vertices mapping to it
2. Single-element fibers: copy table directly
3. Multi-element fibers: Cartesian product of rows with column union
4. Size gate: reject products exceeding `max_product_size`

#### W-Type Complete

`wtype_pi` currently only handles injective migrations (single-element fibers). Multi-element fiber products for trees require Cartesian products of subtrees, which is deferred to a future release.

### Schema Pushout

**File**: `crates/panproto-schema/src/colimit.rs`

#### `SchemaOverlap`

Specifies which elements of two schemas are identified:

```rust
pub struct SchemaOverlap {
    pub vertex_pairs: Vec<(Name, Name)>,
    pub edge_pairs: Vec<(Edge, Edge)>,
}
```

#### `schema_pushout` Algorithm

1. Merged vertices: all left vertices + non-overlapping right vertices (overlapping right vertices merge with their left counterpart)
2. Merged edges: all left edges + right edges with remapped vertex references
3. Constraint/metadata union for overlapping vertices
4. Build morphisms: left->pushout (identity for left vertices), right->pushout (identity or remapped for right vertices)
5. Rebuild adjacency indices

#### Universal Property

For any schema C with morphisms from both left and right that agree on the overlap, there exists a unique morphism from the pushout to C.

### Overlap Discovery

**File**: `crates/panproto-mig/src/overlap.rs`

Uses `find_best_morphism` with `monic: true` in both directions (left->right and right->left), taking whichever produces more vertex pairs. The vertex map entries from the best morphism define the overlap pairs.

### Chase Algorithm

**File**: `crates/panproto-mig/src/chase.rs`

#### Embedded Dependencies

```rust
pub struct EmbeddedDependency {
    pub pattern_vertex: Name,
    pub pattern_values: HashMap<String, Value>,
    pub consequence_vertex: Name,
    pub consequence_values: HashMap<String, Value>,
}
```

#### Algorithm

1. Clone instance
2. For each iteration (up to `max_iterations`):
   a. For each dependency, find pattern matches without consequences
   b. Add missing consequence rows
   c. If nothing changed, return fixpoint
3. If limit reached, return `ChaseError::NonTermination`

#### Termination

Guaranteed for acyclic dependencies (no dependency's consequence can trigger itself). For cyclic dependencies, the iteration bound prevents infinite loops. panproto's equational theories are purely algebraic, so termination is guaranteed for all standard schema theories.

### From Morphisms to Protolenses {#sec-morphisms-to-protolenses}

The homomorphism search, extend/restrict/complete operations, and overlap discovery described here feed directly into the protolens engine (@sec-protolens-engine) and the automated lens generation pipeline (@sec-auto-lens-pipeline).

#### Pipeline Connection

The key integration point is the `factorize` function in `panproto-gat/src/factorize.rs`. Given a `TheoryMorphism` produced by `hom_search`, `factorize` decomposes it into elementary `TheoryEndofunctor` steps. Each step maps to an elementary protolens constructor, producing a `ProtolensChain` that can be instantiated at any schema satisfying the preconditions.

This extends the morphism tower from three levels to four:

1. **Theory morphism** (this section): structure-preserving map between GATs.
2. **Factorization** (@sec-factorize-algorithm): decomposition into elementary transforms.
3. **Protolens chain** (@sec-protolens-engine): schema-parameterized family of lenses.
4. **Concrete lens** (@sec-lenses): instantiated lens for a specific schema pair.

#### Overlap and Symmetric Lenses

The overlap discovery algorithm (`find_best_morphism` with `monic: true`) is reused by the protolens pipeline to construct symmetric lenses. Given two schemas with partial overlap, the pipeline factorizes the morphisms from each schema to the overlap, constructs protolens chains in both directions, and combines them into a `SymmetricLens`. See @sec-symmetric-from-overlap for details.

#### Performance Implications

Because `hom_search` is the most expensive step (exponential worst case), the automated lens pipeline caches morphism search results. Once a morphism is found, factorization and protolens construction are both linear in the theory size, so the incremental cost of producing a protolens chain from a cached morphism is negligible.

## The Type-Checking Pipeline {#sec-type-checking-pipeline-section}

Type-checking in panproto operates at three levels, each building on the one below.

### Pipeline Overview

```{mermaid}
%%| label: fig-typecheck-pipeline
%%| fig-cap: "The type-checking pipeline flows from raw terms through GAT-level checks, VCS-level diagnostics, and CLI-level formatting."
flowchart TD
    T["Raw Term / Equation"] --> TC["typecheck_term / typecheck_equation<br>(panproto-gat)"]
    TC -->|GatError| GD["GatDiagnostics<br>(panproto-vcs gat_validate)"]
    GD -->|type_errors, equation_errors| VS["ValidationStatus<br>(panproto-vcs index)"]
    VS -->|Invalid(reasons)| VF["VcsError::ValidationFailed<br>(panproto-vcs repo)"]
    VF -->|miette diagnostic| CLI["CLI error output<br>(panproto-cli)"]

    style TC fill:#e8f0fe,stroke:#1a73e8
    style GD fill:#fef3e8,stroke:#e87a1a
    style VS fill:#fef3e8,stroke:#e87a1a
    style VF fill:#fee8e8,stroke:#e81a1a
    style CLI fill:#fee8e8,stroke:#e81a1a
```

**Level 0 -- GAT (panproto-gat)**: The `typecheck` module provides the primitive type-checking functions. These operate on `Term`, `Equation`, and `Theory` values and return `Result<_, GatError>`. This level has no knowledge of schemas, migrations, or the VCS.

**Level 1 -- VCS (panproto-vcs)**: The `gat_validate` module wraps the GAT-level functions and collects their results into `GatDiagnostics`, a structured report of type errors, equation violations, and migration warnings. The `index` module stores these diagnostics in the staging area, and the `repo` module consults them during commit.

**Level 2 -- CLI (panproto-cli)**: The CLI commands convert `VcsError::ValidationFailed` into user-facing `miette` diagnostics with source context, suggestions, and error codes.

### Level 0: GAT Type-Checking

The GAT level provides four public functions that form the foundation of the pipeline:

#### typecheck_term

```rust
pub fn typecheck_term(
    term: &Term,
    ctx: &VarContext,
    theory: &Theory,
) -> Result<Arc<str>, GatError>
```

Given a term, a variable typing context, and a theory, infers the output sort of the term. This is the recursive core: for `App { op, args }`, it looks up `op` in the theory, typechecks each argument recursively, checks that each argument's inferred sort matches the operation's declared input sort, and returns the output sort.

#### infer_var_sorts

```rust
pub fn infer_var_sorts(eq: &Equation, theory: &Theory) -> Result<VarContext, GatError>
```

Infers variable sorts from an equation by walking both sides and collecting sort constraints from operation application sites. This eliminates the need for explicit sort annotations on equation variables --- the sorts are determined by the positions where variables appear.

#### typecheck_equation

```rust
pub fn typecheck_equation(eq: &Equation, theory: &Theory) -> Result<(), GatError>
```

Combines `infer_var_sorts` and `typecheck_term`: infers the variable context, typechecks both sides, and verifies they produce the same output sort.

#### typecheck_theory

```rust
pub fn typecheck_theory(theory: &Theory) -> Result<(), GatError>
```

Iterates over all equations in a theory and typechecks each one. Returns the first error encountered. This is the function that the VCS layer calls to validate protocol theories.

#### Error Types

All type-checking errors are variants of `GatError`:

| Variant | Meaning | Example |
|---|---|---|
| `UnboundVariable(name)` | Variable not in context | Using `x` in a term without declaring its sort |
| `OpNotFound(name)` | Operation not in theory | Referencing `compose` in a theory that has no such op |
| `TermArityMismatch { op, expected, got }` | Wrong argument count | Passing 1 arg to a binary op |
| `ArgTypeMismatch { op, arg_index, expected, got }` | Wrong argument sort | Passing an `Edge` where `Vertex` is expected |
| `EquationSortMismatch { equation, lhs_sort, rhs_sort }` | Equation sides differ | LHS has sort `Edge`, RHS has sort `Vertex` |
| `ConflictingVarSort { var, sort1, sort2 }` | Variable at incompatible positions | `x` used as `Vertex` on LHS and `Edge` on RHS |

: GatError type-checking variants. {#tbl-gat-errors}

These error types carry enough structured data for downstream layers to produce precise, actionable diagnostics.

### Level 1: VCS Integration

The VCS layer integrates type-checking at three points in the schema evolution workflow: staging (`schema add`), committing (`schema commit`), and theory validation.

#### GatDiagnostics

The `GatDiagnostics` struct in `panproto-vcs::gat_validate` is the bridge between GAT-level errors and VCS-level validation:

```rust
pub struct GatDiagnostics {
    pub type_errors: Vec<String>,
    pub equation_errors: Vec<String>,
    pub migration_warnings: Vec<String>,
}
```

- `type_errors` collects results from `typecheck_theory`. Each entry is the `Display` output of a `GatError`.
- `equation_errors` collects results from `check_model_with_options`. Each entry describes a specific equation violation with its variable assignment and the values that disagreed.
- `migration_warnings` collects structural warnings from `validate_migration` --- these are non-blocking issues like empty vertex maps or incoherent edge endpoints.

The `is_clean()` method returns `true` only when both `type_errors` and `equation_errors` are empty. Migration warnings do not block commits.

#### Integration Point: schema add (Repository::add)

When `Repository::add` stages a schema, it runs GAT validation on the auto-derived migration:

1. Compute the schema diff between HEAD and the new schema.
2. Call `auto_mig::derive_migration` to produce a `Migration`.
3. Call `gat_validate::validate_migration` to check migration structure --- the result populates `GatDiagnostics`.
4. Store the diagnostics in the `StagedSchema::gat_diagnostics` field.
5. If `gat_diagnostics.has_errors()`, set `ValidationStatus::Invalid(diag.all_errors())`.

The index file (`.panproto/index.json`) persists the diagnostics so that `schema commit` can consult them without re-running validation.

#### Integration Point: schema commit (Repository::commit_with_options)

When `Repository::commit_with_options` creates a commit, it checks two gates:

1. **ValidationStatus**: if `ValidationStatus::Invalid`, the commit is blocked with `VcsError::ValidationFailed`.
2. **GatDiagnostics**: even if `ValidationStatus` is `Valid`, the commit checks `gat_diagnostics.has_errors()` as a secondary gate. This handles the case where validation status was manually set to `Valid` but diagnostics were populated later.

Both gates are bypassed when `CommitOptions::skip_verify` is `true`. The `--skip-verify` CLI flag sets this option, providing an escape hatch for work-in-progress schemas.

```rust
pub struct CommitOptions {
    pub skip_verify: bool,
}
```

#### Integration Point: Theory Validation

The `validate_theory_equations` function provides standalone theory type-checking:

```rust
pub fn validate_theory_equations(theory: &Theory) -> GatDiagnostics
```

This is called by the `schema validate` and `schema typecheck` CLI commands for schemas that are not part of a VCS repository. It runs `typecheck_theory` and collects any errors into `GatDiagnostics::type_errors`.

The `validate_schema_equations` function goes further by building a model (a concrete instance that satisfies a theory's requirements --- like a Rust struct implementing a trait) from the schema and running `check_model_with_options`:

```rust
pub fn validate_schema_equations(
    schema: &Schema,
    theory: &Theory,
    model: &Model,
) -> GatDiagnostics
```

This checks whether the schema's interpretation of the theory satisfies all equations. The check uses `CheckModelOptions::max_assignments = 10_000` to bound enumeration time. If the assignment count exceeds the limit, the error is reported as a non-blocking diagnostic rather than a hard failure.

### Level 2: CLI Error Formatting

The CLI layer converts `VcsError::ValidationFailed` into user-facing diagnostics using the `miette` crate.

#### Error Flow

```
GatError::ArgTypeMismatch { op: "src", arg_index: 0, expected: "Edge", got: "Vertex" }
  -> GatDiagnostics { type_errors: ["type mismatch for src argument 0: expected Edge, got Vertex"] }
    -> ValidationStatus::Invalid(["type error: type mismatch for src argument 0: ..."])
      -> VcsError::ValidationFailed { reasons: ["type error: ..."] }
        -> miette::Report with labeled span and suggestion
```

Each layer adds context:

- **GatError**: the raw error with structured fields (`op`, `arg_index`, `expected`, `got`).
- **GatDiagnostics**: the error serialized to a string via `Display`, prefixed with its category (`"type error: "` or `"equation violation: "`).
- **ValidationStatus**: the collected error strings from `all_errors()`.
- **VcsError**: the error wrapped in a `ValidationFailed` variant with a `reasons` field.
- **miette**: the error rendered with ANSI colors, source labels, and help text.

#### Equation Violation Formatting

Equation violations receive special formatting in `gat_validate::format_violation`:

```
equation 'assoc' violated when a=2, b=3, c=1: LHS=6, RHS=4
```

This format shows the equation name, the variable assignment that triggered the violation, and the concrete values that the two sides evaluated to. For models with symbolic term representations (as produced by `free_model`), the values are printed as term strings:

```
equation 'left_id' violated when a=mul(unit(), unit()): LHS=mul(unit(), mul(unit(), unit())), RHS=mul(unit(), unit())
```

### Cross-Cutting Integration Points

The type-checking pipeline touches several subsystems beyond the primary `add`/`commit` path:

#### auto_mig

The `auto_mig::derive_migration` function in `panproto-vcs` produces the migration that is subsequently validated. While `auto_mig` itself does not call type-checking functions, the quality of its output directly affects whether GAT validation passes. A poorly-derived migration (e.g., one that maps vertices to vertices of incompatible kinds) will produce `GatDiagnostics` with migration warnings.

#### Index Persistence

The `StagedSchema` struct in `panproto-vcs::index` stores `gat_diagnostics: Option<GatDiagnostics>`. This field is serialized to `.panproto/index.json` via serde. The `#[serde(default)]` attribute ensures backward compatibility: index files written before the GAT completeness feature was added deserialize with `gat_diagnostics: None`, which is treated as "no diagnostics available" rather than "diagnostics clean".

#### Merge and Rebase

The merge and rebase pipelines do not currently run GAT type-checking on the merged result. Three-way merge operates at the schema level (vertices, edges, constraints), and the merged schema is validated structurally. GAT validation is deferred to the next `schema add` that stages the merged schema for further evolution.

The merge pipeline does use GAT-level operations: it calls `pullback` to compute structural overlap between branches, and it constructs migration morphisms that are validated by `check_morphism`. These are correctness checks on the merge algorithm itself, not on the user's schema.

#### CLI Commands

The following CLI commands invoke parts of the type-checking pipeline:

| Command | Type-checking performed |
|---|---|
| `schema validate` | `typecheck_theory` on protocol theory |
| `schema typecheck` | `typecheck_theory` on schema theory; optionally validates migration morphism |
| `schema verify` | `check_model_with_options` on schema model |
| `schema check --typecheck` | `validate_migration` + `validate_theory_equations` |
| `schema add` | `validate_migration` on auto-derived migration |
| `schema commit` | Checks stored `GatDiagnostics` from staging |
| `schema commit --skip-verify` | Bypasses `GatDiagnostics` check |

### Design Rationale

#### Why Three Levels

The three-level design separates concerns cleanly:

- **Level 0** is pure mathematics. It knows about terms, sorts, and operations, nothing about files, schemas, or version control. This makes it easy to test in isolation and reuse in non-VCS contexts (e.g., the WASM boundary, the TypeScript SDK).
- **Level 1** is integration logic. It knows how to translate between GAT errors and VCS concepts (staging, commits, validation status). It serializes diagnostics for persistence.
- **Level 2** is presentation. It knows how to render errors for humans with colors, source labels, and suggestions. It uses `miette`, which is a CLI-only dependency.

#### Why Store Diagnostics in the Index

Storing `GatDiagnostics` in the index rather than re-computing them at commit time has two benefits:

1. **Performance**: type-checking and model checking can be expensive for large theories. Running them once at staging time and persisting the results avoids redundant work.
2. **Auditability**: the index file records exactly what diagnostics were produced, even if the user later modifies the schema file. The commit gate checks the stored diagnostics, not the current file state.

#### Why skip_verify Exists

The `--skip-verify` flag is an explicit escape hatch for situations where the type-checking pipeline produces false positives or where a work-in-progress schema intentionally violates equations. Without this flag, users would be blocked from committing legitimate intermediate states. The flag is named to make its risk visible: you are skipping verification, not disabling a cosmetic check.

#### Why Equation Checking is Bounded

The `max_assignments` bound in `CheckModelOptions` prevents combinatorial explosion. An equation with 3 variables over a carrier of 100 elements requires 1,000,000 assignments; with 4 variables it is 100,000,000. The default bound of 10,000 catches most violations while keeping verification fast. Users who need exhaustive checking can raise the bound via `schema verify --max-assignments`.
