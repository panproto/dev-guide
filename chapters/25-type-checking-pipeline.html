21  The Type-Checking Pipeline

This chapter describes how GAT type-checking flows through panproto’s architecture, from raw terms in panproto-gat through VCS validation in panproto-vcs to user-facing error messages in the CLI. Understanding this pipeline is essential for contributors who work on schema validation, migration derivation, or error reporting.

21.1 Pipeline Overview

Type-checking in panproto operates at three levels, each building on the one below:

Level 0 – GAT (panproto-gat): The typecheck module provides the primitive type-checking functions. These operate on Term, Equation, and Theory values and return Result<_, GatError>. This level has no knowledge of schemas, migrations, or the VCS.

Level 1 – VCS (panproto-vcs): The gat_validate module wraps the GAT-level functions and collects their results into GatDiagnostics, a structured report of type errors, equation violations, and migration warnings. The index module stores these diagnostics in the staging area, and the repo module consults them during commit.

Level 2 – CLI (panproto-cli): The CLI commands convert VcsError::ValidationFailed into user-facing miette diagnostics with source context, suggestions, and error codes.

21.2 Level 0: GAT Type-Checking

The GAT level provides four public functions that form the foundation of the pipeline:

21.2.1 typecheck_term

pub fn typecheck_term(
    term: &Term,
    ctx: &VarContext,
    theory: &Theory,
) -> Result<Arc<str>, GatError>

Given a term, a variable typing context, and a theory, infers the output sort of the term. This is the recursive core: for App { op, args }, it looks up op in the theory, typechecks each argument recursively, checks that each argument’s inferred sort matches the operation’s declared input sort, and returns the output sort.

21.2.2 infer_var_sorts

pub fn infer_var_sorts(eq: &Equation, theory: &Theory) -> Result<VarContext, GatError>

Infers variable sorts from an equation by walking both sides and collecting sort constraints from operation application sites. This eliminates the need for explicit sort annotations on equation variables – the sorts are determined by the positions where variables appear.

21.2.3 typecheck_equation

pub fn typecheck_equation(eq: &Equation, theory: &Theory) -> Result<(), GatError>

Combines infer_var_sorts and typecheck_term: infers the variable context, typechecks both sides, and verifies they produce the same output sort.

21.2.4 typecheck_theory

pub fn typecheck_theory(theory: &Theory) -> Result<(), GatError>

Iterates over all equations in a theory and typechecks each one. Returns the first error encountered. This is the function that the VCS layer calls to validate protocol theories.

21.3 Level 1: VCS Integration

The VCS layer integrates type-checking at three points in the schema evolution workflow: staging (schema add), committing (schema commit), and theory validation.

21.3.1 GatDiagnostics

The GatDiagnostics struct in panproto-vcs::gat_validate is the bridge between GAT-level errors and VCS-level validation:

pub struct GatDiagnostics {
    pub type_errors: Vec<String>,
    pub equation_errors: Vec<String>,
    pub migration_warnings: Vec<String>,
}
  • type_errors collects results from typecheck_theory. Each entry is the Display output of a GatError.
  • equation_errors collects results from check_model_with_options. Each entry describes a specific equation violation with its variable assignment and the values that disagreed.
  • migration_warnings collects structural warnings from validate_migration – these are non-blocking issues like empty vertex maps or incoherent edge endpoints.

The is_clean() method returns true only when both type_errors and equation_errors are empty. Migration warnings do not block commits.

21.3.2 Integration Point: schema add (Repository::add)

When Repository::add stages a schema, it runs GAT validation on the auto-derived migration:

  1. Compute the schema diff between HEAD and the new schema.
  2. Call auto_mig::derive_migration to produce a Migration.
  3. Call gat_validate::validate_migration to check migration structure – the result populates GatDiagnostics.
  4. Store the diagnostics in the StagedSchema::gat_diagnostics field.
  5. If gat_diagnostics.has_errors(), set ValidationStatus::Invalid(diag.all_errors()).

The index file (.panproto/index.json) persists the diagnostics so that schema commit can consult them without re-running validation.

21.3.3 Integration Point: schema commit (Repository::commit_with_options)

When Repository::commit_with_options creates a commit, it checks two gates:

  1. ValidationStatus: if ValidationStatus::Invalid, the commit is blocked with VcsError::ValidationFailed.
  2. GatDiagnostics: even if ValidationStatus is Valid, the commit checks gat_diagnostics.has_errors() as a secondary gate. This handles the case where validation status was manually set to Valid but diagnostics were populated later.

Both gates are bypassed when CommitOptions::skip_verify is true. The --skip-verify CLI flag sets this option, providing an escape hatch for work-in-progress schemas.

pub struct CommitOptions {
    pub skip_verify: bool,
}

21.3.4 Integration Point: Theory Validation

The validate_theory_equations function provides standalone theory type-checking:

pub fn validate_theory_equations(theory: &Theory) -> GatDiagnostics

This is called by the schema validate and schema typecheck CLI commands for schemas that are not part of a VCS repository. It runs typecheck_theory and collects any errors into GatDiagnostics::type_errors.

The validate_schema_equations function goes further by building a model from the schema and running check_model_with_options:

pub fn validate_schema_equations(
    schema: &Schema,
    theory: &Theory,
    model: &Model,
) -> GatDiagnostics

This checks whether the schema’s interpretation of the theory satisfies all equations. The check uses CheckModelOptions::max_assignments = 10_000 to bound enumeration time. If the assignment count exceeds the limit, the error is reported as a non-blocking diagnostic rather than a hard failure.

21.4 Level 2: CLI Error Formatting

The CLI layer converts VcsError::ValidationFailed into user-facing diagnostics using the miette crate.

21.4.1 Error Flow

GatError::ArgTypeMismatch { op: "src", arg_index: 0, expected: "Edge", got: "Vertex" }
  → GatDiagnostics { type_errors: ["type mismatch for src argument 0: expected Edge, got Vertex"] }
    → ValidationStatus::Invalid(["type error: type mismatch for src argument 0: ..."])
      → VcsError::ValidationFailed { reasons: ["type error: ..."] }
        → miette::Report with labeled span and suggestion

Each layer adds context:

  • GatError: the raw error with structured fields (op, arg_index, expected, got).
  • GatDiagnostics: the error serialized to a string via Display, prefixed with its category ("type error: " or "equation violation: ").
  • ValidationStatus: the collected error strings from all_errors().
  • VcsError: the error wrapped in a ValidationFailed variant with a reasons field.
  • miette: the error rendered with ANSI colors, source labels, and help text.

21.4.2 Equation Violation Formatting

Equation violations receive special formatting in gat_validate::format_violation:

equation 'assoc' violated when a=2, b=3, c=1: LHS=6, RHS=4

This format shows the equation name, the variable assignment that triggered the violation, and the concrete values that the two sides evaluated to. For models with symbolic term representations (as produced by free_model), the values are printed as term strings:

equation 'left_id' violated when a=mul(unit(), unit()): LHS=mul(unit(), mul(unit(), unit())), RHS=mul(unit(), unit())

21.5 Cross-Cutting Integration Points

The type-checking pipeline touches several subsystems beyond the primary add/commit path:

21.5.1 auto_mig

The auto_mig::derive_migration function in panproto-vcs produces the migration that is subsequently validated. While auto_mig itself does not call type-checking functions, the quality of its output directly affects whether GAT validation passes. A poorly-derived migration (e.g., one that maps vertices to vertices of incompatible kinds) will produce GatDiagnostics with migration warnings.

21.5.2 Index Persistence

The StagedSchema struct in panproto-vcs::index stores gat_diagnostics: Option<GatDiagnostics>. This field is serialized to .panproto/index.json via serde. The #[serde(default)] attribute ensures backward compatibility: index files written before the GAT completeness feature was added deserialize with gat_diagnostics: None, which is treated as “no diagnostics available” rather than “diagnostics clean”.

21.5.3 Merge and Rebase

The merge and rebase pipelines do not currently run GAT type-checking on the merged result. This is intentional: three-way merge operates at the schema level (vertices, edges, constraints), and the merged schema is validated structurally. GAT validation is deferred to the next schema add that stages the merged schema for further evolution.

However, the merge pipeline does use GAT-level operations: it calls pullback to compute structural overlap between branches, and it constructs migration morphisms that are validated by check_morphism. These are correctness checks on the merge algorithm itself, not on the user’s schema.

21.6 Design Rationale

21.6.1 Why Three Levels

The three-level design separates concerns cleanly:

  • Level 0 is pure mathematics. It knows about terms, sorts, and operations, nothing about files, schemas, or version control. This makes it easy to test in isolation and reuse in non-VCS contexts (e.g., the WASM boundary, the TypeScript SDK).
  • Level 1 is integration logic. It knows how to translate between GAT errors and VCS concepts (staging, commits, validation status). It serializes diagnostics for persistence.
  • Level 2 is presentation. It knows how to render errors for humans with colors, source labels, and suggestions. It uses miette, which is a CLI-only dependency.

21.6.2 Why Store Diagnostics in the Index

Storing GatDiagnostics in the index rather than re-computing them at commit time has two benefits:

  1. Performance: type-checking and model checking can be expensive for large theories. Running them once at staging time and persisting the results avoids redundant work.
  2. Auditability: the index file records exactly what diagnostics were produced, even if the user later modifies the schema file. The commit gate checks the stored diagnostics, not the current file state.

21.6.3 Why skip_verify Exists

The --skip-verify flag is an explicit escape hatch for situations where the type-checking pipeline produces false positives or where a work-in-progress schema intentionally violates equations. Without this flag, users would be blocked from committing legitimate intermediate states. The flag is named to make its risk visible: you are skipping verification, not disabling a cosmetic check.

21.6.4 Why Equation Checking is Bounded

The max_assignments bound in CheckModelOptions prevents combinatorial explosion. An equation with 3 variables over a carrier of 100 elements requires 1,000,000 assignments; with 4 variables it is 100,000,000. The default bound of 10,000 catches most violations while keeping verification fast. Users who need exhaustive checking can raise the bound via schema verify --max-assignments.

## Pipeline Overview

Type-checking in panproto operates at three levels, each building on the one below:

```{mermaid}

%%| fig-cap: "The type-checking pipeline flows from raw terms through GAT-level checks, VCS-level diagnostics, and CLI-level formatting."
```

### Error Types

All type-checking errors are variants of `GatError`:

| Variant | Meaning | Example |
|---|---|---|
| `UnboundVariable(name)` | Variable not in context | Using `x` in a term without declaring its sort |
| `OpNotFound(name)` | Operation not in theory | Referencing `compose` in a theory that has no such op |
| `TermArityMismatch { op, expected, got }` | Wrong argument count | Passing 1 arg to a binary op |
| `ArgTypeMismatch { op, arg_index, expected, got }` | Wrong argument sort | Passing an `Edge` where `Vertex` is expected |
| `EquationSortMismatch { equation, lhs_sort, rhs_sort }` | Equation sides differ | LHS has sort `Edge`, RHS has sort `Vertex` |
| `ConflictingVarSort { var, sort1, sort2 }` | Variable at incompatible positions | `x` used as `Vertex` on LHS and `Edge` on RHS |

## Level 1: VCS Integration

The VCS layer integrates type-checking at three points in the schema evolution workflow: staging (`schema add`), committing (`schema commit`), and theory validation.

### GatDiagnostics

The `GatDiagnostics` struct in `panproto-vcs::gat_validate` is the bridge between GAT-level errors and VCS-level validation:

```rust
pub struct GatDiagnostics {
    pub type_errors: Vec<String>,
```

## Design Rationale

### Why Three Levels

The three-level design separates concerns cleanly:

- **Level 0** is pure mathematics. It knows about terms, sorts, and operations, nothing about files, schemas, or version control. This makes it easy to test in isolation and reuse in non-VCS contexts (e.g., the WASM boundary, the TypeScript SDK).
- **Level 1** is integration logic. It knows how to translate between GAT errors and VCS concepts (staging, commits, validation status). It serializes diagnostics for persistence.
- **Level 2** is presentation. It knows how to render errors for humans with colors, source labels, and suggestions. It uses `miette`, which is a CLI-only dependency.

### Why Store Diagnostics in the Index

Storing `GatDiagnostics` in the index rather than re-computing them at commit time has two benefits:

1. **Performance**: type-checking and model checking can be expensive for large theories. Running them once at staging time and persisting the results avoids redundant work.
2. **Auditability**: the index file records exactly what diagnostics were produced, even if the user later modifies the schema file. The commit gate checks the stored diagnostics, not the current file state.

### Why skip_verify Exists

The `--skip-verify` flag is an explicit escape hatch for situations where the type-checking pipeline produces false positives or where a work-in-progress schema intentionally violates equations. Without this flag, users would be blocked from committing legitimate intermediate states. The flag is named to make its risk visible: you are skipping verification, not disabling a cosmetic check.

### Why Equation Checking is Bounded

The `max_assignments` bound in `CheckModelOptions` prevents combinatorial explosion. An equation with 3 variables over a carrier of 100 elements requires 1,000,000 assignments; with 4 variables it is 100,000,000. The default bound of 10,000 catches most violations while keeping verification fast. Users who need exhaustive checking can raise the bound via `schema verify --max-assignments`.
