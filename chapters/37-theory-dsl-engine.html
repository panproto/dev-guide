31  Theory DSL Engine

The panproto-theory-dsl crate compiles declarative theory specifications (Nickel, JSON, or YAML) into the panproto GAT algebra (Theory + TheoryMorphism + Protocol). This chapter covers the crate’s architecture, compilation logic, and integration points.

31.1 Crate structure

crates/panproto-theory-dsl/
  contracts/theory.ncl       Nickel contract library (bundled via include_str!)
  src/
    lib.rs                   Public API: load(), load_dir(), compile(), load_and_compile()
    document.rs              Serde types: TheoryDocument, TheoryBody (5 variants), all specs
    eval.rs                  Nickel/JSON/YAML evaluation to TheoryDocument
    compile.rs               Unified dispatcher, builtin_resolver(), bundle compilation
    compile_theory.rs        TheorySpec -> Theory (sorts, ops, eqs, typechecking)
    compile_morphism.rs      MorphismSpec -> TheoryMorphism (with check_morphism)
    compile_compose.rs       CompositionBody -> Theory (via recompose)
    compile_protocol.rs      ProtocolSpec -> Protocol (TheoryRef resolution)
    error.rs                 TheoryDslError with miette diagnostics

31.2 Dependencies

Crate Role
nickel-lang 2.0 Nickel evaluator with to_serde deserialization
panproto-gat Theory, TheoryMorphism, CompositionSpec, Sort, Operation, Term, Equation, typecheck_theory, check_morphism, recompose
panproto-schema Protocol, EdgeRule
panproto-protocols Built-in theory constructors (th_graph, th_constraint, etc.)
panproto-expr Expr AST for directed equations and custom policies
panproto-expr-parser tokenize(), parse() for expression strings

31.3 Parallel to panproto-lens-dsl

The architecture mirrors panproto-lens-dsl (Chapter 30):

Lens DSL Theory DSL
LensDocument TheoryDocument
Step (19 variants) TheoryBody (5 variants)
ComposeSpec (vertical/horizontal) CompositionBody (colimit steps)
CompiledLens CompiledTheorySet
contracts/lens.ncl contracts/theory.ncl
Outputs ProtolensChain Outputs Theory + TheoryMorphism + Protocol

31.4 Evaluation layer

Three evaluation paths converge on TheoryDocument:

Nickel (.ncl): The bundled contract library (contracts/theory.ncl) is written to a temp directory. Nickel’s import resolver finds it via import "panproto/theory.ncl". The evaluator runs eval_deep_for_export, which applies all contracts, resolves merges, evaluates functions, and normalizes. The result is deserialized via to_serde::<TheoryDocument>().

JSON (.json): Direct serde_json::from_str.

YAML (.yaml, .yml): Direct yaml_serde::from_str.

31.5 Body discrimination

TheoryBody uses #[serde(untagged)] with five variants. Discrimination works by the presence of distinguishing keys:

  • theory key present: TheoryBody::Theory
  • morphism key present: TheoryBody::Morphism
  • compose key present: TheoryBody::Composition
  • protocol key present: TheoryBody::Protocol
  • bundle key present: TheoryBody::Bundle

The ProtocolSpec and BundleSpec variants are Box-wrapped to reduce enum size variance (clippy large_enum_variant).

31.6 Theory compilation

compile_theory() in compile_theory.rs:

  1. Sorts: Each SortSpec maps to Sort. SortKindSpec::Structural becomes SortKind::Structural; Val, Coercion, and Merger map to their respective SortKind variants with parsed ValueKind and CoercionClass. Dependent sorts use SortParam::new.

  2. Operations: input shorthand produces Operation::unary (parameter name derived from first character of input sort). inputs produces Operation::new with explicit (Arc<str>, Arc<str>) pairs. Neither produces Operation::nullary.

  3. Equations: lhs and rhs strings are parsed by parse_term(), a recursive-descent parser supporting Var and App forms: parse_term("f(g(x), y)") produces App { op: "f", args: [App { op: "g", args: [Var("x")] }, Var("y")] }.

  4. Directed equations: Term strings parsed via parse_term(), implementation expressions via panproto_expr_parser::tokenize() + parse().

  5. Policies: Strategy variants map directly; Custom requires expression parsing.

  6. Construction: Theory::full(name, extends, sorts, ops, eqs, directed_eqs, policies).

  7. Typechecking: panproto_gat::typecheck_theory() verifies equation well-typedness.

31.7 Morphism compilation

compile_morphism() in compile_morphism.rs:

  1. Resolve domain and codomain via the resolver callback.
  2. Convert sort_map and op_map from HashMap<String, String> to HashMap<Arc<str>, Arc<str>>.
  3. Construct TheoryMorphism::new.
  4. Run panproto_gat::check_morphism to verify sort arities, operation type signatures, and equation preservation.

31.8 Composition compilation

compile_composition() in compile_compose.rs:

  1. Build a local registry from local_theories plus resolved bases.
  2. Also resolve any theories referenced in step left/right fields (except step_* intermediates).
  3. Convert each ColimitStepSpec to CompositionStep::Colimit.
  4. Call panproto_gat::recompose() which replays the colimit sequence, naming intermediates step_0, step_1, etc.

31.9 Protocol compilation

compile_protocol() in compile_protocol.rs:

  1. Resolve schema_theory and instance_theory via TheoryRef:
    • Named(s): verify existence in local theories or resolver.
    • Inline(spec): compile the inline TheorySpec.
    • Composed(spec): compile the inline CompositionSpec_.
  2. Map EdgeRuleSpec (singular src_kind/tgt_kind) to EdgeRule (vector src_kinds/tgt_kinds).
  3. Construct Protocol with all feature flags defaulting to false.

31.10 Bundle compilation

compile_bundle_inner() in compile.rs processes definitions in four phases:

  1. Theories: Compile all TheorySpecs (no intra-bundle dependencies).
  2. Compositions: Compile all CompositionSpec_s (may reference bundle theories).
  3. Morphisms: Compile all MorphismSpecs (reference theories by name).
  4. Protocols: Compile all ProtocolSpecs (reference theories by name or inline).

Each phase creates a fresh combined resolver (local theories + external resolver) to avoid borrow checker conflicts with the mutable result set.

31.11 Builtin resolver

builtin_resolver() returns a closure resolving 11 built-in theory names:

Name Source
ThGraph panproto_protocols::theories::th_graph()
ThConstraint panproto_protocols::theories::th_constraint()
ThMulti panproto_protocols::theories::th_multi()
ThWType panproto_protocols::theories::th_wtype()
ThMeta panproto_protocols::theories::th_meta()
ThSimpleGraph panproto_protocols::theories::th_simple_graph()
ThHypergraph panproto_protocols::theories::th_hypergraph()
ThInterface panproto_protocols::theories::th_interface()
ThFunctor panproto_protocols::theories::th_functor()
ThFlat panproto_protocols::theories::th_flat()
ThGraphInstance panproto_protocols::theories::th_graph_instance()

The last six were previously private in panproto-protocols; they were made pub to support this resolver.

31.12 Error handling

TheoryDslError has 12 variants covering the full pipeline:

  • Evaluation: NickelEval, Json, Yaml, UnsupportedExtension
  • Document: NoBody, MultipleBodies
  • Parsing: TermParse, ExprParse
  • Resolution: TheoryNotFound
  • Validation: TypeCheck, MorphismCheck, ColimitFailed
  • Bundle: Duplicate, DependencyCycle
  • Passthrough: Io, Gat

All variants carry miette diagnostic codes (panproto_theory_dsl::*).

31.13 CLI integration

The schema theory subcommand group is wired via TheoryAction enum in panproto-cli/src/main.rs, dispatching to cmd::theory::* functions. Each command calls load() + compile() with the builtin resolver and formats output as either human-readable text or JSON (--json flag).

# Theory DSL Engine {#sec-theory-dsl-engine}

The `panproto-theory-dsl` crate compiles declarative theory specifications (Nickel, JSON, or YAML) into the panproto GAT algebra (`Theory` + `TheoryMorphism` + `Protocol`). This chapter covers the crate's architecture, compilation logic, and integration points.

## Crate structure

```
crates/panproto-theory-dsl/
  contracts/theory.ncl       Nickel contract library (bundled via include_str!)
  src/
    lib.rs                   Public API: load(), load_dir(), compile(), load_and_compile()
    document.rs              Serde types: TheoryDocument, TheoryBody (5 variants), all specs
    eval.rs                  Nickel/JSON/YAML evaluation to TheoryDocument
    compile.rs               Unified dispatcher, builtin_resolver(), bundle compilation
    compile_theory.rs        TheorySpec -> Theory (sorts, ops, eqs, typechecking)
    compile_morphism.rs      MorphismSpec -> TheoryMorphism (with check_morphism)
    compile_compose.rs       CompositionBody -> Theory (via recompose)
    compile_protocol.rs      ProtocolSpec -> Protocol (TheoryRef resolution)
    error.rs                 TheoryDslError with miette diagnostics
```

## Dependencies

| Crate | Role |
|-------|------|
| `nickel-lang` 2.0 | Nickel evaluator with `to_serde` deserialization |
| `panproto-gat` | `Theory`, `TheoryMorphism`, `CompositionSpec`, `Sort`, `Operation`, `Term`, `Equation`, `typecheck_theory`, `check_morphism`, `recompose` |
| `panproto-schema` | `Protocol`, `EdgeRule` |
| `panproto-protocols` | Built-in theory constructors (`th_graph`, `th_constraint`, etc.) |
| `panproto-expr` | `Expr` AST for directed equations and custom policies |
| `panproto-expr-parser` | `tokenize()`, `parse()` for expression strings |

## Parallel to panproto-lens-dsl

The architecture mirrors `panproto-lens-dsl` (@sec-lens-dsl-engine):

| Lens DSL | Theory DSL |
|----------|------------|
| `LensDocument` | `TheoryDocument` |
| `Step` (19 variants) | `TheoryBody` (5 variants) |
| `ComposeSpec` (vertical/horizontal) | `CompositionBody` (colimit steps) |
| `CompiledLens` | `CompiledTheorySet` |
| `contracts/lens.ncl` | `contracts/theory.ncl` |
| Outputs `ProtolensChain` | Outputs `Theory` + `TheoryMorphism` + `Protocol` |

## Evaluation layer

Three evaluation paths converge on `TheoryDocument`:

**Nickel** (`.ncl`): The bundled contract library (`contracts/theory.ncl`) is written to a temp directory. Nickel's import resolver finds it via `import "panproto/theory.ncl"`. The evaluator runs `eval_deep_for_export`, which applies all contracts, resolves merges, evaluates functions, and normalizes. The result is deserialized via `to_serde::<TheoryDocument>()`.

**JSON** (`.json`): Direct `serde_json::from_str`.

**YAML** (`.yaml`, `.yml`): Direct `yaml_serde::from_str`.

## Body discrimination

`TheoryBody` uses `#[serde(untagged)]` with five variants. Discrimination works by the presence of distinguishing keys:

- `theory` key present: `TheoryBody::Theory`
- `morphism` key present: `TheoryBody::Morphism`
- `compose` key present: `TheoryBody::Composition`
- `protocol` key present: `TheoryBody::Protocol`
- `bundle` key present: `TheoryBody::Bundle`

The `ProtocolSpec` and `BundleSpec` variants are `Box`-wrapped to reduce enum size variance (clippy `large_enum_variant`).

## Theory compilation

`compile_theory()` in `compile_theory.rs`:

1. **Sorts**: Each `SortSpec` maps to `Sort`. `SortKindSpec::Structural` becomes `SortKind::Structural`; `Val`, `Coercion`, and `Merger` map to their respective `SortKind` variants with parsed `ValueKind` and `CoercionClass`. Dependent sorts use `SortParam::new`.

2. **Operations**: `input` shorthand produces `Operation::unary` (parameter name derived from first character of input sort). `inputs` produces `Operation::new` with explicit `(Arc<str>, Arc<str>)` pairs. Neither produces `Operation::nullary`.

3. **Equations**: `lhs` and `rhs` strings are parsed by `parse_term()`, a recursive-descent parser supporting `Var` and `App` forms: `parse_term("f(g(x), y)")` produces `App { op: "f", args: [App { op: "g", args: [Var("x")] }, Var("y")] }`.

4. **Directed equations**: Term strings parsed via `parse_term()`, implementation expressions via `panproto_expr_parser::tokenize()` + `parse()`.

5. **Policies**: Strategy variants map directly; `Custom` requires expression parsing.

6. **Construction**: `Theory::full(name, extends, sorts, ops, eqs, directed_eqs, policies)`.

7. **Typechecking**: `panproto_gat::typecheck_theory()` verifies equation well-typedness.

## Morphism compilation

`compile_morphism()` in `compile_morphism.rs`:

1. Resolve `domain` and `codomain` via the resolver callback.
2. Convert `sort_map` and `op_map` from `HashMap<String, String>` to `HashMap<Arc<str>, Arc<str>>`.
3. Construct `TheoryMorphism::new`.
4. Run `panproto_gat::check_morphism` to verify sort arities, operation type signatures, and equation preservation.

## Composition compilation

`compile_composition()` in `compile_compose.rs`:

1. Build a local registry from `local_theories` plus resolved `bases`.
2. Also resolve any theories referenced in step `left`/`right` fields (except `step_*` intermediates).
3. Convert each `ColimitStepSpec` to `CompositionStep::Colimit`.
4. Call `panproto_gat::recompose()` which replays the colimit sequence, naming intermediates `step_0`, `step_1`, etc.

## Protocol compilation

`compile_protocol()` in `compile_protocol.rs`:

1. Resolve `schema_theory` and `instance_theory` via `TheoryRef`:
   - `Named(s)`: verify existence in local theories or resolver.
   - `Inline(spec)`: compile the inline `TheorySpec`.
   - `Composed(spec)`: compile the inline `CompositionSpec_`.
2. Map `EdgeRuleSpec` (singular `src_kind`/`tgt_kind`) to `EdgeRule` (vector `src_kinds`/`tgt_kinds`).
3. Construct `Protocol` with all feature flags defaulting to `false`.

## Bundle compilation

`compile_bundle_inner()` in `compile.rs` processes definitions in four phases:

1. **Theories**: Compile all `TheorySpec`s (no intra-bundle dependencies).
2. **Compositions**: Compile all `CompositionSpec_`s (may reference bundle theories).
3. **Morphisms**: Compile all `MorphismSpec`s (reference theories by name).
4. **Protocols**: Compile all `ProtocolSpec`s (reference theories by name or inline).

Each phase creates a fresh combined resolver (`local theories + external resolver`) to avoid borrow checker conflicts with the mutable result set.

## Error handling

`TheoryDslError` has 12 variants covering the full pipeline:

- Evaluation: `NickelEval`, `Json`, `Yaml`, `UnsupportedExtension`
- Document: `NoBody`, `MultipleBodies`
- Parsing: `TermParse`, `ExprParse`
- Resolution: `TheoryNotFound`
- Validation: `TypeCheck`, `MorphismCheck`, `ColimitFailed`
- Bundle: `Duplicate`, `DependencyCycle`
- Passthrough: `Io`, `Gat`

All variants carry miette diagnostic codes (`panproto_theory_dsl::*`).
